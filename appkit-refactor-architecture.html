CCTerm 纯 AppKit MVVM-C 重构架构

一份把 CCTerm 从「SwiftUI 优先、AppKit 例外」迁往「纯 AppKit、MVVM-C、零 SwiftUI」的落地重构手册。它顺着现存类型逐层归位——五层职责、模块边界、组合根注入、控制器容器化、Coordinator 导航、Session 领域、笨视图、列表数据源、状态数据流与 ViewModel 取舍，直到内存·并发·可测试性收尾。每一章都从 CCTerm 今天的样子出发，钉出它该变成什么。

总览：CCTerm 现状与目标架构

这一章不改一行代码，只做一件事：把 CCTerm 今天的样子和它要去的样子摊在同一张桌子上，画出从此到彼的地图。今天的 CCTerm 是「SwiftUI 优先、AppKit 例外」——一层薄薄的 AppKit 骨架撑在几处承重接缝上，其余全是 SwiftUI 叶子。目标是走到光谱的另一极：纯 AppKit、MVVM-C、零 SwiftUI。本手册通篇建立的五层分层与「数据向下、事件向上」（见「纯 AppKit 开发规范手册」第 1 章）就是这张地图的坐标系；本章把每一个现存类型钉到它在坐标系里的理想归宿，让后续每一章都能顺着一条现存名字追溯到它该变成什么。迁移的本质不是重写，而是一次重新归类——SwiftUI 那层黏合剂溶解之后，耦合藏身的地方会自己暴露出来。

AppKit-by-exception（今）CCTerm 现状的架构口径：默认用 SwiftUI，只有在 SwiftUI 无法满足（性能、生命周期时序、缺失能力）时才退回 AppKit。承重的 AppKit 面只有五处：转录（transcript）、主窗口 split/router、侧栏 outline、工具栏、AppDelegate。
纯 AppKit / MVVM-C（目标）目标口径：删除或移植每一个 @main App 场景、Commands 块、NSHostingController/NSHostingView、NSViewRepresentable、EnvironmentKey 注入、ButtonStyle/ViewModifier 与 #Preview。每一个被宿主的 SwiftUI 叶子变成一个程序化构建的 NSView/NSViewController，靠 configure(...) 显示、靠闭包/delegate 上报。
重新归类（reclassification）迁移的主要动作：把每个现存类型明确落到五层之一（Model / View / Controller / Service·Store / Coordinator）。落不进去、或横跨两层的类型，正是当前隐藏耦合的所在——例如「携带 NSImage 的值」既不是纯 Model 也不是纯 View。
承重例外（load-bearing exception）迁移中刻意不动的硬约束。CCTerm 有两条：NativeTranscript2 自绘 NSTableView 的性能契约，与选择切换的「同源相位同步结构性边」。它们不是技术债，是被测量与时序证明过的正确决定。

1.1 CCTerm 的今天：一层薄骨架撑住的 SwiftUI

先看清起点。今天的 CCTerm 把 AppKit 压到最小，只在五处「SwiftUI 跟不上」的接缝上露出骨架，其余一律 SwiftUI + NSHostingController/NSHostingView 托管。这五处是理解全局的钥匙——它们既是现状的承重墙，也决定了迁移的先后。

这五处恰好覆盖了五层里的每一层：转录是 View + Service（自绘 + 桥接），主窗口根是 Controller + Coordinator，侧栏是 Controller，工具栏是 View/Service 边缘，AppDelegate 是组合根。换句话说，CCTerm 今天已经无意中把最难的接缝做成了 AppKit——迁移不是从零建骨架，而是把剩下的 SwiftUI 叶子换成叶子。认清这一点，才不会把「移植」误当成「重写」。

1.3 三个结构性动作

把上表的落位落到运行时，整场迁移收敛成三个结构性动作。它们是后续每一章的总纲。

动作一：一个组合根

AppDelegate.applicationDidFinishLaunching 成为唯一 new 具体服务、装配对象图的地方（组合根，见规范手册第 4 章）。它构建进程级的 AppContext（旧 AppState → 新 AppContext，且从 @Observable 可变袋降级为只读 struct），通过 MainMenuBuilder 装上完整的 NSApp.mainMenu，再把整张对象图交给根 AppCoordinator。原本赖在 AppDelegate 里的导航/窗口路由决策（Settings/About 辅窗、主窗口构建）上移到协调层。

旧 CCTermApp / AppCommands旧的 SwiftUI @main CCTermApp 场景与 AppCommands（Commands 块）删除。CCTermApp.init 里的启动副作用（模型预取、主线程看门狗）迁入组合根的 assemble()；AppCommands 只覆盖了 3 组菜单 + Find，纯 AppKit 下无人再自动合成 App/Edit/Window/Help，必须由 MainMenuBuilder 手搓整棵 NSMenu 树。
易被忽视的坑：菜单栏与标准编辑动作

SwiftUI 的 App 会自动合成 App/Edit/Window/Help 四组菜单，其中 Edit 组里那些 nil-target 的标准第一响应者选择子（Cut/Copy/Paste/Undo/Redo/Select All）是输入栏和搜索框赖以工作的隐形基础。纯 AppKit 下这些不再自动出现，MainMenuBuilder 必须显式把它们建进去——否则文本编辑与窗口快捷键会静默失效，且不会有编译错误提示你。这是迁移里最容易漏、最难查的一处。

动作二：@Observable → 显式通道

SwiftUI 的自动依赖追踪被第 7.1 节的机制表替代，按基数与方向逐个改写：

  • 窗口级选择：旧 MainSelectionModel（@Observable）→ 新 WindowNavigationStore，其 set(_:) 触发一条同源相位、同步的 delegate（旧 MainSelectionObserver → 新 WindowNavigationObserver）——这是全应用唯一的结构性向上边，原样保留（见 1.5）。
  • 视图派生状态：只有确有可测逻辑的地方才落成 Combine @Published 的 ViewModel（全应用一共只有七个，见后续章节）。
  • 会话运行时可观察性：Session 的 isRunning/status/pendingPermissions/turnUsage 增加显式的同步 onXxxChange 闭包，替换 withObservationTracking。
硬前置：会话可观察性必须先落地

删掉 SwiftUI 就删掉了对 Session.isRunning 等属性的自动追踪。在聊天面板/输入栏能编译之前，Services/Session 域必须先加上显式的同步 onXxxChange 闭包（尤其是在运行时写点触发的 onIsRunningChange）。今天的 TranscriptSwapCoordinator.startRunningObservation 用的是「withObservationTracking 装进一个 Task」——那会在未来某个 source 相位才 resume，本身就与它对外承诺的「同步切换」相矛盾，必须改成一个直接的同步 sink。这是一条跨域的硬前置，不是软建议。

动作三：Coordinator 决策，Router 执行

旧 DetailRouterViewController 一分为二：DetailNavigationCoordinator（决定挂哪个子 VC、注入 DetailContext、承接 app→detail 信号）与 DetailContainerViewController（纯挂载/crossfade 机制，不做任何路由决策）。这正是第 5 章「协调器决策、Router 执行」的落地。两条硬不变量原样穿过：源相位同步切换，与 NativeTranscript2 自绘 NSTableView 的性能契约。

1.4 承重例外一：NativeTranscript2 的性能契约不动

纯 AppKit 化的默认动作是「把自绘/命令式换成声明式」，但转录是反向的例外——它今天就是自绘 NSTableView，迁移后继续自绘，一行不改其渲染路径。这不是没来得及移植，而是刻意保留：List 从来就扛不住转录的行数与自定义布局，这条决定在现状里已经成立，纯 AppKit 只是让它更名正言顺。

应当

转录的更新走一条命令式精确增量通道：桥接器把一个已算好的行 delta 直接交给自绘表，而不是逼它 diff/reload。这条通道同步在运行时写点触发，绝不 hop 过 Task / .receive(on:) / beforeWaiting。

// 承重例外：不走 Combine 声明式管道，而是精确增量的命令式调用
// （旧 Transcript2EntryBridge → 新 TranscriptEntryBridge，"2" 待重命名统一后去掉）
func apply(_ change: MessagesChange)          // 交给自绘表精确行增量，不 reload
func setLoading(_ running: Bool)              // 由 Session.onIsRunningChange 同步驱动
func setTurnUsage(_ usage: TurnTokenUsage)    // 推一个精确值，不触发 diff
避免

把转录更新也塞进 Combine @Published + sink 的声明式管道。@Published 的下行会晚一个 tick（在 beforeWaiting 才到），而转录的 typeset 必须在切换的同一帧以最终宽度完成——晚一帧就是肉眼可见的错位/撕裂。

为什么这条例外是合法的

项目根 CLAUDE.md 明确开了一道口子：「跨 AppKit↔SwiftUI 边界的命令式调用，仅当正确性依赖一个 @Observable 表达不了的 runloop tick 时才允许」。转录正是这种情形——它要么必须跑在点击的 source 相位，要么要把一个精确 delta 直接递给渲染器而非强制 diff。规范手册第 7 章说「一串状态值用 Combine，一个精确渲染器 delta 或时序敏感调用用命令式方法」，转录落在后者。它不是破坏单向流，而是单向流里被显式许可的窄例外。

1.5 承重例外二：唯一的结构性向上边

第二条不动的承重墙，是选择切换那条边。WindowNavigationStore.set(_:)（旧 MainSelectionModel）→ WindowNavigationObserver（旧 MainSelectionObserver）是全应用唯一的结构性向上边：它在侧栏点击/通知激活的同一 runloop source 相位内同步触发，由 DetailNavigationCoordinator 独家注册，同步驱动 container.swap(...) 与 present(sessionID:)。

避免（明确的非目标）

不要把这条边泛化成 NotificationCenter 广播、不要让它走 Combine、不要再加第二个观察者槽。@Observable/Combine 的 re-eval 会晚一个 tick 落在 beforeWaiting，从而把一次会话切换撕裂到两帧。所有其它「对选择做反应」的需求（工具栏项目 chip 的出现、侧栏高亮）都是非结构性 chrome，走闭包下行（onSelectionChanged/onTargetChange），不碰这条结构性 delegate。

为什么是 delegate，而不是 bus 或 Combine

它必须同步、在源相位内完成。macOS 的一个 runloop iteration 里，你的点击处理跑在 source 相位；而 @Observable/Combine 注册的观察回调要等到 beforeWaiting 相位才 re-eval——那已经是「写地几何还没 settle 就去切换」的下一拍。转录只有在切换的同一帧就拿到最终宽度，才能一次 typeset 到位。一条同步 delegate 恰好在源相位内闭合这条边，这就是它不能被降级成异步机制的原因。

submitSessionInput 与 runBuiltinSlashCommand 这两个源头函数，被不同域分别设计成了「自由函数」（inputbar 域，援引第 5.5 节「别把单步无分支流程包成协调器类」）和「协调器类」（chatpane 域，援引可注入的测试缝）。二者不能同时落地，否则 chat 与 compose 两条路径会分叉。本手册按第 5.5 节取自由函数为默认（依赖以参数传入），仅当某个测试缝确有需要时才升级为可注入的类——但整场迁移只能选一次。

1.7 携带 NSImage 的值：一道贯穿全书的边界

把每个类型钉进五层时，有一类值会反复卡在 Model 与 View 之间，值得在总览里就点名，因为它决定了哪些类型能沉入 UI-free 的 Core 包（见第 2 章）。规范手册第 1.2 节明令：Model 不得携带 NSImage。

避免

把携带 NSImage 的值当成 Model 沉进 Core 包。Attachment、OpenInAppService.Target、以及每一个符合 CompletionItem 的补全 Match——它们都 import AppKit，因此是链接了 AppKit 的值，只能与它的 View/Store 同住，进不了 Core。

应当

唯一的解锁动作：把 CompletionItem（→ 新 CompletionCandidate）的 displayIcon: NSImage? 降成一个无 AppKit 的图标描述符（SF-symbol 名 / kind / 路径），由 cell 在渲染时才解析成 NSImage。这一步让 CompletionEngine 与 Services 层的补全器都变成无 AppKit，UI→Services 的依赖箭头才真正指向下。

为什么在总览里就强调它

因为它是「重新归类」这件事最典型的样本：一个看起来像纯数据的值，只要它引用了 NSColor/NSImage，就被 AppKit 缠住、无法测试、无法跨端复用（规范手册第 2 章的核心收益全丢）。第 2 章会把「哪些类型能进 Core 包」讲透；这里先立下判据——看它 import 什么，而不是看它长得像不像数据。同理，SlashCommandTriggerRule/FileMentionTriggerRule 因为要够到 *.shared 缓存，从 Model 重新归类为「消费 Service 的策略」，其 store 依赖必须显式出现在签名上（注入一个补全能力），而不是内联去摸单例——否则分层就倒挂了。

1.8 明确的非目标

这些是刻意不做的事，写在总览里以免后续被当成「顺手清理」误动（与项目现状文档的非目标一致）：

  • 不引入全局 store / Redux / 聊天区协调 ViewModel——那会把进程/窗口/会话三个作用域压平，逼每个转录 delta 过一遍 reducer。
  • 不把 AppContext 整袋经 .environment 注入——按作用域切成 SidebarContext/DetailContext 小包传递。
  • 不让 DetailNavigationCoordinator 用 withObservationTracking 观察选择——它只通过那条同步 delegate 读 navigation（1.5）。
  • 真缓存单例保留（ModelCatalogStore、FileCompletionStore/SlashCommandStore、CoreDataStack）——它们是进程级/按目录的缓存，不是图省事的注入捷径。
本质

CCTerm 的纯 AppKit 化，是一次「让隐式变显式」的重新归类，而不是重写。SwiftUI 曾把三件事托管给框架：分层里 View↔Model 的绑定、依赖的注入、以及跨相位的时序缝合。去掉它，就是把这三件事逐一交回你手上——分层照第 1 章切、依赖照第 4 章在组合根装、绑定照第 7 章按基数与方向手工焊。整场迁移收敛成三个动作（一个组合根、@Observable 转显式通道、协调器决策/Router 执行），外加两条刻意不动的承重例外（NativeTranscript2 的自绘性能契约、唯一那条同源相位同步的结构性选择边）。后续每一章，都是在这张地图上把某个域从旧名字牵到新归宿。

思考题：有人在纯 AppKit 化的过程中提议「既然要去 SwiftUI 的自动追踪，不如把 Session.isRunning 的变化统一走 Combine @Published——聊天面板订阅它切 send/stop 按钮，转录也订阅它调 setLoading，这样机制统一又干净」。请结合本章的两条承重例外与第 7 章的机制选型，指出这个提议错在哪，正确的接法是什么。

查看答案

错在把「时序敏感的同步调用」误当成「一串状态值的下行推送」。 Combine @Published 的 sink 属于第 7.1 节里「1 对多、状态值推送」的机制，它的下行晚一个 tick——在 beforeWaiting 相位才把新值送到订阅者。对于「send/stop 按钮切换」这种非时序敏感的 chrome，晚一帧无所谓，Combine 完全合适。但对转录的 setLoading，它与选择切换/typeset 同属承重例外那条同步链：转录必须在切换的同一帧、以最终宽度完成布局，晚一帧就是可见的错位。用 Combine 统一二者，等于把承重例外降级成异步机制，撕裂就回来了。

还错在无视硬前置。 今天的 startRunningObservation 用「withObservationTracking 装进 Task」，本身就已经在未来某个 source 相位才 resume，与「同步切换」的承诺相矛盾——这正是要被修掉的问题，而不是要被 Combine「延续」的模式。

正确接法：分两条通道，按时序敏感度选机制。 在 Session 运行时写点加一个显式的同步 onIsRunningChange 闭包（本章动作二的硬前置）。转录侧：这个同步闭包直接调 controller.setLoading(running)（承重例外的命令式精确通道，1.4），绝不 hop 过 Task/.receive(on:)。按钮侧：既可以复用同一个同步闭包，也可以（若只想要简单 chrome）走一个不那么时序敏感的下行；关键是不把转录那条同步链交给 Combine。一句话：时序敏感的同步命令走命令式方法，普通状态值才走 Combine——机制不该为了「统一」而牺牲那一帧。

项目结构与模块化

第 1 章把每个类型钉进了五层——那是职责的地图。本章回答正交的另一问：这些类型在磁盘上怎么摆、谁能 import 谁。CCTerm 今天已经「按功能聚合」（Services/Session/、Content/Chat/、Sidebar/ 各自成域），且用 filesystem-synced 组自动纳入构建——这一半规范已经天然达标，本章只需确认它、并补齐真正欠缺的一步：把无 UI 依赖的核心下沉成一个由编译器强制边界的 Core 模块。第 1.7 节立下的那条判据——「看它 import 什么，而不是看它长得像不像数据」——在这里变成一条物理边界：能进 Core 的，是那些编译期就证明不碰 AppKit 的类型；进不去的，其耦合会在 import AppKit 那一行被编译器当场点名。对标规范手册第 2 章。

功能域（feature domain）围绕一件用户可感知的事聚合的一组协作类型。CCTerm 的域已经成形：会话运行时（Services/Session/）、聊天区（Content/Chat/）、侧栏（Sidebar/）、转录渲染（NativeTranscript2/）。改一个需求，diff 几乎总落在同一个域内。
Core 包（UI-free core）一个不链接 AppKit 的独立模块（本地 SPM package 或独立 framework target）。只装纯值、纯算法、纯 I/O 契约。它的测试无需起窗口、无需主 runloop，秒级跑完。
依赖方向（dependency direction）「谁 import 谁」的箭头。健康的方向单向向下：App(UI) → Core，Core 永不反向 import UI。模块边界让这条箭头由编译器强制，而非靠自觉。
filesystem-synced 组CCTerm 的 .xcodeproj 用 PBXFileSystemSynchronizedRootGroup：磁盘上增删移文件自动进出构建，永不手改 project.pbxproj。因此「目录即模块结构」在 CCTerm 是字面成立的。

规范手册 2.1 的头号纪律是「按 feature 而非按文件类型切分」。这一条 CCTerm 早已达标——它没有 Controllers/ / Views/ / Models/ 三大堆，而是每个域自带它的 VC、视图、状态与协调器。迁移不动这套目录，只在其中做「定层 + 改名」。

为什么这一半不用动

SwiftUI → AppKit 是「同一层内换实现」的迁移（第 1 章：分层骨架不变，变的是绑定接线）。功能聚合是目录纪律，与「用什么 UI 框架」正交——一个 Content/Chat/ 域，无论里面是 SwiftUI 叶子还是程序化 NSView，它作为「变更的自然单位」不变。所以迁移期不要顺手重排目录：那只会把「换实现」的 diff 和「搬文件」的 diff 搅在一起，评审时谁也看不清。真正欠账的是下一节——一条物理的、编译器强制的核心边界。

2.2 欠账：把 UI-free 的核心下沉成 Core 包

CCTerm 今天所有代码在同一个 App target 里，编译器不拦任何 import——一个 Markdown 解析器可以顺手 import AppKit、一个 SessionConfig 可以顺手塞进 NSColor，没有任何东西阻止它。规范手册 2.2：要建立硬边界，唯一可靠的办法是把可复用、可测试的核心抽成独立模块。这也正是第 1.7 节那条判据落到磁盘上的样子。

应当

把「编译期即证明不碰 AppKit」的类型下沉进一个本地 SPM 包 CCTermCore，App target 只向下依赖它；包清单把依赖方向钉死，核心 target 不链接 AppKit。

// CCTermCore/Package.swift —— 依赖只能向下，编译器强制
let package = Package(
    name: "CCTermCore",
    products: [
        .library(name: "MarkdownIR", targets: ["MarkdownIR"]),
        .library(name: "TranscriptBlocks", targets: ["TranscriptBlocks"]),
        .library(name: "SessionModel", targets: ["SessionModel"]),
    ],
    targets: [
        // GFM parser → 内部 IR：纯算法，谁都不依赖
        .target(name: "MarkdownIR"),
        // 块翻译（MessageEntryBlockBuilder / StableBlockID / DiffEngine）：依赖 IR
        .target(name: "TranscriptBlocks", dependencies: ["MarkdownIR"]),
        // 会话不可变快照（SessionConfig / Worktree / SessionRecord / PermissionMode）
        .target(name: "SessionModel"),
        // 每个核心 target 自带无窗口测试
        .testTarget(name: "TranscriptBlocksTests", dependencies: ["TranscriptBlocks"]),
    ]
)
避免

让核心类型反向 import AppKit，把边界变成一纸空文。第 1.7 节点名的那几个「携带 NSImage 的伪 Model」正是反例——它们进不了 Core，硬塞就等于把 AppKit 拖进核心。

// ❌ 核心模块里出现 AppKit = 依赖方向已污染，编译不过（这正是我们想要的拦截）
import AppKit
struct CompletionCandidate {
    var displayIcon: NSImage?          // ❌ 钉死在 AppKit，测不了、跨不了端
}
// ✅ 改成无 AppKit 的图标描述符，cell 渲染时才解析成 NSImage（见 ch.1.7）
enum IconDescriptor: Hashable { case sfSymbol(String); case filePath(String) }
struct CompletionCandidate { var icon: IconDescriptor? }   // 可进 Core
NetNewsWire · CotEditor两个成熟纯原生 macOS 项目都用真正的本地 SPM 包立边界：NetNewsWire 把 Articles/ArticlesDatabase/Account 等抽成单向依赖的包（Articles 最底层纯模型，UI 在最顶端）；CotEditor 把「与 UI 无关的编辑核心」「语法包」「AppKit UI 包」分开。CCTerm 的 MarkdownIR / TranscriptBlocks / SessionModel 对应它们的最底层——用 package 边界而非「概念上的模块」把 AppKit 挡在门外。

哪些能进 Core、哪些不能：一张判据表

类型（新名）进 Core？判据（看它 import 什么）
Markdown IR、DiffEngine、块翻译器（MessageEntryBlockBuilder/StableBlockID）能纯算法/纯值，Foundation-only

迁移风险：Core 下沉不是「一次性大搬家」

不要在迁移第一步就把所有候选类型一次性挪进 Core——那会与「去 SwiftUI」的改动叠加成一个无法评审的巨型 diff。顺序是：先在 App target 内把类型定层、改名、去掉 AppKit 依赖（如 CompletionCandidate 的图标描述符改造），确认它已「编译期无 AppKit」，再把文件移进 CCTermCore 让编译器盖章。Block 家族尤其要等它先脱掉 import SwiftUI（ch.8.1 的 V2 去数字化那次原子改名）才能下沉，否则 Core 会被 SwiftUI 污染。

为什么值得建这条物理边界

三重收益，全是第 1.7 节判据的兑现。编译期拦截：核心 target 未链接 AppKit，谁在 MarkdownIR 里写 import AppKit 就当场编译不过——耦合不再靠 code review 肉眼抓，而是被编译器在第一时间点名。可测试性：Core 的测试不起窗口、不要主 runloop，几百个块翻译/diff 用例秒级跑完，正好补上现状文档里「逻辑测试是 PR 门禁」的那部分。增量编译：改一个叶子 VC 不再波及块翻译器；改 MarkdownIR 才重编依赖它的 target。SwiftUI 时代这条边界模糊，是因为 @Observable 让「纯数据」也顺手挂上框架协议；去掉 SwiftUI 正是把它切干净的时机。

规范手册 2.3 / 2.4 的两条——一类型一文件、角色后缀命名——CCTerm 大体已遵守，迁移期只需在改名时顺手对齐：新名字要么带准确的角色后缀，要么（纯模型）不带任何后缀。第 1 章那张落位表其实已经是这条规范的应用。

用模糊的 Manager 后缀掩盖职责。CCTerm 保留 SessionManager 是例外而非通例——它是「会话注册表 + 生命周期编排」，语义确实是「管理一批会话的中枢」，且是既有稳定名。但新造类型别再叫 XxxManager：一个 Manager 到底是 Service、Store 还是 Coordinator？能用精确角色词就替换掉。也别给纯模型硬套 Model/Data 后缀（SessionConfigModel 是冗余，且暗示它有行为）。

2.4 杜绝 Utils/Helpers 大杂烩：就近扩展

规范手册 2.5：Utils.swift / Helpers/ / Common/ 是熵的聚集地——谁都往里塞、谁都不敢删，且天然跨越所有依赖边界（一个 Utils 同时被 Core 和 UI 引用，2.2 的边界立刻失效）。CCTerm 现状文档已把这条内化：侧栏的 SidebarLayout / SidebarTitleSanitizer 被明确称作「就近扩展，非 Utils 大杂烩」。

应当

散落的工具函数按它扩展的类型就近落到对应模块的 extension，或独立成一个职责明确的小类型。放在 Core 里的 String 扩展继承 Core 的依赖约束——它不可能 import AppKit，编译器替你守着。

// ✅ 相对时间格式化：无 AppKit，归 Core（被 ArchiveViewModel / 侧栏派生共用，ch.9.2）
extension Date {
    func shortRelative(now: Date = .now) -> String { /* 相对时间梯 */ }
}
// ✅ 标题清洗：纯字符串逻辑，就近扩展，随侧栏域走
enum SidebarTitleSanitizer { static func clean(_ raw: String) -> String { … } }
避免

把「无状态纯格式化」塞进 ViewModel 当垃圾桶（ch.9.2 tip：NewSessionConfiguratorViewModel 挣位置靠有状态的 Git 校正，「Start Building <name>」「~ 路径折叠」这类无状态纯函数一律降级为自由函数）；更别为它们建一个 ChatUtils.swift。无主题容器一旦被 Core 与 App 同时引用，边界即漏。

本质

CCTerm 的模块化，一半已由现状达标（按功能聚合、filesystem-synced 组、就近扩展），迁移期真正要补的是把 UI-free 核心下沉成一条编译器强制的物理边界。这条边界是第 1.7 节判据（「看它 import 什么」）的兑现：能进 CCTermCore 的，是编译期就证明不碰 AppKit 的纯值与纯算法（Markdown IR、块翻译、会话快照、路由枚举）；进不去的（携带 NSImage 的伪 Model、@MainActor Service、任何 NSView）留在 App target。配合一类型一文件、角色后缀、就近扩展，「变更的局部性」和「依赖的方向性」都由结构本身保证，而不是靠人自觉——去掉 SwiftUI 的 @Observable 之后，这条边界终于能切干净。

思考题：迁移中有人为了「让块翻译器能被单测」，把整个 Content/Chat/NativeTranscript2Bridge/ 目录连同 TranscriptEntryBridge（Service）、Block（今天 import SwiftUI）、MessageEntryBlockBuilder（纯值）一起挪进了新建的 CCTermCore 包，理由是「它们本来就在一个目录里，一起搬最省事」。请结合本章 2.1/2.2 的边界判据与迁移顺序，指出这个动作错在哪，正确做法是什么。

查看答案

错误一：按目录搬，而非按「import 什么」搬（违反 2.2 判据）。「在一个目录里」是功能聚合（2.1）的结果，与「能不能进 Core」（2.2）是正交的两件事。同一个 NativeTranscript2Bridge/ 目录里，MessageEntryBlockBuilder 是纯值（能进），TranscriptEntryBridge 是 @MainActor Service（不能进——它持 weak controller、走命令式 apply，是 UI 侧的喂表入口），Block 今天还 import SwiftUI（暂不能进）。把三者一起搬，等于把 SwiftUI 和 @MainActor Service 拖进本该 UI-free 的 Core——核心 target 一旦要链接 AppKit/SwiftUI，2.2 的编译期拦截就废了。

错误二：忽略迁移顺序（违反 2.2 warn）。Core 下沉不是一次性大搬家。正确顺序是先在 App target 内把类型改造到「编译期无 AppKit/SwiftUI」，再搬：Block 必须先随 ch.8.1 的 V2 去数字化脱掉 import SwiftUI，其纯派生部分才够格下沉；TranscriptEntryBridge 作为 Service 永远留在 App target，不参与下沉。

正确做法：只把已证明 UI-free 的纯派生下沉——MessageEntryBlockBuilder / StableBlockID / MarkdownToBlocks / DiffEngine 进 CCTermCore 的 TranscriptBlocks target；Block 等它去掉 SwiftUI 后再进；TranscriptEntryBridge 留在 Content/Chat/（App target），经初始化器拿到下沉后的纯类型。搬完后在 Core 里为块翻译写无窗口单测——这才是「让块翻译器能被单测」真正想要的收益，且由编译器保证它不再回流 UI。

组合根与依赖注入

前两章我们把 CCTerm 的每个类型归进了五层，又把它们按功能与模块边界摆好了位置。但还有一个悬而未决的问题：这些对象由谁 new 出来、依赖从哪进入？今天的 CCTerm 靠三样东西回答这个问题——SwiftUI 的 @main App 场景负责启动、@Observable 的 AppState 当进程级容器、.environment(...) / EnvironmentKey 把服务塞进视图树。纯 AppKit 里这三样全都不存在。本章把装配收敛到唯一的组合根（AppDelegate），把散落的服务聚成按作用域切分的只读 Context 包，把 .shared 单例压回“真正的进程级缓存”这一极小集合，并用能力协议划出可替换、可测试的接缝。这是 CCTerm 从 SwiftUI 迁往纯 AppKit 的第一根承重梁。

本章相对通用规范的定位

本手册规范篇的第 4 章已经把依赖注入的通则讲透了：初始化器注入、面向协议依赖、组合根一处装配、Context 聚合、单例的举证责任。本章不复述这些通则，而是把它们具体落到 CCTerm 的每一个真实类型上——回答“旧 AppState 变成什么”“ModelStore.shared 保不保留”“injectDetailEnvironment 删了之后依赖怎么流”这类迁移问题。读本章前请先掌握规范 ch.4。

组合根（Composition Root）整个应用中唯一 new 具体类型、把它们连成对象图的地方。CCTerm 里就是 AppDelegate.applicationDidFinishLaunching(_:) 及其调用的 assemble()。
作用域 Context 包把“会一起向下传递、生命周期一致”的依赖聚成一个只读 struct。CCTerm 分三级：进程级 AppContext、窗口级 SidebarContext / DetailContext、会话级 InputBarContext。
合法单例（legitimate singleton）语义本就是“每进程/每工作目录唯一一份”的昂贵缓存或系统资源，用 .shared 表达是诚实的，且必须写明理由。CCTerm 只有四个。
能力协议（capability protocol）描述“做什么”而不暴露“怎么做”的协议，是测试替身与实现替换的插入口。只为“会被替换 / 跨模块 / 需打桩”的依赖抽取。

3.1 组合根：AppDelegate 是唯一装配点

今天的 CCTerm 用 SwiftUI 的 @main struct CCTermApp: App 做入口，靠 @NSApplicationDelegateAdaptor 把 AppDelegate 挂进来，再用一个 Settings { EmptyView() } 占位场景承载 .commands。纯 AppKit 没有 App / Scene 协议，这一整套要塌缩成传统 AppKit 入口：一个 main.swift（或 AppDelegate 上打 @main）把 NSApplication.shared.delegate 设好，装配全部搬进 AppDelegate。

CCTermApp（SwiftUI @main App）旧的 @main App 结构体连同 Settings{EmptyView()} 占位场景、@NSApplicationDelegateAdaptor 全部溶解成一个常规 AppKit 入口。CCTermApp.init 里的启动副作用（ModelStore.shared.prefetchIfNeeded()、MainThreadWatchdog.start()、NSInitialToolTipDelay）迁进 AppDelegate.applicationDidFinishLaunching。
AppDelegate（旧：@NSApplicationDelegateAdaptor 挂载，兼当组合根 + 持有 AppState/TranscriptSearchBus/MainSelectionModel/各窗口控制器）保留其组合根身份，但卸掉“开哪个窗口”的路由决策（交给 AppCoordinator，见 ch.5）。它只保留真正的 NSApplication 生命周期（启动、terminate 并行关停、reopen 转发），外加“build 对象图 + 装 NSApp.mainMenu”。
应当

把所有 new 具体类型 + 连线的代码集中到 applicationDidFinishLaunching 调用的 assemble()。这里、且只有这里，同时认识各具体实现；再把成品自上而下交给根 Coordinator。

@MainActor
final class AppDelegate: NSObject, NSApplicationDelegate {
    private var rootCoordinator: AppCoordinator?          // 唯一强持有的根

    func applicationDidFinishLaunching(_ notification: Notification) {
        assemble()
    }

    private func assemble() {
        // ① 启动副作用（旧 CCTermApp.init 里的三件事）
        ModelCatalogStore.shared.prefetchIfNeeded()
        MainThreadWatchdog.start()

        // ② new 出进程级服务（此处才出现具体类型）
        let context = makeAppContext()

        // ③ 服务级副作用接线（旧 AppState 里做的）
        context.notificationService.onActivateSession = { [weak self] id in
            self?.rootCoordinator?.activateSession(id)
        }
        context.notificationService.bootstrap()
        Task { await context.syntaxEngine.load() }
        context.openInService.refresh()

        // ④ 窗口级状态 + 命令
        let navigation = WindowNavigationStore(target: .none)
        let searchCommand = TranscriptSearchCommand()

        // ⑤ 手搓整棵菜单树（SwiftUI 不再替你合成，见 3.5）
        let root = AppCoordinator(context: context,
                                  navigation: navigation,
                                  searchCommand: searchCommand)
        NSApp.mainMenu = MainMenuBuilder.makeMainMenu(target: root)

        // ⑥ 启动根流程
        self.rootCoordinator = root
        root.start()
    }

    // macOS-26 上 @MainActor deinit 的执行器跳转会崩溃，保留空的 nonisolated deinit
    nonisolated deinit {}
}
避免

把“开设置窗口 / 开关于窗口 / 造主窗口”的决策留在 AppDelegate（旧 showSettingsWindow / showAboutWindow）。那是导航，不是生命周期——它属于 AppCoordinator（ch.5）。AppDelegate 只负责“把图装出来、把根 Coordinator 交出去”，之后就退场。

为什么组合根只此一处

组合根是唯一被允许知道“具体用哪个实现、谁持有谁、谁活多久”的地方（规范 ch.4.4）。CCTerm 迁移期尤其吃这个纪律：旧代码里启动副作用散在 CCTermApp.init、服务接线散在 AppState、窗口路由散在 AppDelegate——三处各改各的，谁也说不清对象图。收敛到一个 assemble() 后，读一个函数就读懂整张图，换实现、注入替身、追生命周期都有单一落点。

3.2 AppState → AppContext：从可变袋子到只读清单

CCTerm 现在的进程级容器 AppState 是个 @Observable @MainActor final class，装着八个服务，还顺手在里面接线（turn-end/permission 通知、eager 语法加载、open-in 探测）。纯 AppKit 里 SwiftUI 消失了，观察机制也就没了存在理由；同时规范 ch.4.5 说得很清楚：Context 是只读的依赖清单，不是共享可变状态容器。所以 AppState 要降级成一个普通 struct AppContext。

AppState（旧：@Observable @MainActor final class，八服务 + 接线副作用）旧 AppState → 新 struct AppContext。去掉 @Observable（它本身不持有任何可观察状态，只持有一组服务引用）；那些接线副作用（onTurnEndedNotice、eager syntax load、open-in 探测）上移进 AppDelegate.assemble()，让 Context 退回纯清单。这与既有的 SidebarContext / DetailContext 已经是纯 struct 的现实对齐。
应当

进程级依赖聚成一个只读 struct，所有成员 let，没有副作用方法。

@MainActor
struct AppContext {
    let sessionManager: SessionManager
    let syntaxEngine: SyntaxHighlightingService     // 旧 SyntaxHighlightEngine
    let recentProjects: RecentProjectsStore
    let inputDraftStore: InputDraftStore
    let sidebarGroupOrder: SidebarSessionGroupOrderStore
    let activationTracker: AppActivationTracker
    let notificationService: NotificationService
    let openInService: OpenInAppService
}
避免

把 AppContext 当“全局可变袋子”——保留 @Observable、让下游既读又写它、或往里塞运行时可变状态。可变的窗口/会话状态属于各自作用域的 Store（如 WindowNavigationStore），不混进依赖清单。Context 只回答“有哪些服务可用”，不回答“当前选中了谁”。

本质

@Observable 是 SwiftUI 的自动依赖追踪机制；纯 AppKit 用显式渠道取代它（Combine @Published、闭包、delegate——详见状态与数据流一章）。因此“把 @Observable 从 AppState 上摘掉”不是丢功能，而是把一个本就不该观察的纯依赖清单还原成 struct；真正需要被观察的状态各回各的 Store，用各自的显式渠道发布。

3.3 作用域切分：三级 Context 包

规范 ch.4.5 要求按作用域切 Context：依赖活多久，就归到多大的作用域。CCTerm 已经天然是三级——这套结构在 SwiftUI 期就存在，纯 AppKit 只是把“注入方式”从 environment 改成 initializer。

进程级   AppContext        ← AppDelegate.assemble() 造，活得和 App 一样长
   │  MainSplitViewController 拆包
   ├─ 窗口级 SidebarContext  ← 侧栏这一栏需要的子集
   └─ 窗口级 DetailContext   ← detail 侧各子 VC 需要的子集
                │  InputBarViewController 再收窄
                └─ 会话级 InputBarContext  ← 输入栏 + 补全能力
MainSplitViewController（旧：从 AppState 拆出 SidebarContext / DetailContext，并 injectDetailEnvironment 注入 SwiftUI 环境）保留其“把 AppContext 拆成两个更窄的作用域包”的职责——这正是规范 ch.4.5 的现实落地。变化只在下发方式：detail 子 VC 从初始化器拿 DetailContext，而不再靠 injectDetailEnvironment 注入 SwiftUI 环境。
应当

系统容器 VC（MainSplitViewController）在装配子 VC 时，从进程级 AppContext 取出各作用域需要的成员，打成更窄的包，经 init 交给子 VC。

@MainActor
struct SidebarContext {
    let navigation: WindowNavigationStore      // 旧 model: MainSelectionModel
    let sessionManager: SessionManager
    let groupOrderStore: SidebarSessionGroupOrderStore
    let openInService: OpenInAppService
}

@MainActor
struct DetailContext {
    let navigation: WindowNavigationStore
    let sessionManager: SessionManager
    let recentProjects: RecentProjectsStore
    let inputDraftStore: InputDraftStore

@MainActor
final class MainSplitViewController: NSSplitViewController {

    init(navigation: WindowNavigationStore, context: AppContext) {
        self.navigation = navigation

    @available(*, unavailable) required init?(coder: NSCoder) { fatalError() }

    override func loadView() {

        let sidebarCtx = SidebarContext(navigation: navigation,
                                        sessionManager: context.sessionManager,

                                        openInService: context.openInService)
        let detailCtx = DetailContext(navigation: navigation,
                                      sessionManager: context.sessionManager,
                                      recentProjects: context.recentProjects,

    }

View.injectDetailEnvironment(_:)（旧：一次把四个 detail 服务注入 SwiftUI @Environment）删除。SwiftUI 环境注入在纯 AppKit 没有对应物——依赖靠初始化器流动。旧代码每个子 View 调一次 injectDetailEnvironment，现在替换成把 DetailContext 传进子 VC 的 init；这恰是该功能在 SwiftUI 之前的原始形态。
为什么按作用域切、而不是一个大包传到底

依赖的生命周期就是它的作用域。用一个大而全的 AppContext 传到每个叶子 VC，会让“短命的输入栏”看得见“进程级的语法引擎、通知服务、激活追踪”——耦合面无谓地拉满，也让“谁持有谁”变模糊。切成 SidebarContext / DetailContext / InputBarContext 后，每层只拿到它作用域内该看到的东西，图更窄、销毁边界更清晰、评审时“这个 VC 凭什么摸得到语法引擎”一眼可查。

3.4 单例：CCTerm 只保留四个合法单例

规范 ch.4.6 给单例设了“举证责任”：只有确有进程级唯一性理由的对象才配 .shared，且必须写明理由。CCTerm 现有一批 static let shared，迁移时要逐一过筛——哪些是诚实的进程级缓存（留下），哪些只是“懒得传”的伪全局（回组合根经 init 注入）。

类型（旧名）裁决理由
ModelStore → ModelCatalogStore保留 .shared启动 fork 一个 CLI 子进程枚举可用模型，昂贵、与任何窗口无关——规范 ch.4.6 点名的“启动子进程枚举清单缓存”原型
FileCompletionStore保留 .shared按工作目录共享的文件补全缓存，spawn git/find/fzf 子进程 + FSEvents 增量
SlashCommandStore保留 .shared按路径共享的斜杠命令缓存，spawn 临时 CLI
CoreDataStack保留 .shared每进程一份持久化存储是诚实的单例理由；生产走 private init()，测试走 init(inMemory:) 接缝
EffortDefaultStore → EffortPreferenceStore去 .shared业务偏好持久化，不是进程缓存——组合根经 init(defaults:) 注入
NewSessionDefaultsStore → NewSessionPreferenceStore去 .shared同上，用户偏好而非缓存
RecentProjectsStore去 .sharedApp 业务状态，组合根经 init 注入并进 AppContext
应当

合法单例在源码里写明“它为什么是单例”，并通过其唯一持有者（如 SessionRepository 持 CoreDataStack、CompletionPrewarmer 门面统辖两个补全缓存）被触达，而不是被业务 VC 到处 .shared 直取。

/// 进程级缓存：启动时 fork 一个 CLI 子进程枚举可用模型，结果昂贵、
/// 与任何具体窗口无关，全进程共享一份。这是它作为单例的“理由”。
@MainActor
final class ModelCatalogStore {
    static let shared = ModelCatalogStore()          // ch.4.6 合法单例
    @Published private(set) var models: [ModelInfo]
    @Published private(set) var isLoading: Bool
    private init() { … }
    func prefetchIfNeeded() { … }
    nonisolated deinit {}
}
避免

把 EffortPreferenceStore / NewSessionPreferenceStore / RecentProjectsStore 继续当 .shared 用。它们是用户偏好/业务状态，没有“语义本就全局”的举证——单例只会让它们绕过对象图、无法在测试间隔离、依赖彻底隐形。它们都已具备 init(defaults:)，直接在 assemble() 里造出来注入即可。

迁移风险：合法单例不能沦为隐藏注入捷径

四个保留的 .shared 各自要带一句理由注释；评审时须确认没有业务代码把它们当“懒得传”的后门——补全缓存应经 CompletionPrewarmer 触达、CoreData 应经注入的 SessionRepository 触达，而不是在 VC 里散落 FileCompletionStore.shared.complete(...)。TranscriptSearchCommand（旧 TranscriptSearchBus）是窗口生命周期的命令持有者，由 AppCoordinator/窗口持有，不是单例。

3.5 一个 SwiftUI 迁移的连带账：菜单栏必须手搓

这是组合根迁移里一个容易漏、且漏了会静默出问题的点，必须在本章讲清。SwiftUI 的 App 协议会自动合成标准主菜单（App 的 Hide/Quit、Edit 的 Cut/Copy/Paste/Undo/Redo/Select All、Window、Help）；旧的 AppCommands（一个 SwiftUI Commands 块）只覆盖了 .appInfo（关于）、.appSettings（⌘,）并新增了 Find（⌘F）。纯 AppKit 没有这套自动合成——如果你只搬了 AppCommands 那三项，其余菜单会整个消失，输入栏和搜索框依赖的文本编辑快捷键（复制/粘贴/撤销）会静默失效。

AppCommands（旧：SwiftUI Commands，仅覆盖 .appInfo/.appSettings + 新增 Find，其余靠 App 协议自动合成）旧 AppCommands → 新 MainMenuBuilder。纯 AppKit 无自动合成，MainMenuBuilder 必须手搓整棵 NSMenu 树：App / Edit / Window / Help 全套，外加自定义 Find。AppDelegate 把它赋给 NSApp.mainMenu。
应当

写一个无状态的菜单工厂 MainMenuBuilder 构造全树：标准编辑项用第一响应者 selector + nil target（沿响应链路由到聚焦的输入栏/搜索框），自定义项（关于/设置/Find）target 指向 MainMenuActionTarget（由 AppCoordinator 实现）。

@MainActor
enum MainMenuBuilder {
    /// 构造完整主菜单：App / Edit / Window / Help + 自定义 Find。
    static func makeMainMenu(target: MainMenuActionTarget) -> NSMenu { … }
}

@MainActor
protocol MainMenuActionTarget: AnyObject {
    func openAboutWindow(_ sender: Any?)
    func openSettingsWindow(_ sender: Any?)
    func focusTranscriptSearch(_ sender: Any?)     // ⌘F
}
为什么这属于组合根的账

MainMenuBuilder 只组装菜单结构、不做导航决策，所以按规范 ch.1.2（Coordinator 只管流程、不 build UI）它是一个无状态工厂（Extension/Util 层）；但它装在哪、target 指向谁是组合根的事——AppDelegate.assemble() 在造完 AppCoordinator 后立刻 NSApp.mainMenu = MainMenuBuilder.makeMainMenu(target: root)。导航决策（打开哪个窗口）留在 AppCoordinator 经 MainMenuActionTarget 承接。菜单是启动装配的一部分，漏了它 App 能跑但残废，所以必须和组合根一起交付。

TranscriptSearchBus（旧：@Observable，focusRequestCounter 自增 + withObservationTracking 监听）旧 TranscriptSearchBus → 新 TranscriptSearchCommand。那个“计数器自增 + 观察”的 dance 只为把 SwiftUI 菜单跨越可重建的订阅边界桥到 AppKit 搜索框。纯 AppKit 里 ⌘F 菜单项直接 target 到当前窗口的搜索控制器：TranscriptSearchCommand 只弱持一个 activeSearchController 并 forward focus()——一个窗口生命周期的命令持有者，由组合根/协调器持有。

3.6 协议接缝：只为“会被替换”的依赖抽取

规范 ch.4.3 要求面向协议依赖、ch.4.8 强调协议是测试替身的插入口；但同一节的红线是“不要过度协议化”。CCTerm 迁移里，协议缝要精准打在会被打桩、会跨 actor、会跨模块的地方，其余保持具体类型。

SyntaxHighlightEngine（旧：actor，经 EnvironmentValues.syntaxEngine 注入）旧 SyntaxHighlightEngine → 新 SyntaxHighlightingService（actor）藏在普通协议 SyntaxHighlighting 之后。关键陷阱：该协议不能标 @MainActor——Swift 的 actor 无法遵从全局 actor 隔离的协议（error: actor cannot conform to global-actor-isolated protocol）。async 需求自带跨 actor 跳转，actor 自身的隔离管执行。旧的 SyntaxEngineKey / EnvironmentValues.syntaxEngine 两个 SwiftUI 环境接缝一并删除，改初始化器注入。
应当

为跨 actor / 会打桩的服务抽能力协议；actor 实现遵从非 @MainActor 的协议，用 async 需求承载跳转。

// 注意：不能标 @MainActor —— actor 无法遵从全局 actor 隔离的协议
protocol SyntaxHighlighting: AnyObject, Sendable {
    func load() async
    func highlight(code: String, language: String?) async -> [SyntaxToken]
}

actor SyntaxHighlightingService: SyntaxHighlighting {
    func load() { … }                              // actor 上同步，调用方 await 跳转
    func highlight(code: String, language: String?) async -> [SyntaxToken] { … }
}

// 另一个例子：把 NSApp 活动态抽成协议，让 NotificationService 能用桩测试
@MainActor
protocol AppActivationChecking: AnyObject { var isAppActive: Bool { get } }

@MainActor
final class AppActivationTracker: AppActivationChecking {
    @Published private(set) var isAppActive: Bool
    nonisolated deinit {}
}
// NotificationService.init(activation: AppActivationChecking) —— 测试传桩，不碰 NSApp
避免

给纯值模型（PermissionMode、SyntaxToken、InputDraft）、或本进程内绝不会有第二种实现的小类型硬抽协议。它们不跨边界、不需替身，抽协议只增样板与跳转成本。规范是“为替换而抽象”，不是“为抽象而抽象”。

迁移期最高风险的协议缝：WindowNavigationStore 的观察路径

旧 MainSelectionModel 是 @Observable，被本域外的多个 SwiftUI 消费者读取（侧栏 cell、输入栏 chrome、compose 配置器）。摘掉 @Observable 只有在这些读者都已迁到 AppKit 后才安全；否则要么 Store 暂留一条兼容观察路径，要么迁移排序为侧栏/detail 优先。这是整个切片里最重的跨域耦合，组合根改动必须与这些消费者的迁移协同排期。而那条唯一的同步结构化边（set(_:) → 路由器，旧 select(_:)）无论如何保持不变——它绝不能变异步（详见导航与状态章）。

本质

本章把 CCTerm 的启动装配从“SwiftUI 三处分散”收敛成“一处组合根”：启动副作用、服务 new、接线、菜单、根流程全在 AppDelegate.assemble() 一个函数里；AppState 从可变的 @Observable 袋子降级成只读的 AppContext 清单；依赖顺持有链、按作用域切成三级 Context 包经初始化器下发，injectDetailEnvironment 这类 SwiftUI 环境注入被删除；.shared 被压回四个诚实的进程级缓存，其余偏好/业务状态回组合根注入；协议缝只打在跨 actor、会打桩、跨模块的接缝上。做到这些，你面对的是一张能读懂、能替换、能测试的对象图，而不是一团藏在 @Observable 与 .shared 背后、谁也不敢动的隐式全局状态。

思考题：迁移中有人把旧 AppState 直接改名成 AppContext 但保留了 @Observable，理由是“反正 ModelCatalogStore 的 models 会变，视图要跟着刷新，得让 Context 可观察”；同时他为了省事，让 GeneralSettingsViewController 里直接写 EffortPreferenceStore.shared.remember(...)。请结合本章的 Context 只读原则、单例举证责任、以及“可观察状态归谁”，指出两处错误并给出修法。

查看答案

错误一：给 Context 保留 @Observable（违反 3.2 / 规范 ch.4.5）。 Context 是只读依赖清单，它本身不持有任何可变状态——它只持有服务的引用。ModelCatalogStore.models 会变，那是 ModelCatalogStore 这个 Store 的可观察状态，不是 AppContext 的。让 Context 可观察是把“依赖清单”和“可变状态”两件事搅在一起。
→ 修法：AppContext 退回纯 struct（成员全 let，无 @Observable）。需要刷新模型列表的视图，其 VC 直接 sink context.modelCatalog.$models（Combine @Published）——观察的是那个 Store，不是 Context。

错误二：EffortPreferenceStore.shared 直取（违反 3.4 / 规范 ch.4.6）。 EffortPreferenceStore 是用户偏好持久化，没有“语义本就进程级”的举证——它不该是单例，本章明确把它从 .shared 降级为组合根经 init(defaults:) 注入。VC 里 .shared 直取制造隐藏依赖、绕过对象图、让测试无法隔离（一个测试写脏了 UserDefaults，下一个受影响）。
→ 修法：在 assemble() 里 new 出 EffortPreferenceStore(defaults:)，经 SettingsStore / 相应 Context 注入 GeneralSettingsViewController；VC 通过 init 拿到它，用户改动走 action-up 调用注入进来的实例，而非全局单例。

本质。 两处错误共享同一个病根——把“状态”和“依赖供给”混为一谈。Context 只供给依赖（只读、不可观察），可变状态归各自的 Store（可观察、经显式渠道发布），供给方式一律是组合根注入而非全局单例。分清这三者，迁移期的对象图才不会重新塌回“可观察大袋子 + 满地 .shared”的旧形状。

控制器体系与容器化

前三章把 CCTerm 的类型摊平进五层、划出 Core 包边界、并把每个类型钉在某一层上。本章把镜头对准控制器骨架：谁拥有窗口（NSWindowController）、谁拥有一屏／一区的 UI（NSViewController）、以及怎样用容器化把会膨胀成千行的路由控制器拆成一组各司其职、可确定性拆卸的子控制器。CCTerm 的这套骨架本就是 AppKit——这一层几乎不是「移植」，而是「归位」：把当前混在一起的「决定展示谁」与「怎样挂载谁」拆成两个对象，把 SwiftUI 托管消融成子 VC 直接四边钉住的原生视图，并保住那条唯一的、同源相位（source-phase）同步换面板的结构边。对标手册规范以纯 AppKit 手册第 3 章为准。

WindowController拥有一个 NSWindow 生命周期的控制器：程序化建窗、frame 持久化（autosave）、工具栏（NSToolbar）装配、窗口级菜单响应，并以 contentViewController 挂载内容根 VC。
系统容器 VCNSSplitViewController / NSTabViewController 一族：用 NSSplitViewItem 把每个分栏包成子 VC，自动处理拖拽分隔条、折叠、源列表外观与分隔条位置持久化。
自定义容器 VC系统容器不满足需求时（例如「按选择动态换掉单个子 VC」的路由容器）自己写的容器：手动 addChild / removeFromParent + 四边钉住子视图。
每-attach 资源子 VC 每次被装入容器时才创建、随移除即释放的资源（Combine 订阅、in-flight 任务、per-attach 滚动视图、sheet presenter、isRunning 观察）。生命周期绑定「在场／离场」，而非对象的 ARC 存活。

4.1 WindowController 与 VC：一条职责红线

纯 AppKit 里窗口层逻辑必须只有一个归属。CCTerm 现有的窗口骨架已是这个形状，本章只做两件事：确认它、并把散在里面的 SwiftUI 与导航决策各归各位。

MainWindowController旧 MainWindowController（NSWindowController, NSToolbarDelegate）→ 新名不变，仍是 WindowController 层：拥有 NSWindow + NSToolbar、按选择增删「项目 chip / 归档过滤」工具栏项、承接 frame autosave。它只是脱掉 SwiftUI——两个 NSHostingView 工具栏 chip 变成程序化 NSView（旧 TranscriptProjectChip → 新 TranscriptProjectChipView、旧 ArchiveFilterToolbarButton → 新 ArchiveFilterToolbarView）。
应当

WindowController 独占窗口级职责（窗口、工具栏、autosave、增删条件工具栏项）；VC 只管一屏／一区的视图树。「打开哪个辅助窗口」这类导航决策不属于窗口层，上移到 Coordinator（第 5 章）。

@MainActor
final class MainWindowController: NSWindowController, NSToolbarDelegate {
    let navigation: WindowNavigationStore  // 窗口作用域选择状态（旧 MainSelectionModel）
    let context: AppContext                // 进程作用域依赖包（旧 AppState）
    let searchCommand: TranscriptSearchCommand   // ⌘F 焦点命令（旧 TranscriptSearchBus）

         context: AppContext,
         searchCommand: TranscriptSearchCommand)

                 itemForItemIdentifier: NSToolbarItem.Identifier,

    nonisolated deinit {}                  // 见 §4.6
}
避免

在 AppDelegate 里裸持有 NSWindow 并把「关窗确认 / autosave / 打开设置窗口」写在那里；也别让 VC 反向操纵自己的 window 生命周期。CCTerm 旧的 AppDelegate.showSettingsWindow / showAboutWindow 正是这条越权——它们搬到 AppCoordinator（第 5 章），AppDelegate 只保留真正的 NSApplication 生命周期。

为什么把 hosting 从工具栏拆掉

contentViewController 是关键接缝：窗口把「内容尺寸从哪来」整个下放给根 VC。两个 SwiftUI chip 曾经作为 NSHostingView 挂在工具栏槽里（属于第 6/7 章要消灭的 hosting）；换成程序化 NSView 后，工具栏项的自量高由 NSToolbar 负责，chip 的派生逻辑（目录名、同步 git 分支探测、非历史时折叠）从视图里抽到 TranscriptProjectChipViewModel——视图变笨，窗口层不再夹带一次 SwiftUI body 求值。

4.2 系统容器：MainSplitViewController 组合两个子 VC

CCTerm 的主窗口是经典两栏：侧栏 + 详情。这正是纯 AppKit 手册 3.3 的系统容器范式——一个 NSSplitViewController 装两个各自独立的子 VC，各自拥有数据源与选择状态，绝不塞进一个巨型 VC。

MainSplitViewController旧 MainSplitViewController（NSSplitViewController）→ 保留原样。它是把 AppContext 拆成两个更窄作用域包的地方：旧 SidebarContext（喂侧栏 VC）与旧 DetailContext（喂详情侧子 VC），二者都保留（第 3 章的作用域切分依赖包落地）。这里没有任何 SwiftUI。
@MainActor
final class MainSplitViewController: NSSplitViewController {
    let navigation: WindowNavigationStore
    let context: AppContext
    let detailContainer: DetailContainerViewController   // 见 §4.3

    init(navigation: WindowNavigationStore, context: AppContext)

    override func loadView() {
        // sidebarItem = NSSplitViewItem(sidebarWithViewController: sidebarVC)
        //   ↑ 系统赋予源列表外观 + 折叠动画；addSplitViewItem 已内含 addChild

        // splitView.autosaveName = "MainSplit"（分隔条位置持久化）

不要重复 addChild

addSplitViewItem(_:) 会「顺带」把子 VC addChild 进来，splitViewItems 与 children 始终同步。手册 3.3 已点名：用系统容器时不要再手动 addChild，否则父子关系登记两遍。

4.3 自定义容器：DetailRouterViewController 一拆为二

CCTerm 详情侧是「按侧栏选择动态换掉单个子 VC」的路由——系统 split/tab 都不提供这种语义，所以它是一个自定义容器。旧的 DetailRouterViewController 把两件截然不同的事焊在一起：决定换哪个子 VC（会话／compose／草稿落地／归档／DEBUG demo），与执行挂载（addChild、四边钉住、crossfade）。本章按纯 AppKit 手册 5.2 的「决策／执行分离」把它拆成两个对象。

DetailRouterViewController旧 DetailRouterViewController（单一自定义容器 VC，既决策又执行）→ 拆为 DetailNavigationCoordinator（决策：为一个选择挑出并构造子 VC、注入 DetailContext）+ DetailContainerViewController（执行：addChild / 四边钉住 / crossfade / 拆卸，不做任何路由决策）。旧内部嵌套 DetailRouterViewController.ChildKind → 提升为顶层 Model DetailScene（复用 vs 换面板的判定值）。
// 执行半：纯挂载机制，零路由决策（手册 3.3 自定义容器）
@MainActor
final class DetailContainerViewController: NSViewController {
    override func loadView() { view = NSView() }

    private var currentChild: (NSViewController & DetachableDetailChild)?

    /// 换到某个已构造好的子 VC —— 顺序固定，见 §4.4
    func swap(to child: NSViewController & DetachableDetailChild,
              scene: DetailScene, animated: Bool)
    nonisolated deinit {}
}

// 决策半：一个「叶子」Coordinator（第 5 章），决定「展示谁、用什么依赖造」
@MainActor
final class DetailNavigationCoordinator: WindowNavigationObserver {
    private unowned let container: DetailContainerViewController
    private let context: DetailContext

    // 纯函数：选择 → 基础场景（可单测，无需挂窗口）
    static func baseScene(for target: NavigationTarget) -> DetailScene
    // SessionManager 感知的细化：一个 .session 到底是活动会话还是草稿落地
    func scene(for target: NavigationTarget) -> DetailScene
    nonisolated deinit {}

应当

让容器只回答「怎样挂载」，让 Coordinator 回答「挂载谁、拿什么依赖造」。加入时 addChild → 加 view → 四边约束；移除时先拆卸资源（§4.5）→ 移 view → removeFromParent。子 VC 由 Coordinator 用 DetailContext 初始化器注入构造。

避免

把「baseScene(for:) 判定」写进容器 VC——那会让容器同时知道所有子 VC 的类型与依赖，退回旧 DetailRouterViewController 的巨石。容器不该认识 SessionManager，更不该 new 任何具体子 VC。

为什么值得拆

决策是「流程知识」，执行是「视图机制」，二者变更的理由完全不同：加一个 DEBUG demo 面板只动决策半，改 crossfade 时长只动执行半。分开后 baseScene(for:) 成了一个纯函数，不挂窗口就能单测「选择 X 应该去场景 Y」；而挂载顺序、四边钉住这类易错机制集中到一个容器，读一处即懂。这也是手册 5.2 里 Firefox「coordinator 决策、Router 执行」思路在 CCTerm 的落地。

子 VC 归位：SwiftUI 托管 → 四边钉住的原生视图

三个 fill-pane 详情子 VC 保留身份，只把内容从 SwiftUI hosting 换成程序化 NSView。旧的 mountFillPaneHost（NSHostingController + sizingOptions=[] + 四边钉住的整套配方）随 SwiftUI 一并消融：子 VC 直接在 loadView 里把自己的内容 NSView 四边钉住即可，没有 hosting controller、没有 sizing-options 体操、没有窗口塌陷风险。

ChatSessionViewController / ComposeSessionViewController / DraftSessionLandingViewController三者均保留（NSViewController，各自不同生命周期，故 compose 与 draft-landing 不合并——这是显式非目标）。旧 mountFillPaneHost(_:in:)（泛型 SwiftUI 托管自由函数）→ 删除，或降级为一个极薄的 NSViewController.embedFilling(_:) 容器辅助（addChild → 四边钉住），仅当每处调用点都要重复四边钉住时才保留。

4.4 loadView 建树，viewDidLoad 绑定

纯 AppKit 手册 3.2 最值得立规的一条：loadView() 只建视图树，viewDidLoad() 才做绑定与订阅。CCTerm 的详情子 VC 一律遵此分工。

应当

loadView()：建视图树、四边钉住内容、赋 view；程序化重写不调 super.loadView()（默认实现会去按 nib 名加载归档，白跑一遍）。viewDidLoad()：dataSource/delegate 绑定、Combine 订阅、per-attach 资源装配（此时视图树已完整）。

@MainActor
final class ChatSessionViewController: NSViewController, DetachableDetailChild {
    let context: DetailContext
    private var swapCoordinator: TranscriptSwapCoordinator?

    init(context: DetailContext)

    override func loadView() {
        // 只建树：scrims + 输入条 chrome host + 权限卡 overlay，四边／底部锚定；赋 view
        // 不订阅、不 attach transcript、不调 super
    }

    // 路由器在挂载并定帧后同步调用（见 §4.5 的同源相位换面板）
    func present(sessionID: String?, animated: Bool)
    func prepareForRemoval()
    nonisolated deinit {}
}
避免

在 loadView() 里做 transcript attach、Combine 订阅、磁盘/CLI 请求；也别把子视图声明成 var host: NSView! 再到处判空。前者让「建树」与「生命周期语义」耦合、可能在视图未就绪时就收到数据回调；后者把时序错误从编译期推迟到崩溃时。

回调频次CCTerm 里放什么
loadView()一次（惰性）建 scrims / restingBar host / permissionCardHost 树、四边约束、赋 view
viewDidLoad()一次一次性配置（compose 的 ensureDraftSession()、从草稿配置视图）
present(sessionID:)每次换会话路由器同步驱动 transcript attach / rebind（不是 viewWillAppear，是路由器显式调）
will/did Appear 一族必须调 super

重写 viewWillAppear() / viewDidAppear() 时必须调对应 super，否则掐断系统在父子容器间传递外观事件的链路，子 VC 收不到 appear。这与程序化 loadView() 的「不调 super」正好相反——两条规则方向相反，别记混。

4.5 确定性拆卸：prepareForRemoval 与同源相位换面板

容器化的收益要靠「拆卸也确定性」兑现。CCTerm 的 crossfade 会让离场子 VC 在 removeFromParent() 之后仍被容器短暂持有约 300ms——这段时间里它的订阅、in-flight backfill、isRunning 观察若还在跑，就可能对已离场的自绘表做 reloadData。手册 3.4 的答案是：给子 VC 一个显式拆卸钩子，容器在移除前显式调用，绝不寄望 deinit。

DetailRouterChild → DetachableDetailChild旧 DetailRouterChild（@MainActor protocol: NSViewController，唯一方法 prepareForRemoval()）→ 更名 DetachableDetailChild，对齐手册 3.4 的 DetachableChild 词汇；语义不变。另拆出更窄的 SessionPresentingDetailChild（只含 present(sessionID:animated:)），只有会话／草稿落地两个子 VC 实现它——路由器按类型分派时用它而非旧的到处 as? 强转。
@MainActor protocol DetachableDetailChild: NSViewController {
    func prepareForRemoval()          // 容器移除前显式调用，释放 per-attach 资源
}
@MainActor protocol SessionPresentingDetailChild: DetachableDetailChild {
    func present(sessionID: String?, animated: Bool)
}

// 容器侧：先拆卸资源，再断父子关系
extension DetailContainerViewController {
    func swap(to next: NSViewController & DetachableDetailChild,
              scene: DetailScene, animated: Bool) {
        currentChild?.prepareForRemoval()   // 断订阅 / 取消 backfill / 解 isRunning 观察 / 释放 per-attach scroll + sheet presenter
        // …再 removeFromSuperview / removeFromParent，然后 addChild + 四边钉住 next…
    }
}
避免

把订阅／任务清理只寄望于 deinit。removeFromParent() 断的是父子容器关系，不是最后一个强引用；crossfade 期间容器仍强持有离场子 VC，其定时器与订阅仍在改已离场视图，释放时机不可预测。CCTerm 的 TranscriptSwapCoordinator 持有的 per-attach 资源（每-attach Transcript2ScrollView、Transcript2SheetPresenter、isRunning 观察）必须在 prepareForRemoval() 里当场释放。

结构边必须同源相位、同步执行

CCTerm 有一条唯一的结构性上行边：旧 MainSelectionModel.select(_:)（新 WindowNavigationStore.set(_:)）在点击的同一 runloop 源相位里，同步通知旧 MainSelectionObserver（新 WindowNavigationObserver——它是 1:1、弱持有、可决策的回调，是 delegate 而非广播总线）。DetailNavigationCoordinator 是这条边的唯一注册者，它必须在同一源相位里同步调用 container.swap(...) 与 (child as? SessionPresentingDetailChild)?.present(sessionID:animated:)——绝不能推迟到 Task / beforeWaiting，否则会话换面板会跨帧撕裂（transcript 在错误宽度下先排版一帧）。这条边不得泛化成通知总线或第二个 observer 槽（显式非目标）。

为什么拆卸绑到钩子而非 deinit

把「离场即释放」绑到显式 prepareForRemoval()，资源生命周期就与「在场／离场」对齐——切走一屏，它的订阅、任务、观察当场停摆，绝不「背着屏幕继续跑」。这正是容器化想要的确定性；也是为什么 CCTerm 每个 DetachableDetailChild 都实现它、由 DetailContainerViewController 在 swap 时统一调用，而不是听任 ARC 在某个不确定时刻回收。

4.6 每个 @MainActor 引用类型都带空的 nonisolated deinit

纯 AppKit 没有 GC，且 CCTerm 目标 macOS——在 macOS-26 上 @MainActor deinit 的执行器跳转会在 libswift_Concurrency 里崩溃。规范：本章涉及的每个 @MainActor 引用类型（WindowController、容器 VC、路由 Coordinator、每个详情子 VC、选择 Store、搜索命令）都带一个空的 nonisolated deinit {} 斩断那次跳转。纯值类型 Model（如 DetailScene / NavigationTarget）无 deinit，也不需要。

本质

控制器体系是一棵职责树：MainWindowController 是根，独占窗口层；MainSplitViewController 是系统容器，组合侧栏与详情两个独立子 VC；详情侧的旧 DetailRouterViewController 一拆为二——DetailNavigationCoordinator 决定「展示谁、拿什么依赖造」、DetailContainerViewController 只负责「怎样挂载／拆卸」。每个子 VC 用 loadView/viewDidLoad 分离「建树」与「绑定」，用 DetachableDetailChild.prepareForRemoval() 让 per-attach 资源精确对齐「在场／离场」。反巨石的方法不是分文件，而是让树上每个节点只有一个理由存在、只有一个查看点——同时保住那条唯一的、同源相位同步换面板的结构边，因为它一旦异步就会跨帧撕裂 transcript。

思考题：有人想「简化」，把 DetailNavigationCoordinator 与 DetailContainerViewController 合回一个 DetailRouterViewController，理由是「反正只有一个详情容器，拆两个是过度设计」。同时他打算让 WindowNavigationStore.set(_:) 不再同步调 delegate，而是改用 withObservationTracking 让路由器异步响应选择变化，「和 SwiftUI 时代一样省事」。请结合本章与手册 3.4/5.2 指出这两处会各自引入什么问题。

查看答案

合回一个类的问题：决策（挑子 VC 类型、注入 DetailContext、区分活动会话 vs 草稿落地）与执行（addChild／四边钉住／crossfade／拆卸）变更理由完全不同，焊在一起就退回旧巨石：容器被迫认识 SessionManager 和所有子 VC 类型，baseScene(for:) 无法作为纯函数脱离窗口单测。这不是「单容器所以不必拆」——手册 5.2 的决策/执行分离针对的是职责而非实例数；这里确有两类不同职责，拆分是还债不是预付。（对比：给一个永远单屏、无分支的界面套 coordinator 才是过度设计——本容器有五种场景分支，不属此列。）

改异步观察的问题：这会击穿本章最要命的不变量。set(_:)（旧 select(_:)）→ delegate 是唯一结构性上行边，必须在点击的同一源相位同步跑完 container.swap + present(sessionID:)。withObservationTracking 的重求值落在下一个 beforeWaiting，等于把换面板推迟一个 runloop tick：transcript 会在旧宽度或未定帧的状态下先排版一帧，产生跨帧撕裂的可见闪跳。CCTerm 当年正是为此把这条边定成同步 delegate 而非 @Observable／通知总线，并列为显式非目标——「异步省事」恰恰是它要避免的那件事。修法：保持 delegate 同步边不动；若确有「非结构性」的 chrome 更新（工具栏 chip 增删）想异步，走另一条轻量 onTargetChange 闭包下行，绝不占用结构边。

Coordinator 与导航解耦

前四章我们把 CCTerm 切成了五层、把依赖收敛进组合根。本章处理最后一样赖在控制器里的东西——导航：侧栏点了一项，接下来右侧该展示哪个子 VC、由谁创建它、创建时喂哪些依赖、旧 VC 何时拆卸。今天的 CCTerm 把这段逻辑全塞进了一个 DetailRouterViewController——它既是唯一的结构观察者、又是路由决策者、又是子 VC 构造工厂、又是挂载/交叉淡入的执行机构，是个典型的“路由上帝 VC”。本章按 规范第 5 章把它拆成 Coordinator 决策 + Router 执行 两半，同时守住 CCTerm 一条不可动摇的硬约束：会话切换的结构性换面板必须发生在点击的同一 source phase 里——这条边不能变成 @Observable/Combine 那样晚一个 tick 的异步通知。

命名口径（本章统一）

窗口选择三件套在不同域评审里出现了两套改名。本章统一采用导航语义的一套并贯穿全书：旧 MainSelection → 新 NavigationTarget、旧 MainSelectionModel → 新 WindowNavigationStore、旧 MainSelectionObserver → 新 WindowNavigationObserver、旧 DemoKind → 新 DemoScene。DetailContext 里承载它的成员相应命名为 navigation。这解决了 “WindowNavigation* vs MainSelection*” 的跨域命名冲突——两套名字指的是同三个概念，落地时只能留一套。

5.2 Coordinator 决策、Router 执行：把上帝 VC 劈成两半

规范第 5.2 节引用 Firefox iOS 的做法，把导航拆成两层：Coordinator 决策“下一步去哪、由谁来做”，Router 执行“怎么把它展示出来”。CCTerm 的 detail 路由天然适配这条切法——决策部分（观察选中、判断场景、构造并注入子 VC、持有 app 级信号）上移成一个协调器，而必须依赖真实 view/frame 的挂载机制（addChild、四边约束、交叉淡入、首帧布局门）留在一个纯执行的容器 VC 里。

应当

把 DetailRouterViewController 一劈为二：决策半成为 DetailNavigationCoordinator（不含任何 view），执行半成为 DetailContainerViewController（不做任何路由决策，被动接收一个已经造好的子 VC）。

避免

把决策和挂载留在同一个 VC 里“只是拆成两个方法”。只要构造子 VC 的工厂和 addChild 还在同一个类里，这个类就仍然“知道流程”，无法脱离 detail 面板复用、也无法脱离真实 view 层级去单测路由决策。

决策半：DetailNavigationCoordinator

DetailRouterViewController（决策部分）旧类里的“结构观察者 + childKind(for:) + 子 VC 构造 + onActivateSession/启动失败弹窗”整体上移到这里。它是一个叶子协调器：detail 路由没有子流程，因此刻意不采纳 5.2 的 Coordinator 协议（不背 childCoordinators 空数组），只暴露 start()——这正是 5.5“不要给单屏无分支流程套壳”的应用。
@MainActor
final class DetailNavigationCoordinator: WindowNavigationObserver {
    private let context: DetailContext
    private let notifications: NotificationService
    private unowned let container: DetailContainerViewController   // 执行半，向下不成环故 unowned

执行半：DetailContainerViewController

DetailRouterViewController（执行部分）旧类里的 installChildForCurrentSelection / commitChildTransition / finishFadeOut / loadView 的 vibrancy 背板 / viewDidLayout 首帧门，整体下沉到这里。它是 5.2 意义上的“Router”：持背板、四边固定挂载唯一子 VC、跑交叉淡入、守首帧布局门——但不做任何路由判断，只接收协调器递来的成品子 VC。
@MainActor
final class DetailContainerViewController: NSViewController {
    private(set) var currentScene: DetailScene?
    private(set) var currentChild: (NSViewController & DetachableDetailChild)?
    var onFirstFramedLayout: (() -> Void)?

    override func viewDidLayout()   // 首帧门

    // 协调器决策后调它执行：普通子 VC 容器化（addChild + 四边固定）+ 交叉淡入
    func swap(to child: NSViewController & DetachableDetailChild,
              scene: DetailScene, animated: Bool)

    private func commitCrossfade()
    private func finishFadeOut(expected: NSViewController?)
    nonisolated deinit {}

为什么这样劈

挂载机制必须依赖一个真实的 view/frame 和 viewDidLayout 时机，天然属于 VC；而“给定一个 target 该造哪种 VC、连哪些依赖”是纯流程知识，天然属于协调器。劈开后：路由决策（baseScene(for:)）成了一个可脱窗单测的纯函数；挂载机制（swap(to:)）成了一个“接收成品、四边固定、跑淡入”的笨执行器——今天的 DetailRouterContainmentTests 直接改瞄准 DetailContainerViewController，无需再穿过路由决策。原生子 VC 在自己的 loadView 里固定四边，没有 NSHostingController/sizingOptions 的窗口塌陷隐患，swap 就是一次普通的子 VC 容器化。

5.3 两个协议：可拆卸子 VC 与“会承载会话”的窄接口

今天路由器对子 VC 有两种交互：所有子 VC 都要在被换走前拆卸每-attach 资源，而只有其中两种（transcript 与 draft-landing）需要被“喂一个 sessionId 去展示内容”。这两件事职责不同，必须是两个协议——不能把 present 塞进拆卸协议。

DetailRouterChild → DetachableDetailChild这正是规范 3.4 的 DetachableChild 确定性拆卸钩子，源码里就是单方法 prepareForRemoval()，仅改名对齐规范词汇。容器在 removeFromParent 前调它，让 per-attach 资源（滚动视图、sheet presenter、isRunning 任务）在“离场”时释放，而非等 ARC。
@MainActor
protocol DetachableDetailChild: NSViewController {
    func prepareForRemoval()   // 单方法契约，所有五类子 VC 都实现
}

@MainActor
protocol SessionPresentingDetailChild: NSViewController {
    func present(sessionID: String?, animated: Bool)
}
// 仅由 transcript 子 VC + draft-landing 子 VC 遵守；
// 协调器用 `as? SessionPresentingDetailChild` 派发，
// 精确复刻今天 `as? ChatSessionViewController` / `as? DraftSessionLandingViewController` 的 cast 派发
避免

把 present(sessionID:animated:) 折进 DetachableDetailChild。源码里五类子 VC 只有两类实现 present，路由器正是用 as? cast 派发，因为 compose/archive/demo 三类各自管理自己的内容。非 @objc 的 Swift 协议要求对所有遵守者强制——把 present 写进拆卸协议会导致 compose/archive/demo 被迫实现一个空方法，既编译不过“可选”语义，也谎报了契约。

DetailScene：从路由器的嵌套枚举升成顶层 Model

DetailRouterViewController.ChildKind → DetailScene旧枚举 .transcript/.compose/.draftLanding/.archive/.demo 嵌套在路由器内部，用于“同种复用 vs 换种交换”的比较。升成顶层 Model 后，决策半（Coordinator）与执行半（Container）共享它而互不拥有。“Kind”→“Scene”与 DemoScene 对齐，读作“哪一屏 detail 场景”。

    case transcript
    case compose
    case draftLanding
    case archive
    #if DEBUG
    case demo(DemoScene)

}
为什么升成顶层 Model

baseScene(for:)（NavigationTarget → DetailScene 的纯表）不碰 SessionManager，可以脱窗单测；而 .session 是否细化成 .draftLanding 需要问 SessionManager.isDraftSession，是协调器的实例方法 scene(for:)。把二者分开，正是复刻今天 childKind(for:)（纯）与 resolvedChildKind(for:)（相位相关）的分工——纯表那半永远可测。

5.4 那条唯一的结构性上行边：WindowNavigationStore

窗口选择状态被三方读取——AppKit 侧栏、detail 协调器、工具栏。它是引用类型、有身份、活得和窗口一样长，正是规范 7.4“单窗口选择/导航 → WindowController 持有的 Store”那一格。它不是 ViewModel（不是某个 VC 的私有派生态），也不是全局 store。

MainSelectionModel → WindowNavigationStore今天它是 @Observable。纯 AppKit 里去掉 @Observable：读取暴露成普通属性；那条唯一的同步结构边保留为 set(_:) 内的显式 delegate 调用；其余非结构性变化（侧栏高亮、工具栏、archive 过滤）走 7.4 规定的 delegate/闭包下行链 onTargetChange，而非 Combine @Published——Combine 是 7.4 里“每-VC VM 态”的机制，拿它当窗口导航广播是张冠李戴。
@MainActor
final class WindowNavigationStore {
    private(set) var target: NavigationTarget           // 旧 `selection`，只经 set(_:) 改
    var draftSessionID: String?                         // 保持可写：compose/submit 直接写清它
    var archiveFolderFilter: String?                    // 旧 archiveSelectedFolderPath
    weak var structuralObserver: WindowNavigationObserver?   // 唯一同步结构边
    var onTargetChange: ((NavigationTarget) -> Void)?   // 7.4 非结构性下行链
    var isComposing: Bool { get }                       // target == .newSession
    var effectiveSessionID: String? { get }

    init(target: NavigationTarget)
    func set(_ target: NavigationTarget)                // 旧 select(_:)：改值 + 同步触发 structuralObserver
    func reroute(toSession id: String)                  // 旧 promote(to:)：草稿→活跃，原地重触发观察者
    nonisolated deinit {}
}

@MainActor
protocol WindowNavigationObserver: AnyObject {          // 旧 MainSelectionObserver
    // AppKit sender-first delegate 约定：store 作首参
    func windowNavigation(_ store: WindowNavigationStore, didChangeTo target: NavigationTarget)
}
应当

让 set(_:) 在点击的同一 source phase 内同步调用 structuralObserver.windowNavigation(_:didChangeTo:)。协调器的回调里同步调 container.swap(...) 与 (child as? SessionPresentingDetailChild)?.present(...)，让 layoutSubtreeIfNeeded 先把 frame 结算好，transcript 在最终宽度下一次成型（§2.19 单宽度契约）。

避免

把这条结构边改成 @Observable/Combine，或把 swap/present 推迟到一个 Task/beforeWaiting hop。二者都晚一个 tick——会话切换会跨帧撕裂：旧 transcript 先消失、新的下一帧才成型，中间一帧空白或错宽。

为什么这条边是 delegate 而非总线

@Observable 的重求值挂在 beforeWaiting 的 runloop 观察者上，Combine 默认也要经调度——都比“点击处理函数正在执行”晚一个 tick。而结构性换面板要求“点下去的这一帧就把面板换好”。所以它必须是同步 delegate，在 source phase 内就地跑完。这也是 CCTerm 反复强调的“唯一结构上行边”：不要把它泛化成通知总线、也不要加第二个观察者槽——新的“结构性响应选中”需求一律走协调器。

草稿→活跃：reroute 是同一条边的原地重触发

首次发送把一个草稿会话提升为活跃会话时，target 值不变（仍是 .session(id)），但子 VC 要从 DraftSessionLandingViewController 换成 transcript 子 VC。reroute(toSession:)（旧 promote(to:)）就地重触发同一个观察者回调，让协调器重算 scene(for:) 并 swap——无需伪造一次 target 变更。

为什么激活/失败信号只此一处持有

onActivateSession、onLaunchFailure 都是窗口生命周期级的 app→detail 信号，交给协调器一次性持有，而不是让每个子 VC 各自观察。子 VC 随选中来来去去，若各自订阅这些信号，会出现“旧会话弹了两次失败框”或“离场的子 VC 仍在响应激活”。协调器活得和窗口一样长，是这些信号的唯一诚实归属。

5.6 发送与 /命令：也是导航，归协调层

首次发送要“把草稿提升为活跃会话并换路由”；/new、/clear 要“建一个种子草稿并选中它”。两者的落点都是 WindowNavigationStore.set/reroute——这是路由变更，按第 5 章属于协调层，不属于 Service（Service 不得驱动 UI/导航）。

跨域裁决：协调器类 vs 自由函数——本书取自由函数

两处评审给了矛盾结论：inputbar 域主张 submitSessionInput / runBuiltinSlashCommand 保留为 @MainActor 自由函数（引 5.5“别给单步无分支流程套协调器壳”）；chatpane 域主张升成 SessionInputSubmitter / BuiltinSlashCommandCoordinator 类（为可注入的测试缝）。两者是同两个源函数受到相反处理，若各自落地会让 chat 与 compose 路径分叉。本书统一裁定：二者都保留为自由函数，把 WindowNavigationStore + SessionManager 作参数传入——这更贴合 5.5，且函数本就单一职责、已住在命名良好的文件里（SessionInputSubmit.swift）。仅当某个测试确实需要注入替身时，才局部升级为可注入的对象；不作为默认形态。

为什么归协调层而不是 Service

submitSessionInput 里除了组装消息体、记录 recentProjects，还有一步 navigation.reroute(toSession:)——这是换面板。runBuiltinSlashCommand 结尾是 navigation.set(.session(draftID))——这是换路由。规范 1.2 明确 Service“不引用 View、不做 UI”，一个会驱动换面板的对象绝不能挂 Service 后缀。它们也不叫 …Router——5.2 里 Router 专指 present/dismiss 的执行机构，而这两个函数是在做流程决策，属协调层。而“清草稿要在 onSubmit 前同步跑完”这类拆卸敏感的顺序，正因为它在同一 source phase 内与同步换面板一起就地执行，才不会被随后的拆卸吞掉。

本质

CCTerm 的导航解耦，是把一个“路由上帝 VC”按第 5 章的 Coordinator 决策 / Router 执行 劈开：DetailNavigationCoordinator 独占“选中变了该造哪种 VC、连什么依赖、下一步去哪”，DetailContainerViewController 只做“接成品、四边固定、跑淡入”。贯穿其间的是那条唯一的、同步的、source-phase 内的结构上行边——WindowNavigationStore.set(_:) → WindowNavigationObserver——它是 delegate 而非总线、是同步而非晚一 tick 的 @Observable，因为会话换面板必须在点击那一帧完成，容不得跨帧撕裂。发送与 /命令 因终点是换路由而归协调层；SwiftUI 的环境注入与递增计数器总线被更窄的初始化器注入与一对一命令替代。守住“决策与执行分离”和“那一条同步结构边”，导航就从散在 VC 里的暗线变成一处可读、可测、可复用的明线。

思考题：有人为了“统一”，提议让 DetailNavigationCoordinator 也去 withObservationTracking 观察 WindowNavigationStore.target，这样就不必让 Store 持一个 structuralObserver 弱引用、看起来更“响应式”。同时他把 swap(to:) 挪进了一个 Task { @MainActor in … } 里“避免在观察回调里做重活”。请指出这两处改动分别会破坏本章的哪条硬约束，会产生什么可观察的故障，并给出符合规范的写法。

查看答案

改动一：用 withObservationTracking 取代同步 delegate。破坏“唯一的、同步的、source-phase 内的结构上行边”。Observation 的重求值挂在 beforeWaiting 的 runloop 观察者上，比“侧栏点击处理函数正在执行”晚一个 tick——换面板于是不在点下去的那一帧发生。可观察故障：点侧栏另一个会话，旧 transcript 已被逻辑标记要换走、但新面板要到下一帧才成型，中间出现一帧空白或旧内容以错误宽度停留（撕裂 §2.19 的单宽度契约）。正确写法：保留 Store.set(_:) 内对 structuralObserver.windowNavigation(_:didChangeTo:) 的同步调用，协调器作为唯一结构观察者就地跑完决策。

改动二：把 swap(to:) 推进 Task。同样破坏 source-phase 同步性——即便观察回调改回同步 delegate，只要 swap/present 被塞进 Task，它就要等到未来某个 source phase 才执行，layoutSubtreeIfNeeded 结算 frame 与 present(sessionID:) 的 typeset 被拆到不同帧，transcript 仍会以非最终宽度先成一次型、再重排。正确写法：观察回调内同步调 container.swap(to:scene:animated:)，紧接着同步 (child as? SessionPresentingDetailChild)?.present(sessionID:animated:)，绝不经 Task/beforeWaiting hop。

为什么“看起来更响应式”反而错。响应式（@Observable/Combine）适合的是“数据变了，视图下一帧刷新一下”这类可以容忍一 tick 延迟的下行数据；而这里是结构性换面板，要求与用户点击同帧完成。二者机制不同源：非结构性的下行（侧栏高亮、工具栏 chip）本就走 onTargetChange 闭包链，可以晚一 tick 无所谓；结构边必须同步。把结构边“也做成响应式”不是统一，是把一条有强时序要求的边降级成了没有时序保证的边。

Session 领域：Model 与 Service

前五章把骨架搭好了：五层职责（ch.1）、模块边界（ch.2）、控制器与容器（ch.3）、组合根注入（ch.4）、Coordinator 导航（ch.5）。本章把这套纪律落到 CCTerm 最核心、也最容易分层错位的一片——「会话」。这里挤着十来个当前类型：Session、SessionRuntime、SessionDraft、SessionManager、SessionRepository、SessionConfig、SessionRecord、CLIClient、Worktree、HistoryLoader、TitleGenerator，以及一堆 tracker。它们今天靠 SwiftUI 的 @Observable 自动追踪缝在一起；纯 AppKit 里这层魔法没有了，必须先回答一个前置问题——每个类型到底是 Model、Service、Store 还是 Protocol？其中最要害的一刀是：Session（会话句柄）该退化成一个「像 ViewModel 的状态持有者」，还是坐实为一个 Service？本章给出答案，并把「Session 的可观察性」这条硬前置补齐——它是 chatpane / inputbar 能否编译的地基。

会话句柄（session handle）运行时代表「一个正在进行或已归档的对话」的引用类型对象。它有稳定身份（同一会话全局唯一一份）、长生命周期、被侧栏 / 详情 / 输入栏多处共享读写，并由注册表创建，而非视图自造。旧 Session 即此。
运行时（runtime）封装「与 CLI 子进程通信、状态机推进」这类 I/O + 生命周期的对象。它是会话句柄背后真正跑活的引擎，对外以显式变更通知暴露 isRunning / status / pendingPermissions / turnUsage。旧 SessionRuntime。
不可变快照（immutable snapshot）描述「一个会话在创建时长什么样」的值类型：工作目录、模型、权限模式、源分支等。复制即安全、可 Codable、可脱离 UI 测试。旧 SessionConfig。
显式变更通道（explicit change channel）替代 @Observable 自动依赖追踪的接线：运行时在写入状态的同一源码点同步触发 onXxxChange 闭包（或 Combine @Published），把「变了什么」显式推给订阅方。

6.1 先给会话领域的每个类型定层

ch.1.2 的第一条纪律是「把每个类型明确归到五层之一，归不进去的正是耦合藏身处」。会话领域是这条纪律最好的试金石——因为这里同时住着值、身份、I/O、缓存与快照。先把结论摆成一张表，再逐个讲清「为什么是这一层」。

为什么先定层，而不是先改代码

SwiftUI 时代，这些类型都靠 @Observable 被视图「顺手」读到，层的界线是模糊的——一个 @Observable class Session 既像 Model 又像 ViewModel 还兼 Service。去掉 SwiftUI 后，编译器不再替你缝合，每个跨类型引用都要你亲手接线；接线之前不定层，就会把 I/O 接到视图上、把可变状态复制成两份真相。定层是后续所有接线（6.4 的可观察性、6.5 的注入）的前提，不是形式主义。

6.2 关键裁决：Session 是 Service，不是 ViewModel

会话领域里唯一真正需要「拍板」的一刀，是 Session。它诱人地长得像一个 ViewModel：持有 title / status / messages，视图直接读它渲染。但 ch.1.4 与 ch.5.5 的 ViewModel 判据（有真实派生逻辑、可脱窗口单测、不 import AppKit）它一条都不满足——它没有「把 Model 加工成显示值」的派生逻辑，它有的是身份、长生命周期、跨多个视图的共享可变状态、由注册表创建。这正是 ch.1.2 判 Service 的四要件。

应当

把 Session 定为 Service：@MainActor final class，由 SessionManager（注册表）创建并强持有，经初始化器 / Context 注入给需要它的 VC。它对外发布状态、接收动作，但自己不 import 任何 NSView。

// 旧 Session（@Observable）→ 新 Session（Service，显式变更通道）
@MainActor
final class Session: Identifiable {
    let id: SessionID
    let config: SessionConfig            // 不可变快照（Model，见 6.3）
    let draft: SessionDraft              // 编辑中状态（Store）
    private let runtime: SessionRuntime  // I/O 引擎（Service）

    // 只读投影：视图经 VC sink / 闭包拿到，不反向写
    private(set) var title: String
    var isRunning: Bool { runtime.isRunning }

    // 显式变更通道，替代 @Observable 自动追踪（见 6.4）
    var onIsRunningChange: ((Bool) -> Void)?

    nonisolated deinit {}                // macOS-26：见 ch.9 与 6.6
}
避免

把 Session 当 ViewModel 来「每视图一个」地造。它是共享单例级的会话身份——侧栏行、详情转录、输入栏 chrome 读的必须是同一个 Session。若某视图自己 new 一个 Session 来渲染，立刻产生第二份真相（ch.1.2「可变引用型 Model / 到处 .shared」正是 bug 温床）。

为什么不是 ViewModel

ViewModel 的语义是「把一个 Model 投影成一个视图该显示什么」——它是短命的、每屏一个、无身份的加工层（ch.5.5）。Session 恰好相反：它就是那个被投影的真相，全应用共享一份、活得和会话本身一样长、有全局唯一 id。把真相源误当投影层，会诱使你为每个展示它的视图各造一个「Session ViewModel」，于是同一会话的运行状态散成 N 份，谁改了 isRunning 都说不清。真正的投影层（若需要）应是无 UI 依赖的小 ViewModel（如 TranscriptProjectChipViewModel），从 Session 派生只读展示值，而不是取代它。

CCTerm · Content/Chat/CLAUDE.md 非目标项目明确记录「不引入 session/transcript 协调 ViewModel」。本章的裁决与之一致：Session 是 Service（状态真主），转录的流程编排交给 Coordinator（TranscriptSwapCoordinator，见 ch.5 / 后续章节），二者都不是 ViewModel。会话领域全域只在 inputbar 等消费侧才出现真正的 ViewModel。

6.3 值与引用的分界：快照是 Model，运行是 Service

ch.1.2「Model 用值类型、Service 用引用类型」不是风格，是语义匹配。会话领域里这条线画得最清楚的地方，是 SessionConfig / SessionRecord / Worktree（值）与 Session / SessionRuntime（引用）的对照。

应当

把「一个会话创建时的样子」建成不可变值 SessionConfig（Model），沉进 Core 包（ch.2.2，不 import AppKit）；运行时状态则住在引用型 Session / SessionRuntime（Service）里。快照喂给运行时，运行时不回写快照。

// Model：不可变快照，Foundation-only，可沉进 Core 包
struct SessionConfig: Codable, Hashable {
    let cwd: URL
    let model: String
    let permissionMode: PermissionMode   // 也是 Model（值枚举）
    let sourceBranch: String?
    let additionalDirectories: [URL]
}

// Model：git worktree 的值描述——无行为、无副作用
struct Worktree: Hashable {
    let path: URL
    let branch: String
    let isTemporary: Bool
}
避免

把 SessionConfig 做成可变 class 让运行时中途改它，或往其中塞 NSImage / NSColor 之类表现字段。前者制造「配置的第二份真相」，后者把它钉死在 AppKit 上、进不了 Core 包，也无法脱窗口单测（ch.1.2 红线）。

为什么快照必须是值

「创建时的配置」在语义上就是一个不再改变的事实——它描述「这个会话是以什么参数起的」。用值类型表达它，复制即安全，可以随手传进 SessionRuntime.start(config:)、存进 SessionRecord、在测试里 SessionConfig.fixture() 一行造出来，而不必起一个会话对象。反过来，运行时状态（isRunning、当前轮次、待决权限）天然是「一个持续存在、随时间改变、被多处观察的东西」，只能是引用型 Service。把这两者的语义搞反——可变的 config、值化的 runtime——正是 ch.1.2 点名的大型项目状态失控之源。

6.4 硬前置：Session 的显式可观察性（替代 @Observable）

这是本章、也是整个会话领域向纯 AppKit 迁移的硬前置，不是软优化。去掉 SwiftUI 就去掉了对 Session.isRunning / status / pendingPermissions / turnUsage 的自动追踪——chatpane、inputbar 依赖这些状态推送才能编译。所以 Services/Session 必须先补上显式变更通道，chatpane / inputbar 才能落地。对标 ch.7.1 的机制选型表：这些是「一串状态值持续推给视图」，用 Combine @Published（一对多）或同步 onXxxChange 闭包（源码点同步、一对一），而不是 @Observable。

应当

SessionRuntime 在写入状态的同一源码点同步触发变更通知——尤其是 onIsRunningChange，必须在运行态翻转的那一行就同步发出，而不是绕一个 Task / withObservationTracking。

@MainActor
final class SessionRuntime {
    private(set) var isRunning = false {
        // 写入点即通知点：同一源码点、同一 runloop source phase
        didSet { if oldValue != isRunning { onIsRunningChange?(isRunning) } }
    }
    var onIsRunningChange: ((Bool) -> Void)?
    @Published private(set) var turnUsage: TurnTokenUsage?   // 一对多用 @Published

    func send(_ input: InputSubmission) {
        isRunning = true          // ← 这一行同步触发 onIsRunningChange(true)
        // …驱动 CLIClient…
    }
    nonisolated deinit {}
}
避免

用 withObservationTracking { runtime.isRunning } onChange: { Task { … } } 这类「追踪 + Task」模式。Task 会在未来某个 source phase 才恢复（ch.7 runloop tick 模型），而转录的同会话交换（旧 TranscriptSwapCoordinator.startRunningObservation）承诺同步发生——两者冲突会让交换跨帧撕裂。

CCTerm · TranscriptSwapCoordinator.startRunningObservation今天它用 withObservationTracking 包在一个 Task 里观察 isRunning——这已经和「同步交换」的承诺相冲突。迁移时它必须改成直接订阅 runtime.onIsRunningChange 的同步 sink：旧 controller.setLoading(running) 在闭包体里同步调用，与 ch.7.3「数据向下」的 exact-delta 通道一致（不走 .receive(on:) 慢一拍）。
迁移次序（跨域约束）

Services/Session 补 onXxxChange 是 chatpane / inputbar 的编译前置：这两个消费域的 ChatInputBarChromeView（发送↔停止切换）、转录的 loading 态都靠 onIsRunningChange 驱动。因此本领域的这一改动必须先落地，否则下游根本编译不过。这也呼应 ch.4：状态的真主（SessionRuntime）先把发布口开好，消费方才能在组合根接线。

6.5 会话领域的组合根接线：谁造 Session、谁注入

ch.4.4 规定「new 具体类型 + 连线」只在组合根一处。会话领域的具体化落点是 SessionManager（注册表）——它是唯一 new 出 Session / SessionRuntime 的地方，本身在 AppDelegate.assemble() 里被构造一次，再经 Context 逐层下发（ch.4.5）。

应当

SessionManager 面向协议持有其 I/O 协作者（CLIClient / HistoryLoader / SessionRepository），由组合根注入；它对外是「造会话、按 id 取会话」的注册表。视图 / VC 只通过注入拿到 Session，从不自造。

@MainActor
final class SessionManager {
    private let cliClient: CLIClient          // 协议：可打桩（ch.4.3）
    private let historyLoader: HistoryLoader  // 协议：可打桩
    private let repository: SessionRepository
    private var live: [SessionID: Session] = [:]   // 注册表：同 id 唯一一份

    init(cliClient: CLIClient,
         historyLoader: HistoryLoader,
         repository: SessionRepository) {
        self.cliClient = cliClient
        self.historyLoader = historyLoader
        self.repository = repository
    }

    // 唯一造 Session 的地方——保证「同一会话全局一份」（呼应 6.2）
    func session(for id: SessionID) -> Session {
        if let s = live[id] { return s }
        let s = Session(id: id, /* 注入 runtime/config/draft */)
        live[id] = s
        return s
    }
    nonisolated deinit {}
}
避免

把 SessionManager 或 CLIClient 写成 .shared 单例当默认注入手段（ch.4.6）。它们是业务服务，不是「进程级缓存」那种诚实的单例——真正该留单例的是 ModelCatalogStore（fork 子进程枚举模型）、CoreDataStack（每进程一个持久化栈）这类，且各自写明理由。SessionManager 必须经组合根 init 注入，测试才能塞进假的 CLIClient。

为什么注册表模式而非视图自造

「同一会话全局唯一一份」是 6.2 判 Session 为 Service 的直接推论，而保证唯一性的机制就是注册表：SessionManager.session(for:) 缓存活跃会话，任何消费方拿到的都是同一个引用。若允许视图 new Session(id:)，唯一性立刻破裂。把创建权收敛到注册表、注册表本身经组合根注入，同时满足 ch.4（依赖显式、可替换）与 ch.1.2（单一真相源）。

6.6 会话对象的确定性拆卸

纯 AppKit 没有 GC，会话对象又长命且相互强持有（Session → SessionRuntime → 在飞的 CLIClient 任务），拆卸必须显式（ch.9.1 / ch.3.4）。会话领域有两条规范。

应当

(1) 每个 @MainActor 会话类型（Session / SessionRuntime / SessionManager / SessionRepository / trackers）都带一个空的 nonisolated deinit {}——macOS-26 上 @MainActor deinit 的执行器跳转会在 libswift_Concurrency 里崩，空的 nonisolated deinit 切断这次跳转。(2) 值型（SessionConfig / Worktree / SessionRecord）不需要、也不应有 deinit。变更通道（onIsRunningChange 等）用 [weak self] 捕获订阅方，避免 Session ⇄ 订阅方成环（ch.9.1.2）。

CCTerm · CLAUDE.md「Deterministic teardown」项目已要求「每个 @MainActor @Observable / VC 类型带空 nonisolated deinit {}」。迁移后 @Observable 去掉了，但这条 deinit 纪律原样保留——它对抗的是 macOS-26 的执行器跳转崩溃，与 @Observable 无关。会话领域的引用型 Service 全部照办。
为什么会话对象尤其要显式拆卸

SessionRuntime 挂着一个可能长跑的 CLIClient 子进程通信任务。若只寄望 ARC 在「某个不确定时刻」释放，任务会背着一个已被切走的会话继续跑、继续触发 onIsRunningChange，改动早已离场的 UI（ch.3.4 隐患）。所以运行时的取消要绑定到显式生命周期点（会话归档 / VC 的 prepareForRemoval()），而不是 deinit；deinit 只做「切断执行器跳转」这一件与业务无关的事。

本质

会话领域的迁移，九成是定层而非重写。一旦把每个类型钉进五层——不可变快照与 git 描述是 Model（沉进 Core、可脱窗口测）；会话句柄、运行时、注册表、仓库、CLI 客户端、历史加载、标题生成、trackers 是 Service（身份 + I/O + 状态真主）；草稿是 Store；跨进程与读盘的能力抽成 Protocol 以便打桩——耦合就无处藏身。三条不可退让的裁决支撑起全局：Session 是 Service 不是 ViewModel（真相源不是投影层）；@Observable 必须先换成源码点同步的显式变更通道（否则下游编译不过、同步交换会撕裂）；创建权收敛到注册表并经组合根注入（保证同一会话全局一份）。守住这三点，会话就从「SwiftUI 顺手读到的一团 @Observable」，变成一片依赖单向、状态可追踪、可脱 UI 测试的清晰领域。

思考题：迁移中有人提议——既然 Session 被侧栏、详情、输入栏三处视图读取，干脆给每处各建一个 SessionViewModel（@Published var title / isRunning），从同一个底层 Session 派生；再把 SessionRuntime.isRunning 的观察统一用 withObservationTracking 包进一个 Task，回调里刷新这三个 ViewModel。请结合本章 6.2 / 6.4 与 ch.7 的 runloop 模型，指出这个方案的两处根本错误，并给出符合规范的接线。

查看答案

错误一：把「投影层」误当「真相源的替身」，反而复制了真相（违反 6.2 + ch.1.2）。三处各建一个 SessionViewModel 本身不算错——若它们只做只读派生（如把 Session.originPath 折成短标题，像 TranscriptProjectChipViewModel 那样），是合法的窄 ViewModel。但「各持一份 @Published var isRunning 并各自被 Task 回调刷新」把运行态复制成了三份可写副本：三份可能因为 Task 恢复时机不同而短暂不一致，「到底谁说了算」无从判定。
→ 修正：运行态的唯一真主是 SessionRuntime（Service）。三处视图要么直接经 VC sink runtime.$turnUsage / onIsRunningChange（一对多用 @Published，一对一用同步闭包，ch.7.1），要么各自的窄 ViewModel 只转发而不缓存另一份真相。

错误二：withObservationTracking + Task 破坏「同步」承诺（违反 6.4 + ch.7 runloop 模型）。Task 在未来某个 source phase 才恢复；而转录的同会话交换必须在点击的同一 source phase 同步完成，否则 loading 态与转录内容跨帧撕裂。用 Task 刷新 isRunning，交换就会看到「慢一拍」的运行态。
→ 修正：SessionRuntime 在写入 isRunning 的 didSet 里同步触发 onIsRunningChange（6.4 的示例）；TranscriptSwapCoordinator 与 ChatInputBarChromeView 直接订阅这个同步闭包，闭包体内同步调用 controller.setLoading(_:)，全程不经 Task / .receive(on:)。

本质：本方案两处错误同源——都在「投影」与「真相」之间划错了线。Session / SessionRuntime 是真相源（Service），只能有一份、且在源码点同步发布；ViewModel 只能是它的只读投影，既不复制可写状态，也不引入异步跳转。守住 6.2（真相是 Service 不是 VM）与 6.4（源码点同步的显式通道），三处视图的接线就退回一条干净的「数据向下」。

视图层：从 SwiftUI 叶子到 AppKit 笨视图

这是整份手册的枢纽章——前面几章反复援引「第 7 章的机制表」「按第 7 章手工焊接线」，本章就是那张表和那套接线。CCTerm 今天是「AppKit 外壳 + SwiftUI 叶子」：外壳早已归位（ch.3/4/5），要迁的是叶子——每一个被 NSHostingController/NSHostingView 托管的 SwiftUI View，都要变成一个程序化构建的 NSView，靠 configure(...) 接收数据（数据向下）、靠闭包/delegate/target-action 上报意图（事件向上），且自己不持有真相、不做业务判断——一个「笨视图」。本章先立机制选型表（§7.1，即全书援引的那张），把 CCTerm 每个 SwiftUI 叶子逐一映射到它的原生对应物（§7.2），给出程序化建树、Auto Layout 默认、手动 layout() 热点例外的落地规范（§7.3/§7.4），最后专门回答本次迁移里最难的一问——CCTerm 的输入栏控件群里，哪些该配 ViewModel、哪些坚决不配（§7.5）。对标规范手册第 6、7 章。

笨视图（dumb view）只做两件事的 NSView：把外部喂进来的成品值显示出来（configure(with:)），把用户操作上报出去（闭包/delegate/target-action）。它不持有真相源、不 import Service、不做业务判断——判断在宿主 VC 或 ViewModel（§7.5 / ch.9）。
托管消融（de-hosting）删除 NSHostingController/NSHostingView 及其 sizingOptions 体操，把被托管的 SwiftUI View 换成程序化 NSView：fill-pane 宿主的子 VC 在 loadView 里直接四边钉住内容，不再有窗口塌陷风险。
纯 Presenter（cell view model 值）把「一行/一张卡该显示什么」建成一个无 AppKit、无状态的值类型（struct），承载已算好的派生显示数据（副标题、diff、chips…）。cell/卡片视图只 configure(with:) 这个成品。它不是 ViewModel——没有身份、没有生命周期、没有异步（规范 8.4）。
ViewModel（本章严格门槛）一个 @MainActor、无 AppKit 依赖、可脱窗单测的引用类型，承载真实的派生 / 异步 / 交互状态机逻辑，用 @Published 暴露成品显示值、接收用户意图转成对 Service 的调用。只有当逻辑「够分量」时才诞生（ch.1.4）；一行判空、一次映射都不配（§7.5）。

7.1 机制选型表：五种接线，按基数与方向选

SwiftUI 的 @State/@Binding/@Observable 把「状态变了→视图重画」托管给框架；纯 AppKit 没有这层魔法，绑定要你手工焊。焊之前先选机制——第一原则是基数（1:1 / 1:多）与方向（要不要回话）。这张表是全书（ch.1 动作二、ch.5、ch.6、ch.9）反复援引的那张。

按「最窄够用」自上而下选：一个接收方且需回话/决策 → delegate；一个接收方只需被触发 → target-action 或闭包；一串成品显示值持续推给视图 → Combine @Published；一个精确渲染器 delta 或时序敏感调用 → 命令式方法（转录唯一例外）。

为什么表里要单列「命令式 delta」这一行

其余五行都是 SwiftUI 消失后的常规替身；命令式 delta 是被严格限定的一个例外，不是第二种默认做法。它只服务转录——转录每秒可能收到几十条流式增量，把「新增一行」表达成「发布整份 [Block] 让表 diff」既算不准（StableBlockID 身份丢失）又太慢，且 @Published 的下行晚一 tick 会打断流式动画（ch.1.4）。所以它走 apply(_ change:) 在同一 source 相位把精确增量直接递给自绘表。只有转录配得上这一行；任何「为了统一而给别处也上命令式」都要被驳回，任何「为了统一而把转录降级成 Combine」也要被驳回。

为什么 @Binding 要拆成「下行 + 上行」两条边

SwiftUI 的双向 @Binding 把「显示当前值」和「回写新值」缝成一根双向线，框架替你保证不打架。纯 AppKit 没有这个保证，双向写会让「当前值到底谁说了算」不可知，引出竞态、重入、无限刷新。所以迁移时每根 @Binding 都拆成两条单向边：configure 把成品值下行灌进控件，闭包/delegate 把用户改动上行报给真相源，真相源改完再走下行刷新——一个闭环，状态只有一处能改（规范 7.3）。compose 卡片、pickers 都是这个形状。

7.3 程序化建树：loadView 建树、viewDidLoad 绑定

SwiftUI 叶子没了，视图树要用代码写（规范 6.1：不用 xib/Storyboard，代码化视图树可 git diff 逐行评审）。CCTerm 的规范承接 ch.4：loadView() 只建树、viewDidLoad() 才绑定，子视图用 lazy var 在闭包里配置自身，「装层级」与「装约束」拆成两个方法。

应当

子视图 lazy var + 闭包内只做自身外观配置（不碰布局、不碰层级）；loadView() 建树 + 赋 view（程序化重写不调 super.loadView()）；viewDidLoad() 建立 Combine 订阅、绑定 delegate、装 per-attach 资源。

@MainActor
final class TodoListViewController: NSViewController {   // popover 内的 view-based 列表
    private let session: Session
    private lazy var scrollView = NSScrollView()
    private lazy var tableView  = NSTableView()
    private var cancellables = Set<AnyCancellable>()

    init(session: Session) { self.session = session; super.init(nibName: nil, bundle: nil) }
    @available(*, unavailable) required init?(coder: NSCoder) { fatalError() }

    override func loadView() {                    // 只建树、赋 view；不调 super
        scrollView.documentView = tableView
        let root = NSView(); root.addSubview(scrollView)
        // …四边约束 + PopoverListMetrics.width / maxHeight 上限…
        view = root
    }
    override func viewDidLoad() {                 // 绑定：数据下行订阅
        super.viewDidLoad()
        tableView.dataSource = self; tableView.delegate = self
        session.$todos.receive(on: DispatchQueue.main)
            .sink { [weak self] _ in self?.reload() }.store(in: &cancellables)
    }
    func prepareForRemoval() { cancellables.removeAll() }   // 确定性拆卸（ch.9）
    nonisolated deinit {}
}
避免

在 loadView() 里做订阅/磁盘请求/RPC（可能在视图未就绪时就收到回调）；把子视图声明成 var content: X! 再到处判空（把时序错误从编译期推迟到崩溃时）；程序化重写 loadView 却调了 super.loadView()（默认实现会去按 nib 名加载归档，白跑一遍）。

7.4 布局：Auto Layout 默认，手动 layout() 只给转录热点

规范手册第 6 章：布局默认声明式（Auto Layout），例外才命令式（手动 layout()）。CCTerm 绝大多数迁移后的笨视图（输入栏、权限卡、compose 卡、pickers、popover 行、todo 行、上下文分解条）都用 Auto Layout；只有转录——第 1.4 节的承重例外——下沉到手动 layout()，因为它是被 profiler 证实的高频重排热点（详见 ch.8）。

应当（默认：Auto Layout）

参与布局的视图一律 translatesAutoresizingMaskIntoConstraints = false，NSLayoutConstraint.activate([...]) 批量激活，优先 anchor，线性排布用 NSStackView；同一行谁伸缩谁固定用内容拥抱/抗压优先级表达。TodoStatusGlyphView 的旋转环用 CABasicAnimation（keyPath transform.rotation.z），不用逐帧重画。

避免（普通视图别手写 layout()）

给普通笨视图（输入栏、pickers、popover、设置面板）手写 layout()「图性能」——拿不到可测收益，还丢掉自适应（多语言/动态字号/缩放）与可读性；也别在同一视图对同一子视图混用约束与手动 frame（约束会覆盖 frame，随机错位）。对纯 Auto Layout 视图重写 isFlipped 也是噪声。

host 尺寸不再是问题，但别忘了它消失的原因

SwiftUI 时代 CCTerm 要小心 NSHostingController 的 sizingOptions：fill-pane 宿主用 []（否则 fittingSize 泄进 split 塌陷窗口），组件宿主用 [.intrinsicContentSize]。托管消融后这整套体操消失——但底层道理仍在：一个视图要么填满容器（容器定尺寸），要么作为组件内嵌（内容定尺寸）。输入栏是后者（居中、宽度封顶、高度由内在尺寸给），权限卡宿主是前者（regime-A 四边填满、几何固定，卡片原地淡入不顶动底栏）。去掉 hosting 不等于可以不想尺寸归属，只是不再借 sizingOptions 表达。

7.5 ViewModel 取舍：两个该有、四个不该有

这是本次视图层迁移最容易做错的一步：SwiftUI 里每个 View 背后常挂一个 @Observable，一迁移就有人机械地给每个笨视图都配一个 ViewModel。规范手册 ch.1.4 / ch.5.5 的门槛是——ViewModel 只为「真实、可脱窗单测、无 AppKit」的派生/异步/交互状态机而生；一行判空、一次映射、一段纯逻辑都不配。CCTerm 的输入栏控件群恰好把两边都占全，是讲清这条门槛的最佳样本。

该有 VM（1）：AskUserQuestion 多步选择器 → AskUserQuestionViewModel

这是一个真正的多问题交互状态机：多问题翻页、单选/多选、自由文本「Other」、confirmEnabled 校验、答案回填、最终 payload 组装。逻辑够分量、分支多、且天然可脱窗单测——正是 ch.1.4「该抽 VM」的典型。

    private var otherText: String

    func selectOption(_ i: Int); func toggleMulti(_ i: Int); func engageOther(_ text: String)
    func goBack(); func confirm() -> SubmitPayload?    // 校验 + 组装最终答案
}

薄 AskUserQuestionCardViewController 只把 VM 的 @Published 绑到控件、把 onSubmit/onCancel 上报给 PermissionCardViewController（后者映射成 request.allowOnce(updatedInput:) / deny()）。焦点驱动的「Other 行收起」落成 controlTextDidEndEditing；Esc→cancelOperation；确认默认键 keyEquivalent "\r"。

该有 VM（2）：模型 / effort / 权限模式选项解析 → InputChromeOptionsViewModel

它把异步到达的 ModelStore 目录与每会话状态对账，算出 visibleModels / activeModel / effectiveEffort / activeEffortLevels（含 .ultracode 追加）/ visiblePermissionModes（.auto 门控），并跑幂等的 seed / backfill——这些逻辑在 SwiftUI 里被摊在两个 picker view 的 .task(id:) + 私有方法里，重复且难测。收进一个无 AppKit 的 VM（由 chrome VC 持有），目录时序规则就能单测，重复的 activeModel 逻辑也从三处收敛到一处。写仍走 session.setModel/setEffort/setFastMode/setPermissionMode（事件上行）。

不该有 VM：这四类各有更轻的归属

① 权限卡种类解析（PermissionCardKind）——纯 request→category 映射，无派生/异步/交互态，留作 Model enum + 静态 kind(for:)，被 PermissionCardView 的 body-switch 直接消费。包 VM 只是为抽象而抽象。
② 各权限卡体的显示派生——shell/fileWrite/sed/mcp/task… 每种是纯 request→显示值派生，落成一个 Permission*CardPresenter 值（cell-view-model 模式，规范 8.4），NSView 只 configure(with:)。无异步、无交互态 ⇒ Presenter 值，不是 VM 对象。
③ 后台任务输出流（BackgroundTaskOutputStream）——它拥有 fd + DispatchSource + 定时器、有 path 身份与 start/stop 生命周期、不碰 NSView，是 Service（ch.1.2 / ch.9），不是 VM：它做的是 I/O，不是「视图形状的派生」。VC 注入它、观察 $text（数据下行），在 viewWillDisappear 拆卸。
④ sed 解析——SedEditParser / ShellTokenizer 是无状态纯逻辑 enum（Util），SedEditInfo 是带纯 apply(to:) 的 Model。无状态、无 UI ⇒ 无 VM。

为什么门槛必须这么严

ViewModel 的价值是「把可测的派生逻辑从视图里拎出来」；它的成本是「多一个可变引用类型、多一层间接、多一份要同步的状态」。当逻辑只是一行判空或一次映射时，收益为零、成本照付——凭空多一个可变副本、测试收益为零，还诱导后来者往里塞更多。所以门槛是逻辑分量：有多问题状态机（AskUserQuestion）、有异步对账（InputChromeOptions）才诞生 VM；纯映射（PermissionCardKind）、纯派生显示（Presenter）、纯 I/O（Service）、纯逻辑（Util）各归各位。判据不是「它背后原来是不是 @Observable」，而是「拎出来的逻辑值不值一个可单测的类」。

BackgroundTaskOutputStream → BackgroundTaskOutputStreamService旧类是个 @Observable，很容易被误当 ViewModel 迁移。但它 tail 一个 spool 文件（fd + DispatchSource + backstop 定时器），有 path 身份、start/stop 生命周期、零 NSView 依赖——按 ch.1.2「有身份、有生命周期、封装 I/O、被共享」的定义，它是 Service。归错成 VM 会把「文件 tail 的真相源」和「视图形状的派生」混为一谈；归成 Service，则详情 VC 注入它、订阅 $text 数据下行、在离场时 stop()，资源生命周期与「在场/离场」对齐（ch.3.4 / ch.9.1.4）。
一个必须随迁移一起修的 open concern

PermissionFileWriteCardPresenter / PermissionSedEditCardPresenter 在卡片 configure 时做同步文件读 + diff（String(contentsOfFile:) + SedEditInfo.apply），可能在主线程读大文件/网络卷——违反 ch.9.2「重 I/O 不得阻塞主 RunLoop」。上游已对 Edit 模式避免读盘；迁移时 Write/Sed 的 ideal 形态是把「读+diff」丢给一个小 DiffPreviewService 异步算，回来再 set diff 视图，或限制读取字节数。这是朴素照搬会引入的主要正确性/性能风险，Presenter 保持纯值不变，脏活挪进 Service。

本质

视图层迁移只有一句话：让叶子变笨，让绑定手工焊，让 ViewModel 只在够分量时才诞生。每个 SwiftUI 叶子 → 程序化 NSView，只显示（configure 数据下行）、只上报（闭包/delegate/target-action 事件上行）；双向 @Binding 拆成两条单向边，托管消融连同 sizingOptions 体操一并消失。接线按 §7.1 的表选最窄机制。而 ViewModel 的门槛最见功力——CCTerm 的输入栏控件群里，只有 AskUserQuestionViewModel（多步状态机）与 InputChromeOptionsViewModel（异步目录对账）两个够格；权限卡种类是纯映射、卡体是纯 Presenter、输出流是 Service、sed 是 Util，四类都不配 VM。判据始终是「拎出来的逻辑值不值一个可单测的类」，而不是「它原来是不是 @Observable」。

思考题：迁移 chrome 行时，有人给每一个控件都配了 ViewModel——ModelEffortPickerViewModel、PermissionModePickerViewModel 各自去读 ModelStore.shared 算 visibleModels；又给权限卡的 shell 体配了 ShellCardViewModel，其唯一逻辑是「把 request.command 拼成一行字符串」；还把 BackgroundTaskOutputStream 直接改名成 BackgroundTaskOutputViewModel。请按 §7.5 的门槛，指出这三处的归属错误并给出正确形态。

查看答案

错误一：两个 picker 各配一个 VM，逻辑重复。两者都要「把异步到达的 ModelStore 目录与会话状态对账」，各配一个 VM 会把同一份 activeModel/visibleModels 逻辑抄两遍，且目录时序规则散在两处难测。
→ 修法：合成一个 InputChromeOptionsViewModel，由 InputBarSessionChromeViewController 持有，同时喂两个笨触发 view + 各自的 popover VC。VM 该有——但只该有一个（§7.5 该有 VM ②）。

错误二：为「拼一行字符串」造 ShellCardViewModel。把 request.command 拼成显示串是一次纯派生，无异步、无交互态、无身份——不配一个可变引用类型的 VM。
→ 修法：删掉它，落成一个纯 PermissionShellCardPresenter 值（cell-view-model 模式），PermissionShellCardBodyView 只 configure(with:)（§7.5 不该有 VM ②）。

错误三：把 BackgroundTaskOutputStream 改名成 VM。它拥有 fd + DispatchSource + 定时器、有 path 身份与生命周期、不碰 NSView——这是 Service，不是 VM。改名成 VM 会误导后来者把它当「视图形状的派生」而非「文件 tail 的真相源」。
→ 修法：BackgroundTaskOutputStreamService，被详情 VC 注入、$text 下行、viewWillDisappear 里 stop()（§7.5 不该有 VM ③）。
本质：三处同一个病根——把「原来是 @Observable」当成「该配 VM」。正确判据是逻辑分量：够分量且可单测才抽 VM，纯派生落 Presenter，纯 I/O 落 Service，且同一份逻辑只该有一个 VM。

列表与数据源

CCTerm 里两块最吃重的界面都是列表：左侧的会话树（旧 SidebarViewController，NSOutlineView）与中央的聊天记录（旧 NativeTranscript2，NSTableView + Core Text 自绘）。它们恰好落在「列表规范」的两个极端——侧栏是规矩的 view-based 树，转录是被性能逼到自绘的热点表。本章把这两块放到同一张镜头下：dataSource / delegate 如何分工、为什么一律 view-based、复用标识符怎样常量化、行内容怎样建成 cell view model、什么时候用 diffable 快照、什么时候必须刻意 reloadData/精确 delta，以及转录为什么允许下沉到手动 layout()。§8.9 专门给出转录引擎的逐类型归层——因为「转录引擎那么多类到底各属哪一层」正是本次重构最容易含糊的地方。规范手册第 8 章给出通用做法；本章是它在 CCTerm 上的落地，并把两条不可动摇的红线钉死：转录的性能契约神圣不可侵犯，侧栏那条「选中→路由」的同源同相写入不可绕开。

view-based 列表每行/每节点由一个真正的 NSView（NSTableCellView 及其子类）承载。CCTerm 两处列表都 view-based；转录更进一步，在 cell 内用 Core Text 自绘正文。
dataSource / delegate 分工dataSource 回答「有多少行/节点、对应哪份数据」；delegate 回答「长什么样、多高、选中后怎样」。数据真相在 Store/Model，二者都是它的只读投影。
cell view model把「这一行显示什么」建成一个已算好的值（struct）。cell 只做幂等绑定，不反查全局、不临场格式化。
精确 delta（imperative delta）不交给框架 diff，而是把「第 N 行插入/该行状态变了/本回合结束」这样的已算好的增量直接推给自绘表。CCTerm 转录的唯一变更通道，理由见 §8.5。

8.1 两块列表，一张镜头

先把两处列表在分层里的归属讲清楚——这也顺带填掉「侧栏/转录到底属于哪一层」的空白。二者都是纯 AppKit 子系统，迁移是重新归类，不是重写。

SidebarViewController / SidebarTreeModel / SidebarItemModel侧栏的类型按第 1 章五层各归其位：SidebarTreeModel / SidebarItemModel 是 Model（稳定身份的树节点，值类型）；SidebarCellViews / SidebarStatusIndicatorView / SidebarShimmerLabel / SidebarLoadingDotsView 是 View；SidebarViewController 是 Controller；SidebarContextMenuController 坐在 Controller↔Coordinator 边界（右键流程编排）；旧 SidebarSessionGroupOrderStore 是 Store；SidebarLayout / SidebarTitleSanitizer 是纯派生工具（就近扩展，非 Utils 大杂烩）。
NativeTranscript2 / Transcript2Controller / Block转录渲染器自成一域，是重命名 + 归类，不是重写：全域的 Transcript2* 前缀里的「2」是「第二次重写」的迁移遗留，落地时逐一去掉数字（Transcript2Controller→TranscriptController 等）。§2 性能契约的每一条逐字保留——改名纯属外观。详细逐类型归层见 §8.9。
为什么先归类再谈接口

侧栏与转录被多个域引用（旧 SidebarContext 里带 SidebarSessionGroupOrderStore；旧 TranscriptSwapCoordinator 持有转录的每-attach 滚动视图与 bridge）。若不先钉死「节点是 Model、表控制器是 Controller、bridge 是 Service」，跨域改动就会各自发明一套命名与归属。归类是第 2、3 章规范在这两块上的先决条件。

8.2 职责划分：dataSource 管数据，delegate 管样子

view-based 列表的两个协议分工是刚性的（规范手册 §8.1）。CCTerm 两处都遵守，但两条列表读的「数据快照」来源不同：侧栏读 SidebarTreeModel，转录读 TranscriptCoordinator.blocks。

应当

侧栏：dataSource 回答「某节点有几个子节点/第 i 个是谁/是否可展开」，delegate 的 outlineView(_:viewFor:item:) 返回复用后的 cell，选择走 outlineViewSelectionDidChange。转录同理：dataSource 只报 blocks.count，delegate 只出行视图与行高。

避免

在 viewFor / heightOfRow 里改数据、发请求、从 cell 回写 model。这些回调滚动时高频且乱序调用，任何副作用都会变成竞态与卡顿——转录尤甚，它每帧要出几十行。

为什么这条对转录是生死线

转录的正文由 Core Text 在 cell 里自绘，heightOfRow 要靠已按当前宽度排好的版来量高（答案来自 layoutCache 命中，未命中则同步算，无估算高度）。若把「排版」混进 dataSource/delegate 的读路径，滚动时就会反复重排、掉帧。正确形态是：排版结果预先算进 RowLayout（Model）并 memo 进缓存，delegate 只把成品 draw 出来——delegate 是纯投影，不产生真相。

8.3 一律 view-based；复用标识符常量化

两处都必须 view-based（cell-based 的 NSCell 无法承载子视图、控件事件与无障碍，是 10.7 前的遗产）。复用的键——NSUserInterfaceItemIdentifier——必须常量化：裸字符串写错只会静默返回 nil（表现为空白行/空白节点），编译器不报警。这一条也呼应转录的行复用键：旧 CenteredRowView 配 "BlockRow" 标识符，给 NSTableView 一个稳定的 row-view 复用键（§2.17）。

应当

把所有复用标识符集中成 extension NSUserInterfaceItemIdentifier 的静态常量；注册端与 makeView 取用端引用同一常量。

extension NSUserInterfaceItemIdentifier {
    // 侧栏：会话节点 / 分组节点
    static let sidebarSessionCell = NSUserInterfaceItemIdentifier("SidebarSessionCell")
    static let sidebarGroupCell   = NSUserInterfaceItemIdentifier("SidebarGroupCell")
    // 转录：block cell 复用池 + 行视图复用键
    static let transcriptBlockCell = NSUserInterfaceItemIdentifier("TranscriptBlockCell")
    static let transcriptRow       = NSUserInterfaceItemIdentifier("BlockRow")   // §2.17 稳定 row-reuse 键
}
避免
// 裸字符串散落：写错一个字母 → 静默 nil → 空白节点，无编译告警
let v = outlineView.makeView(
    withIdentifier: NSUserInterfaceItemIdentifier("sidbarSessionCell"), // typo
    owner: self)

8.4 cell view model：把「显示什么」建成值

行内容建成值类型，cell 只做幂等绑定——这是可测性与稳妥 diff 的共同前提（规范手册 §8.4）。CCTerm 的关键纪律：cell 不认识 SessionManager、不读 .shared、不在 viewFor 里 new DateFormatter。侧栏行的相对时间、标题清洗（旧 SidebarTitleSanitizer）、状态语义，都提前算进节点的显示模型。

应当

为每种行定义一个 Hashable 的显示模型；cell 暴露 configure(with:)，绑定必须幂等（复用时把每个字段都写一遍，不依赖上一次残留）。id 承载身份，其余字段承载内容——这个拆分是 §8.5 diff 的基础。

// 侧栏会话节点的显示模型：身份(id) 与内容分离；时间/标题已算好
struct SidebarSessionRow: Hashable {
    let id: String            // 稳定身份：既作 outline item，也是选中的通货
    let title: String         // 已过 SidebarTitleSanitizer
    let subtitle: String      // 已格式化的相对时间
    let isRunning: Bool       // 决定状态圆点，而非让 cell 反查 Session
}

final class SidebarSessionCellView: NSTableCellView {
    private let titleLabel = NSTextField(labelWithString: "")
    private let statusDot  = SidebarStatusIndicatorView()

    func configure(with row: SidebarSessionRow) {   // 幂等：每字段都写
        titleLabel.stringValue = row.title
        statusDot.isRunning    = row.isRunning
    }
}
避免
// cell 直接持领域对象 + 临场格式化 + 反查全局：热点路径三重罪
func configure(with session: Session) {
    let df = DateFormatter(); df.dateStyle = .short      // 滚动时反复 new，昂贵
    subtitleLabel.stringValue = df.string(from: session.updatedAt)
    statusDot.isRunning = SessionManager.shared.isRunning(session.id) // 反查全局
}
为什么

view model 把「决定显示什么」（业务，可测）和「把它画出来」（视图，无逻辑）切开。cell 成了纯函数式投影：同样的 row 得同样的样子。这既让侧栏 cell 能脱离窗口单测，也让内容变更可用值相等性判定——身份留给 id，内容变更走重绑，正是下一节 diff 的支点。转录的 RowLayout 是同一思想推到极致：它是 (block, width, state) 的纯函数产物、可缓存、可脱主线程算（§8.9）。

8.5 diffable 还是刻意 reloadData：两条列表两种答案

这是本章的分水岭。规范手册 §8.5 推荐动态列表用 NSTableViewDiffableDataSource 提交快照、让框架算增删移。CCTerm 的两条列表刻意给出相反的答案——因为它们的约束不同。

侧栏：适合快照式更新（身份稳定、量级有限）

应当

侧栏会话树的增删改移用稳定身份表达：item 标识符是会话 id（String），内容变更（改名、状态圆点）走对该 id 的重绑（reloadItems / macOS 13+ 的 reconfigureItems），不把整个 SidebarSessionRow 当标识符——否则改个状态会被当成「删旧插新」，动画错乱且丢选择。

树形数据的取舍

AppKit 侧没有 NSOutlineView 的「分层快照」等价物（UIKit 才有 section snapshot），NSTableViewDiffableDataSource 的快照本质扁平。侧栏的分组折叠与拖拽换组（旧的 pasteboardWriterForItem/validateDrop/acceptDrop 三件套 + expandItem/collapseItem 内建动画）用经典 NSOutlineViewDataSource 往往比强套 diffable 更清晰、更可控。规范：会话行的增删可走快照；树结构与拖放走经典数据源。不要为「统一」硬上 diffable。

转录：禁止 diff，唯一通道是精确 delta（性能契约）

避免

对转录用快照 diff 或一把梭 reloadData()。转录动辄上万行、正文 Core Text 自绘、流式逐字追加——让框架去 diff 一份上万项的快照，或整表 reload，都会掉帧、丢滚动位置、打断流式动画。

应当

转录只接受已算好的精确 delta：旧 Transcript2EntryBridge.apply(_:) 把「第 N 行插入 / 该行状态改了 / 本回合结束」这样的增量直接喂给 TranscriptController（内部走 insertRows/reloadData(forRowIndexes:) 的精确索引，绝不 reloadData()——§2.11），而不是让它自己去 diff。回填（旧 TranscriptBackfillPipeline）从后台产出页，喂进同一个 apply 入口——绝不开第二条变更通道（§3.1）。

// 转录的单一变更通道：把 Model 层算好的增量交给自绘表
enum MessagesChange {                 // 已算好的 delta，不是整份快照
    case appended(range: Range<Int>)  // 新行区间
    case reloaded(rows: IndexSet)      // 身份不变、内容变了的行
    case turnFinished
}

@MainActor
final class TranscriptEntryBridge {   // Service：喂表的唯一入口（旧 Transcript2EntryBridge）
    private weak var controller: TranscriptController?
    func apply(_ change: MessagesChange) { /* → controller 精确 insert/reload */ }
}
为什么转录反着来

规范手册的「diff 优先」成立于「行数有限、内容 Hashable、diff 便宜」的前提。转录三条都不满足：行数无上界、内容是排好版的 Core Text 行、流式追加要求零打断。这里正是根 CLAUDE.md 说的「当正确性依赖于 @Observable 表达不出的 runloop tick 时，才允许跨界的命令式调用」——精确 delta 把一次已知增量在同一 source 阶段推给表，而 diff 会把它推迟到 beforeWaiting 再算，跨帧撕裂流式动画。这条性能契约神圣不可侵犯：任何「为了统一而给转录套 diffable」的改动都要被驳回。

8.6 选择处理：pull 模型，且侧栏那条边是唯一结构上行边

选择的权威在 NSOutlineView/NSTableView 自身（selectedRow），controller 不另存索引——行会因增删位移，缓存的索引立刻失真（规范手册 §8.6）。需要「选中的是谁」时现查，并立即转成稳定 id 上传。

但侧栏的「选中→上报」不是一条普通回调，而是 CCTerm 全局唯一的结构性上行边，第 5、7 章反复强调过它，这里给出它在列表层的落点。

应当

侧栏在 outlineViewSelectionDidChange 里，把选中节点转成 NavigationTarget，同步、同一 source 阶段写入旧 MainSelectionModel（迁移后 WindowNavigationStore）的 set(_:)（旧 select(_:)）；后者立刻触发那条唯一的结构性 delegate（旧 MainSelectionObserver → WindowNavigationObserver），由详情路由在点击的同一帧换详情、把 sessionID 交给转录。

extension SidebarViewController: NSOutlineViewDelegate {
    func outlineViewSelectionDidChange(_ notification: Notification) {
        let row = outlineView.selectedRow
        guard row >= 0,
              let item = outlineView.item(atRow: row) as? SidebarItemModel
        else { return }
        // 只上报稳定身份；同步写入，落在点击的 source 阶段——不 dispatch、不 Task
        navigation.set(item.navigationTarget)   // WindowNavigationStore.set(_:)，旧 select(_:)
    }
}
避免

把这条边改成 @Observable 追踪、Combine、或 NotificationCenter 广播；也别把 selectedRow（整型索引）存进 model 或跨层传，或把 NSTableCellView 实例传出列表（随时被复用给别的行）。

为什么这条边不能异步、不能泛化

@Observable/Combine 的重算落在 beforeWaiting，比点击晚一个 tick；而详情换页 + 转录必须在同一帧完成，否则转录会先以错误宽度排一次版、下一帧再重排，产生可见闪烁（第 7 章 runloop 模型 + §2.19 单宽度 attach 契约）。所以它必须是同步 delegate，且是唯一一条结构上行边——「对选中做结构响应」的新需求一律走路由，不得再加第二个观察位（这是明确的非目标）。侧栏高亮、工具栏项目 chip 这类非结构的选中响应，则走普通 closure 下行，与这条结构边分开。

8.7 性能：复用 + 定高 + 转录的手动 layout()（热点例外）

列表性能第一原则是复用（§8.3 已保证），其次是行高与 cell 内布局成本。侧栏行结构简单、量级有限，用 Auto Layout + 合理行高即可流畅，不必手写 layout()。转录则是第 6 章「测量确认的高频重排热点」的教科书案例。

应当（转录）

转录 cell 内不用 Auto Layout：正文用 Core Text 自绘，子视图坐标在 layout() 里按已排好的版直接算 frame，绕开约束求解；行高由排版结果给出并缓存进 layoutCache（§2.4，无 LRU），delegate 的 heightOfRow 直接读缓存，不现算。cell 用 wantsLayer + .onSetNeedsDisplay 缓存位图（§2.2），滚动时 GPU 合成缓存位图、零 draw(_:)。这与第 6 章「性能热点才手动布局，且需 profiler 证明」一脉相承。

避免

给侧栏 cell 也手写 layout()「图性能」——拿不到可测收益，还丢掉自适应与可读性。手动布局是转录这类被 profiler 点名的热点的局部例外，不是默认口径；也别在同一视图里对同一子视图混用约束与手动 frame（约束会覆盖 frame，产生随机错位）。

为什么只有转录配得上手动布局

手动 layout() 用几行算术换掉约束求解，只在「每秒重排几十上百次 × 规则可公式化」时才回本。转录满足：快速滚动 + 流式刷新是高频重排，正文按宽度排版后行内元素位置是固定公式，且它的「按宽度测高」算法还能脱离视图直接算行高（省一趟量高 pass）。侧栏三者都不满足，付这个可维护性代价纯属亏本。

8.8 拆卸：列表的每-attach 资源随离场释放

第 3 章的确定性拆卸在这里有具体落点：转录的每-attach 资源（每次 attach 才建的滚动视图、旧 Transcript2SheetPresenter、回填任务、isRunning 观察）由旧 TranscriptSwapCoordinator 拥有，在详情子 VC 的 prepareForRemoval() 里显式释放——而非等 ARC。切换动画会让离场 VC 多活约 300ms，期间若不显式停掉，回填仍在对已离场的表 apply。

一个必须随迁移修掉的旧隐患

旧 TranscriptSwapCoordinator.startRunningObservation 用 withObservationTracking 包在一个 Task 里观察 isRunning——它在未来某个 source 阶段才 resume，与「同步换页」的承诺相冲。迁移必须把它改成 Session.onIsRunningChange 的同步 sink：在 runtime 写点直接回调，驱动 controller.setLoading(_:) 与输入栏的「发送↔停止」切换。这属于 Services/Session 域必须先补的显式同步通道，否则转录/输入栏无法脱离 SwiftUI 编译。

本质

CCTerm 的列表规范就是把三件事分离，并按约束选更新策略：数据真相（SidebarTreeModel / TranscriptCoordinator.blocks，唯一）、视觉投影（view-based cell + 幂等 configure，纯函数式）、更新表达——侧栏交给快照 diff（身份稳定、量级有限），转录只走精确 delta（行数无界、Core Text 自绘、流式零打断）。把身份（稳定 id）与内容（Hashable 值）拆开，越界崩溃/丢选择/滚动闪烁三类经典 bug 就从根上消失；复用与定高保证流畅，手动 layout() 只在被 profiler 点名的转录出场。转录引擎那 60+ 个类的迁移是去数字化 + 归类：Controller / Coordinator / Model（布局值）/ View（自绘 cell）/ Service（高亮 Store）各归各位，两个 SwiftUI sheet 正文端到 AppKit，性能契约一字不改。两条不可动摇：转录的性能契约、以及侧栏那条同步同相的「选中→路由」结构边。

思考题：有人提议「统一列表更新策略」——把转录也改用 NSTableViewDiffableDataSource，每次收到新消息就把 blocks 全量转成一份快照 apply，并顺手把侧栏的「选中→换详情」从同步 delegate 改成对 WindowNavigationStore 的 @Observable 追踪，「这样两块列表就一致了」；同时他觉得 TranscriptController 那些 @Observable 字段「看着就是个 ViewModel」，想把搜索框的显示字符串也搬上去做成计算属性。请逐条指出后果并给出正确做法。

查看答案

转录改 diffable 全量快照——违反性能契约，必须驳回。转录行数无上界、每行是 Core Text 排好的版、消息流式逐字追加。全量快照意味着每来一个 token 就构造一份上万项的快照让框架 diff：(1) diff 本身在大集合上昂贵，掉帧；(2) diff 结果落在 beforeWaiting，比「收到增量」的 source 阶段晚一个 tick，打断流式动画；(3) 整表参与 diff 易丢滚动位置。正确做法：保持唯一变更通道 TranscriptEntryBridge.apply(_:)，把已算好的精确 delta 在同一 source 阶段推给 TranscriptController（内部精确 insertRows/reloadData(forRowIndexes:)，绝不 reloadData()）；回填喂进同一个 apply，不开第二条通道。

侧栏选中改 @Observable 追踪——破坏同源同相的结构边，必须驳回。@Observable/Combine 的重算落在 beforeWaiting，比点击晚一个 tick；而「换详情子 VC + 交 sessionID 给转录 + 转录以最终宽度排一次版」必须在点击的同一帧完成（§2.19）。晚一 tick 会让转录先以错误宽度排版、下一帧再重排，产生可见闪烁。正确做法：outlineViewSelectionDidChange 里同步调 WindowNavigationStore.set(_:)，由它同步触发唯一的结构 delegate（WindowNavigationObserver），路由同帧换页。

把搜索显示字符串搬上 TranscriptController——误把命令通道镜像当 ViewModel，必须驳回。那些 @Observable 字段是 coordinator/search sink 单向下推的镜像，不是双向可绑定的派生态。把「搜索框该显示什么」的格式化搬上去，就是让 Controller 悄悄漂向 ViewModel——而转录引擎明令不设 ViewModel（命令通道就是设计要点）。正确做法：派生的显示格式化留在宿主（读 searchState 现算），镜像字段保持单向下推。

本质：「一致」不是目标，各自满足自身约束才是。侧栏身份稳定、可容忍一 tick，适合快照 diff + 普通回调；转录无界、自绘、流式、选中换页与排版严格同帧，必须精确 delta + 同步结构边 + 无 ViewModel。强行统一等于用错误的策略去满足错误的约束。

状态与数据流 · ViewModel 取舍

前面几章把 CCTerm 的类型逐一归了层：Model 是值、Service/Store 是状态的主人、Coordinator 管流程、View 只显示上报。本章回答最后一个、也是纯 AppKit 化里最容易做过头的问题——ViewModel 到底什么时候该有、什么时候绝不能有。删掉 SwiftUI 意味着删掉 @Observable 的自动依赖追踪，于是"状态怎么流到界面"这条线要重新手工接。CCTerm 的答案不是"给每个 VC 配一个 VM"，而是把接线收敛到两条明确的下行通道 + 三种上行机制，并在整条链上只保留 8 个真正挣到位置的 ViewModel。核心口径承接第 7 章：选最窄够用的机制，数据向下、事件向上，聊天区永远不要 ViewModel。

ViewModel（在本手册的严格定义）介于 View 与 Model 之间、不 import AppKit、把 Model 转成"视图可直接显示的成品值"并用 Combine @Published 发布、把用户意图作为方法调用接收进来的普通 class。它有可被单测的表现/派生逻辑，且这逻辑值得脱离窗口验证。
命令式增量通道（imperative delta channel）跨 AppKit↔渲染器边界的一个直接方法调用，把一份已算好的 delta 交给自绘 NSTableView，绕开 diff。CCTerm 里就是旧 Transcript2EntryBridge.apply(_:) 这条唯一被批准的例外通道。
结构性上行边（structural upward edge）选择变化触发详情面板换页这条边。它必须在点击的同一 source phase 同步跑完，@Observable/Combine 会晚一拍到 beforeWaiting，撕裂换页。CCTerm 全局只有这一条，走同步 delegate。

为什么是"两条下行"而不是"一条"

绝大多数 UI 靠通道 A 就够了——它是 SwiftUI @Observable 的直接替身，类型安全、生命周期由 AnyCancellable 显式管。但自绘 transcript 是例外：它每秒可能收到几十条流式增量，若把"新增一行"表达成"发布整份 [Block] 让表去 diff"，既算不准（稳定 id 语义丢失）又太慢（§2 性能契约）。所以旧 Transcript2EntryBridge 保留一条命令式 apply(_ change:) 把精确 delta 直接喂给自绘表——这正是根 CLAUDE.md 批准的"只有当正确性依赖 @Observable 表达不了的 runloop tick 时，才允许跨边界命令式调用"。通道 B 是被严格限定的一个例外，不是第二种默认做法。

9.2 ViewModel 什么时候挣到了位置

第 1 章说过"架构是用来还债的，不是用来预付的"。ViewModel 的准入门槛就是这句话的落地：一个 VM 只有同时满足三条才该存在——(a) 有真实、非平凡或有状态的表现/派生逻辑；(b) 这逻辑值得脱离窗口被单测；(c) 它不 import AppKit（第 7 章：VM 不碰视图）。如果"派生"只是一次判空、一行格式化、或对着一个已持有真相的 Store 做透传 setter，那它就不是 VM——留作自由函数 / 纯 Model 助手 / VC 的私有计算属性即可。

应当（VM 挣到了位置）

把"有状态、可测、无 AppKit"的派生收进 VM，用 @Published 发布成品值：

import Combine   // 注意：不 import AppKit —— VM 不碰视图

    @Published private(set) var candidates: [CompletionCandidate] = []

    @Published private(set) var isLoading = false

    func moveSelection(by delta: Int) { /* 循环选中导航 */ }
    func confirm() -> Replacement? { /* 词范围 + 替换算术 */ }
}
避免（不该造 VM）

为"三个透传 setter + 一个判空"套一层 VM，只会把 Store 自己的 setter 再暴露一遍：

// 反例：ComposeSessionView 的"逻辑"其实是往 SessionDraft(Store) 转写
final class ComposeViewModel {              // ← 不该存在
    var cwd: URL? { didSet { draft.cwd = cwd } }        // 透传
    var model: String? { didSet { draft.model = model } } // 透传
    var submitEnabled: Bool { draft.cwd != nil }          // 一行判空

// 正确：笨 View + delegate 上报，draft(SessionDraft Store)就是唯一真相
为什么门槛这么高

CompletionEngine（旧 CompletionState）能挣到位置，是因为它装着触发匹配、150ms 去抖 + 50ms loading 闩锁、换代取消、循环选中、词范围与替换算术——喂进文本/光标就能断言 @Published 输出，全程零窗口。而 compose 卡片的"逻辑"是三个写回 SessionDraft（Store）的 setter 加一个 cwd != nil：真相本就在 Store 里，包一层 VM 只是把 Store 的 setter 换个壳再导出一遍，凭空多一个可变副本、一次注入仪式，测试收益为零。门槛高，是为了让"每个 VC 配个 VM"这种 SwiftUI 惯性在准入处就被挡下。

9.3 ViewModel 绝不能出现的地方

比"哪里该有 VM"更重要的是"哪里绝不能有"。CCTerm 有一批表面像 VM、实则必须是别的层的东西；给它们套 VM，轻则空转样板，重则毁掉一条必须同步跑的时序契约。

避免：聊天区 / transcript 的协调 ViewModel

这是 CCTerm 明令的非目标（Content/Chat/CLAUDE.md）：没有会话/transcript 协调 ViewModel。旧 TranscriptSwapCoordinator 是 Coordinator（拥有 currentSession + 那条 source-phase 同步换页），不是 VM；旧 Transcript2EntryBridge 是 Service（通道 B 的唯一命令式增量入口）。把它们任一改叫 VM，都会模糊 Service/Coordinator 边界，并诱使有人再造一个"聊天区总 VM"，把进程/窗口/会话三个作用域拍扁成一个 reducer。

为什么聊天区宁可命令式也不要 VM

一个 VM 的产物是"发布出去等谁来订阅"的显示值；而 transcript 的正确性依赖在哪一拍把 delta 交给自绘表。旧 Session.onIsRunningChange（替代 withObservationTracking 的同步闭包）→ TranscriptSwapCoordinator → controller.setLoading(running)、以及 Transcript2EntryBridge.apply / setTurnUsage，都必须在运行时写入点同步触发——一旦经 VM 的 @Published 走一趟 beforeWaiting，send↔stop 切换和行增量就会跨帧撕裂。VM 在这里不是"多余"，而是"会引入一拍延迟的错误抽象"。

9.4 那条唯一的结构性上行边

整个数据流里有一条边不走上面任何一种常规机制，必须特判：侧栏选中 / 通知激活 → 详情面板换页。它是 CCTerm 唯一的"结构性上行边"，旧 MainSelectionModel.select(_:) 触发旧 MainSelectionObserver，纯 AppKit 化后是 WindowNavigationStore.set(_:) 触发 WindowNavigationObserver。

应当

把它保留为同步 delegate，在点击的同一 source phase 内跑完；delegate 遵循 AppKit sender-first 约定，唯一注册者是详情 Coordinator。

// 旧 MainSelectionObserver → 新 WindowNavigationObserver：同步、源相位、sender first
@MainActor
protocol WindowNavigationObserver: AnyObject {
    func windowNavigation(_ store: WindowNavigationStore,
                          didChangeTo target: NavigationTarget)
}

@MainActor
final class WindowNavigationStore {                 // 旧 MainSelectionModel（去掉 @Observable）
    private(set) var target: NavigationTarget       // 旧 selection
    weak var structuralObserver: WindowNavigationObserver?  // ↑ 弱、同步、唯一
    var onTargetChange: ((NavigationTarget) -> Void)?       // 非结构性下行（ch.7.4 闭包）

    func set(_ target: NavigationTarget) {          // 旧 select(_:)：改值 + 同步回调
        self.target = target
        structuralObserver?.windowNavigation(self, didChangeTo: target)  // 就地同步
        onTargetChange?(target)
    }
    nonisolated deinit {}
}
避免

让详情路由用 withObservationTracking / Combine 去"观察"选择变化；也别把这条边泛化成第二个 observer 槽或 NotificationCenter 广播。

为什么它不能是通道 A

结合第 7/8 章的 runloop tick 模型：@Observable 与 Combine 的 body/闭包重算都落在 beforeWaiting，比点击的 source phase 晚一拍。选择一变就要在同一拍内完成 container.swap(...) + (child as? SessionPresentingDetailChild)?.present(sessionID:)，让 transcript 在最终宽度上一帧内 typeset（§2.19 单宽度契约）；晚一拍就会看到旧页闪一下再换、或 transcript 在错误宽度上先排一次版。所以这条边只能是同步 delegate，跑在 click 的源相位里。至于"非结构性"的选择反应（侧栏高亮、工具栏 chip 有无、archive 过滤），它们不撕帧，走 onTargetChange 闭包下行即可——这正是 ch.7.4 规定的窗口导航广播用 delegate/闭包、不用 Combine。

MainSelectionModel → WindowNavigationStore旧 MainSelectionModel 是 @Observable，被侧栏单元格、输入栏 isComposeMode、compose 配置器、archive 过滤等多域同时读。这也是纯 AppKit 化里风险最高的一处跨域耦合：只有当所有读它的 SwiftUI 消费者都已改成 AppKit，才能安全摘掉 @Observable；否则要么给 Store 留一条兼容观察路径，要么按域排期（侧栏/详情先行）。draftSessionId 仍是可写 var——它被 ComposeSessionViewController 与 SessionInputSubmit 直接写/清，端口迁移时这些赋值要一起改指到 WindowNavigationStore.draftSessionID。

9.5 一个被删掉的反面教材：计数器泵

SwiftUI 里有一种惯用法：用一个单调自增的计数器发一个"无值可 diff"的 @Observable 变化，让 .onChange 收到边沿去执行一次性动作。旧 TranscriptSearchBus 的 focusRequestCounter 就是这么给工具栏搜索框"抢焦点"的。纯 AppKit 里它没有存在理由。

避免：计数器泵（SwiftUI 惯性）
// 旧 TranscriptSearchBus：为了让 .onChange 有值可 diff，被迫自增计数器
@Observable @MainActor
final class TranscriptSearchBus {
    private(set) var focusRequestCounter = 0
    func requestFocus() { focusRequestCounter += 1 }  // 没有语义的"边沿"
}
应当：一次性命令（1:1 delegate）
// 新 TranscriptSearchFocusing：⌘F 就是一条 1:1 fire-and-forget 命令
@MainActor
protocol TranscriptSearchFocusing: AnyObject {
    func focusTranscriptSearch()          // 无计数、无观察
}
// MainWindowController 持有 NSSearchToolbarItem → 遵此协议；
// Find 菜单动作直接调 focusTranscriptSearch()
为什么计数器泵该整体删除

它是"用状态假装事件"：抢焦点是一个命令（做一次就完），不是一份状态（可被订阅、可被读当前值）。计数器只是为了迁就 .onChange 需要"值变了"才回调而硬造的。纯 AppKit 里 Find 菜单项到搜索框宿主是一对一、无返回、无历史——正是 ch.7.1 的 delegate/闭包行。删掉 @Observable 对象、删掉计数器、删掉每会话边界重建的顾虑，只留一个单方法协议由 MainWindowController 遵守。识别信号：任何"自增一个没人读其数值的整数、只为触发一次副作用"的字段，都是一条被误建成状态的命令，删掉换 delegate/闭包。

本质

纯 AppKit 化在数据流上做的，是把 SwiftUI 代管的"依赖追踪 + 自动拆线"接回自己手里，然后抵住"给每个 VC 配 VM"的惯性。规范收敛成一句：显示值的流用通道 A（Combine @Published），精确到帧的渲染增量用通道 B（命令式 apply，仅 transcript），一次性意图用闭包/target-action，需回话的 1:1 协作用 delegate，而那条会撕帧的选择换页边用同步 delegate 特判——永不用 NotificationCenter 传"一个控制器要知道一件事"，永不用 Combine 承载结构性换页。ViewModel 只在有真实、可脱窗单测、无 AppKit 依赖的派生逻辑时才诞生，全局 8 个；聊天区一个都不要——那里的正确性属于 Coordinator 的时序与 Service 的命令式增量，不属于任何 VM。

思考题：迁移中，有人提议把旧 Transcript2EntryBridge 改造成一个 TranscriptViewModel，用 @Published var blocks: [Block] 发布整份区块数组，让自绘表的宿主 VC sink 后调 reloadData；同时把窗口选择 MainSelectionModel 也改成发布 @Published var selection，让详情 Coordinator 订阅它来换页——"这样两处都统一走 Combine，最干净"。请逐一指出这两个改动为什么违反本章规范，会引发什么具体故障，并给出正确形态。

查看答案

改动一（transcript 发布整份 blocks）错在把通道 B 降级成通道 A。 transcript 每秒可能收到几十条流式增量；发布整份 [Block] 让表去 diff，一是算不准——稳定 id（旧 StableBlockID）驱动的折叠/选中/高亮键空间会被"整份替换"抹掉身份；二是太慢/闪烁——违反 §2 性能契约，且 @Published 的重算落在 beforeWaiting，流式更新会跨帧堆积。正确形态：保留旧 Transcript2EntryBridge 作为 Service，走通道 B 的命令式 apply(_ change: MessagesChange)，把精确 delta（tail .append / older .prepend）在运行时写入点同步喂给自绘表——这是根 CLAUDE.md 批准的唯一命令式例外，不是 VM。

改动二（选择走 Combine 订阅）错在把结构性上行边异步化。 选择变化到详情换页必须在点击的同一 source phase 同步完成；@Published/Combine 的订阅回调落在 beforeWaiting，晚一拍。具体故障：点侧栏后旧页会闪一下才换、或 transcript 先在错误宽度上排一次版再重排（§2.19 单宽度契约被破坏），会话切换在视觉上撕裂。正确形态：这条边保留为同步 delegate——WindowNavigationStore.set(_:)（旧 select(_:)）就地同步调用 WindowNavigationObserver.windowNavigation(_:didChangeTo:)，由 DetailNavigationCoordinator 作唯一注册者，在回调里同步 container.swap(...) + (child as? SessionPresentingDetailChild)?.present(sessionID:)，绝不 hop 到 Task/beforeWaiting。

共同的错误根源是"统一走 Combine 最干净"这个直觉本身：本章的规范恰恰是不追求机制统一，而追求机制与语义/时序匹配。显示值的流才配通道 A；精确到帧的渲染增量走通道 B；会撕帧的换页边走同步 delegate 特判。两处都不该、也都不需要 ViewModel——它们分别是 Service（命令式增量）与 Coordinator（源相位换页）的职责。

内存 · 并发 · 可测试性 · 收尾

前九章把 CCTerm 从「SwiftUI 优先」牵到了「纯 AppKit、MVVM-C」的门口：分层、模块边界、组合根、控制器容器、Coordinator 导航、Session 领域、笨视图、列表数据源、状态数据流都各就各位。本章补上让它长期不崩、不漏、可测的最后一层，并为整场迁移收尾。三条主线同源——都在回答「谁持有谁、谁在哪个线程跑、谁能被替换」：引用关系用 weak/[weak self] 与确定性拆卸写在明处，UI 类型用 @MainActor 钉在主线程，业务对象经依赖注入换 mock、只经公开面驱动测试。CCTerm 有两条自己的硬约束贯穿本章：macOS-26 的 nonisolated deinit 纪律，与「测试绝不逼产品让步」的工程原则。对标规范手册第 9 章。

确定性拆卸（deterministic teardown）对象在明确时机（prepareForRemoval()、会话归档、显式 invalidate()）主动释放它持有的订阅/任务/资源，而非听任 ARC「何时想释放就何时释放」。CCTerm 的 per-attach 资源与长跑 CLI 任务都靠它。
nonisolated deinitCCTerm 全库纪律：每个 @MainActor 引用类型带一个空的 nonisolated deinit {}，斩断 macOS-26 上 @MainActor deinit 执行器跳转在 libswift_Concurrency 里的崩溃。与 @Observable 无关，去 SwiftUI 后照样保留。
@MainActor把「UI 只在主线程」从口头约定升级为编译期强制。CCTerm 的 VC/View/以及直接读写 UI 的 Service 都标它；纯值 Model 与后台 I/O（SyntaxHighlightingService actor、CLIClient）不标。
经公开面测试（drive the public surface）注入 mock、调用公开方法、断言可观察输出（@Published 值、mock 调用记录、weak 引用是否归 nil）。绝不开 forceXxxForTest() 后门、不为断言私有字段放宽访问级别。

10.1 内存：谁持有谁，一眼看清

AppKit 无 GC，问题几乎都不是「分配太多」而是「该释放的没释放」。规范手册 9.1：默认强引用，只有形成环时才降级 weak/unowned，且降级位置遵循固定套路。CCTerm 去掉 SwiftUI 后，那些原本由框架代管的订阅拆线全部回到手上，这一节尤其吃重。

CCTerm 场景环的形状拆解规范
唯一结构边：Store ⇄ 协调器WindowNavigationStore 强持 observer + observer 回调回指weak var structuralObserver（ch.5.4）
view ⇄ VC（delegate）VC 强持笨视图 + 视图强持 VC 作 delegate笨视图 delegate 属性 weak，协议约束 AnyObject
VC ⇄ ViewModel / Store（Combine）VC 持 cancellables → sink 闭包 → 强捕 VCsink 用 [weak self]；AnyCancellable 入 Set，随 VC 释放自动 cancel
父子 Coordinator父强持子（数组）+ 子强持父子对父 weak；父在子完成时主动移除（AppCoordinator）
容器 ⇄ 决策协调器（ch.4/5）协调器持容器 + 容器回指协调器对容器 unowned（向下不成环，ch.5）
Session → runtime → 长跑 CLI 任务任务背着已切走的会话继续跑，改已离场 UI取消绑定到显式生命周期点（归档/prepareForRemoval），非 deinit（ch.6.6）
应当

凡被 self 长期持有、回调里又用 self 的闭包（存储型 Combine sink、Session.onIsRunningChange 订阅、长驻 Task），捕获列表写 [weak self] + guard let self else { return }；返回的 AnyCancellable 存进 Set<AnyCancellable>，让订阅生命周期跟 VC 绑定。

@MainActor
final class ChatInputBarChromeViewController: NSViewController {
    private var cancellables = Set<AnyCancellable>()

    override func viewDidLoad() {
        super.viewDidLoad()
        // send↔stop 切换：订阅会话运行态（一对多、可容忍一 tick → Combine，ch.7.1）
        session.$isRunningPublished
            .receive(on: DispatchQueue.main)               // 保证闭包体回主线程（10.2）
            .sink { [weak self] running in                 // 必须 weak，防 VC 泄漏
                self?.inputBar.configure(canSend: !running, isRunning: running)
            }
            .store(in: &cancellables)                       // 入 Set，随 VC 释放自动 cancel
    }
    nonisolated deinit {}                                  // 见 10.1.1
}
避免

忘记 .store(in:)（AnyCancellable 一被丢弃订阅当场取消，界面「一次都不刷新」）；sink 里强捕 self（VC → cancellables → 闭包 → VC 成环，VC 挂着的整棵 view 树、乃至它引的 Session 一起泄漏）；把长跑 CLI 任务的取消只寄望 deinit（deinit 时机不确定，任务会背着已切走的会话继续触发 onIsRunningChange，改早已离场的 UI，ch.3.4/6.6）。

10.1.1 nonisolated deinit：CCTerm 的全库纪律

CCTerm · CLAUDE.md「Deterministic teardown」项目要求「每个 @MainActor @Observable / VC 类型带空 nonisolated deinit {}」，对抗 macOS-26 上 @MainActor deinit 执行器跳转的崩溃。迁移去掉 @Observable 后这条纪律原样保留——它对抗的是执行器跳转，与 @Observable 无关。范围覆盖迁移后每个 @MainActor 引用类型：WindowController、容器/路由 VC、协调器、每个笨视图与子 VC、WindowNavigationStore、TranscriptSearchCommand、会话领域全部 Service。纯值 Model（NavigationTarget/DetailScene/SessionConfig）无 deinit、也不需要。

10.1.2 确定性拆卸的两个 CCTerm 落点

应当

(1) per-attach 资源随离场释放（ch.4.5/8.8）：转录的 per-attach 滚动视图、Transcript2SheetPresenter、回填任务、isRunning 观察由 TranscriptSwapCoordinator 拥有，在子 VC 的 prepareForRemoval() 里当场释放——crossfade 让离场 VC 多活约 300ms，期间若不显式停，回填仍在对已离场表 apply。(2) 系统级资源显式反注册：Timer/文件句柄/旧式 KVO 在明确时机 invalidate/removeObserver，deinit 只兜底；能随属性释放自动清理的（AnyCancellable、闭包版 NSKeyValueObservation）优先靠属性生命周期。

为什么会话对象尤其要显式拆卸

ARC 只管引用计数，管不了「注册在别处的回调」：Timer 被 RunLoop 持有、SessionRuntime 挂着长跑的 CLI 子进程通信任务——它们不会因你的对象引用计数归零就消失。若只等 ARC「某个不确定时刻」释放，任务会背着一个已被切走的会话继续跑、继续触发 onIsRunningChange，改动早已离场的 UI（ch.3.4 的隐患）。所以运行时取消绑定到显式生命周期点（会话归档 / prepareForRemoval()），而 deinit 只做「切断执行器跳转」这一件与业务无关的事。确定性拆卸把「资源释放」从 ARC 的不确定时机变成代码里可见的一行——泄漏在 review 时肉眼可查。

10.2 并发：UI 只在主线程，耗时活儿丢后台

AppKit 非线程安全，违反的后果不是必崩而是偶发崩溃与诡异渲染——最难查的一类 bug。规范手册 9.2：用 @MainActor 把「UI 只在主线程」变成编译期检查，把耗时纯计算/IO 放后台，返回后自动切回主线程改 UI。

应当

所有 NSView/NSViewController 子类、以及直接读写 UI 的服务标 @MainActor；耗时 I/O/解析放非隔离的 async 函数或 actor（SyntaxHighlightingService 是 actor，CLIClient 后台跑）；驱动 UI 的 Combine 订阅在 sink 前串 .receive(on: DispatchQueue.main)——@MainActor 救不了它，Combine 的调度由 receive(on:) 决定，不看类型的 actor 标注。

避免

在后台线程直接改 NSView；在主线程用信号量/sync 等后台结果（阻塞主 RunLoop → 界面冻结转菊花）；给纯计算/纯值类型也贴 @MainActor（把本可并行的活挤回主线程，等于没做并发，也让它进不了 Core 的无窗口测试）。用 RunLoop.main 做 receive(on:)（拖动/滚动等跟踪模式下可能丢投递，改用 DispatchQueue.main）。

硬前置回顾：同步 vs 异步不是风格，是正确性

本手册两条承重例外都卡在「同步/异步」这条线上，且都与并发规范交叉，收尾时再钉一次：(1) 转录的 setLoading / apply 必须在运行时写点同步触发（Session.onIsRunningChange 直连，绝不 .receive(on:) 慢一拍，ch.1.4/6.4）；(2) 唯一结构边 WindowNavigationStore.set(_:) → observer 必须在点击的同一 source 相位同步跑完，绝不 hop 过 Task/beforeWaiting（ch.1.5/5.4/9.4）。而非时序敏感的 chrome（send↔stop 按钮、侧栏高亮、工具栏 chip）才走可容忍一 tick 的 Combine + receive(on: .main)。判据是时序敏感度：与 typeset/换页同帧的走同步命令式，普通状态推送走 Combine。别为「机制统一」把同步链降级成异步（ch.1 思考题、ch.9 思考题反复驳斥的正是这一点）。

10.3 可测试性：经公开面驱动，绝不开测试后门

CCTerm 没有 XCUITest 目标——点击/键入/焦点流程一律靠「直接驱动 session/bridge/controller」的逻辑测试覆盖（现状文档明载）。这与规范手册 9.3 完全一致：把逻辑从 UI 抽出，让 Service/协调器/ViewModel 能被单元测试直接驱动——注入 mock、调公开方法、断言可观察结果。

应当

协作者以协议类型从 init 传入（ch.3.6 的能力协议：CLIClient/HistoryLoader/SyntaxHighlighting/AppActivationChecking），生产注真实、测试注 mock；断言全走公开可观察输出。

// 唯一的「测试专用代码」，且只在测试 target：mock 记录调用、返回可控数据
final class CLIClientMock: CLIClient {
    private(set) var sentInputs: [InputSubmission] = []      // 可观察：断言被正确调用
    func send(_ input: InputSubmission) async throws { sentInputs.append(input) }
}

@MainActor
@Test func 发送后运行态翻转且CLI收到消息() async {
    let cli = CLIClientMock()
    let manager = SessionManager(cliClient: cli, historyLoader: HistoryLoaderMock(),
                                 repository: SessionRepository(stack: .init(inMemory: true)))
    let session = manager.session(for: "s1")
    var running: [Bool] = []
    session.onIsRunningChange = { running.append($0) }      // 订阅公开的同步变更通道

    session.send(.init(text: "hi"))                          // 只驱动公开方法

    #expect(running == [true])                               // 公开可观察输出
    #expect(cli.sentInputs.map(\.text) == ["hi"])            // 协作被正确触发
}
避免

为测试加 forceReloadForTest()、把 private 改 internal 只为断言内部字段、用 env var 绕过真实逻辑、或 SessionManager.shared 直取。测试够不到真实控制点，要改的是测试（换成驱动公开面：调 handle 方法、发 bridge 事件、喂 controller），不是产品。CCTerm 的工程原则明令：绝不为让测试通过而损害产品代码。

CCTerm · CLAUDE.md「Engineering principles」+ CoreDataStack现状文档把「绝不损害产品代码以通过测试」列为红线，并点名反模式：env var 绕逻辑、forceXxxForTest()、为测试钩子放宽访问级别。正面样例是 CoreDataStack：生产走 private init()（合法单例，ch.3.4），测试走 init(inMemory:) 接缝——用注入的构造入口而非全局后门达成隔离。这正是规范 9.3 的「依赖注入让你能注入 mock」。
为什么测试后门比没有测试更坏

测试后门把测试专用路径永久焊进产品：它增加一条「生产不该走、却真实存在」的分支，让代码更难懂、更易被误用，且测的是后门而非真实行为——绿灯并不证明用户路径正确。反过来，「注入 mock + 驱动公开方法 + 断言可观察输出」测的正是发布代码会走的那条路；重构内部实现而不改公开契约时测试仍成立，这才是防回归的价值。CCTerm 用逻辑测试覆盖 click/keystroke/focus 流程，靠的正是这一点：驱动 Session/bridge/controller 的公开面，而非模拟坐标点击。

本质

内存、并发、可测试性是同一件事在三个维度的投影——把隐式关系变显式。内存：谁强持谁、谁负责释放，用 weak 与确定性拆卸（含 CCTerm 的 nonisolated deinit、prepareForRemoval）写在明处；并发：哪段在主线程、哪段在后台、哪条必须同步，用 @MainActor/await/以及两条承重例外的同步链钉在类型签名与调用点；可测试性：依赖谁、如何被替换，用协议注入与公开面暴露，绝不开后门。而整场从 SwiftUI 到纯 AppKit 的迁移，本质也是同一句话的放大——SwiftUI 曾把三件事托管给框架：View↔Model 的绑定、依赖注入、跨相位的时序缝合。去掉它，就是把这三件事逐一交回你手上：分层照 ch.1 切、模块照 ch.2 立、依赖照 ch.3/4 在组合根装、导航照 ch.5 决策/执行分离、Session 照 ch.6 定层、绑定照 ch.7/9 按基数与方向手工焊、列表照 ch.8 按约束选策略、生死照 ch.10 显式管理。收敛成三个结构动作（一个组合根、@Observable 转显式通道、协调器决策/Router 执行），外加两条刻意不动的承重例外（转录的自绘性能契约、唯一那条同源相位同步的结构选择边）。做到这些，CCTerm 就从「一团藏在 @Observable 与 .shared 背后、谁也不敢动的隐式全局状态」，变成一张能读懂、能替换、能测试、能长期演进的对象图。

思考题：收尾自查时发现——ChatSessionViewController 切走后仍偶发崩溃，日志显示一个后台线程在对已离场的转录表调 apply；且有人为「快速验证转录只渲染可见行」加了个 chatVC.forceVisibleRowsForTest() 暴露内部索引。请结合本章 10.1/10.2/10.3 分别指出根因与修法，并说明它们如何共同违反了「把隐式关系变显式」这条本质。

查看答案

内存/拆卸角度（崩溃根因，10.1）：后台任务背着已切走的会话继续对离场表 apply，说明 per-attach 资源（回填任务、isRunning 观察）没在离场时释放，只等 ARC。crossfade 让离场 VC 多活约 300ms，期间这些订阅仍在跑。
→ 修法：把回填任务取消、onIsRunningChange 解绑、per-attach 滚动视图/sheet presenter 释放全部放进 ChatSessionViewController.prepareForRemoval()，由 DetailContainerViewController.swap 在移除前显式调用（ch.4.5/8.8），不寄望 deinit。

并发角度（10.2）：「后台线程调 apply」本身还违反「UI 只在主线程」——即便拆卸修好，回填产出页也必须切回主线程再喂表。
→ 修法：回填 Service 在后台产页，但 apply 必须在主线程调用（@MainActor 上，或 await 回主线程后）。注意转录的 apply 是同步精确 delta 通道，主线程内同步喂表，不经 .receive(on:) 慢一拍（ch.1.4）——「回主线程」与「同步喂表」不矛盾：切回主线程后仍是同一 source 相位内的同步调用。

可测试性角度（10.3）：forceVisibleRowsForTest() 是测试后门——它把测试专用路径焊进产品，且测的是内部索引而非真实行为。
→ 修法：删掉它。要验证「只渲染可见行」，驱动公开面：给转录 controller 喂一批 block、设定可见矩形，断言 dataSource 被查询的行区间 / mock cell 工厂被要求构造的行数（可观察输出），不碰任何 private 字段。
共同本质：三处都是「隐式关系没被显式化」。拆卸隐式（靠 ARC 猜时机）→ 崩溃；线程隐式（后台碰 UI）→ 偶发渲染错；依赖/控制点隐式（靠后门够到内部）→ 测的不是真实路径。把它们分别显式成 prepareForRemoval、@MainActor + 主线程 apply、公开面驱动的注入测试，三个症状同时消失——这正是本章、也是全书的收束。
