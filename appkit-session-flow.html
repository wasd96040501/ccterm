会话域：操作 / 数据流向 · 挂载生命周期

按纯 AppKit 规范（数据向下 · 事件向上 · 真相源唯一）画的理想架构。共三张图：① 层次与流向 → ② 发送闭环 → ③ 挂载生命周期。

蓝实线 ▼ 数据向下（data-down）：真相源用 @Published 把"该显示什么"投影下去，下层 sink 订阅后刷新。
橙虚线 ▲ 事件向上（events-up）：用户操作用 target-action / delegate 只上报意图，自己不改真相。
真相源 某状态唯一权威的存放处。别处看到的（侧栏、归档、transcript）都是它的投影，不是第二份拷贝。

O(1)：渲染缓存在 TranscriptStore（见图 4），不重解析历史。

图 4点击 sidebar → transcript 上屏：两阶段加载（首次 / 非首次统一）

最容易乱的一段。核心设计选择：把渲染缓存从"胖 Session"搬到每会话的 TranscriptStore —— 于是首次和非首次不再是两套代码，而是同一条路径，差别只落在一个"文档是否已加载"的水位标志上。

🧱 新引入的对象 · TranscriptStore（Store · 每会话 · 视图无关）
它持有这个会话的 消息时间线 + 翻译好的 blocks + 行高缓存 layoutCache —— 就是让"再次进入 O(1)"的那份缓存。它不挂在 VM 上（VM 短命），也不挂在 runtime 上（多数被点开的会话根本没在运行）。两个写入者喂它：① 历史回填（磁盘 JSONL，一次性）；② runtime 的 live 事件（在跑时）。VM 只订阅它。进程级有界缓存（LRU）持有它、跨挂载存活。
（这修正了图 3 的简化说法"缓存挂在 runtime R 上"——正确归属是独立的 TranscriptStore。）
同步 相 1 · 源相 —— 就在点击那一个 runloop tick 里跑完
1
选中 → 路由同步换详情 VC
selectionObserver 在点击的源相里同步触发（不走 @Observable 的延后重算），避免会话切换被撕成两帧。
2
造 / 复用每屏 ChatSessionViewModel，绑定该会话的 TranscriptStore
从进程级缓存取 TranscriptStore；没有就建一个空的（首次）。
3
建 per-attach 的 scroll / table 壳（尚未绑 dataSource）
每次挂载都新建、离场即拆——每-attach 资源，不跨挂载复用。
4
settle：layoutSubtreeIfNeeded 把表按真实宽度定尺寸
此刻还没有行可查。这就是"一次源相 = 一个 id 一个宽度"的性能契约：宽度必须在 tile 之前定死。
5
bind dataSource → scrollToTail：渲染 TranscriptStore 此刻拥有的 blocks
读 layoutCache，命中即 O(1)。这一步"渲染现有内容"对首次和非首次是同一行代码——差别只在 store 里此刻有多少东西。
非首次（暖）：store 已满 → 整个 transcript 这一 tick 立刻上屏。
首次（冷）：store 空 → 这一 tick 先空着（若在跑则只有 live tail）。真正的历史在相 2 流进来。
TICK 边界 · beforeWaiting flush —— 布局 / 绘制 / CoreAnimation 提交，画面上屏
异步 相 2 · 延迟相 —— 只有"有缺口"时才干活
6
若 store.historyLoaded == false → 起 backfill 管道
后台线程off-main：反向分页读 JSONL → markdown 解析 → 翻译成 Block → CTLine 排版（全部不占主线程）。
主线程预算 drain：尾页 append + scrollToTail，老页 prepend + 保持可视位置 —— tail-first 流式补齐；每帧只 drain 预算内的页，不卡滚动。
完成 → 置 store.historyLoaded = true。
7
若 store.historyLoaded == true → 相 2 直接 no-op
非首次天然走到这里：文档已满，什么都不用做。这就是"非首次"的全部含义——不是另一套路径，是同一条路径的退化情形。

首次 vs 非首次：同一条路径，只有一个分叉

🧊 首次（冷）O(n)
  1. 相 1 步骤 1–5 照跑；第 5 步渲染时 store 空 → 屏上先空 / 只有 live tail。
  2. 相 2 触发回填：磁盘 → 解析 → 翻译 → 排版，全在后台线程 + 预算化主线程 drain。
  3. 内容 tail-first 流式上屏 → 置 loaded。
O(n) 但不卡：重活都在 off-main，主线程只做有预算的 drain。
🔥 非首次（暖）O(1)
  1. 相 1 步骤 1–5 照跑（同一行代码）；第 5 步渲染时 store 已满 → 整屏立刻上屏。
  2. 相 2 见 loaded == true → 立即返回，零磁盘 / 零解析 / 零排版。
  3. 唯一主线程成本：重建 table 壳 + 一次 tile（读缓存命中）。
不是"跳过了加载"，是"相 2 发现没缺口"。
✅ 为什么这样又干净又快
干净：首次/非首次不再是 if firstTime 分支，而是一个"文档水位"——同一条 attach 路径，相 2 幂等。缓存住在 TranscriptStore（每会话、视图无关），所以 VM 能保持短命、re-visit 仍 O(1)。
快：源相里只做"壳 + 定宽 + 绑定 + 一次 tile（读缓存）"——全是 O(1) / 缓存命中的活；一切磁盘、解析、排版都推到后台线程 + 预算化 drain，永不占点击那一 tick。
一句话读懂三张图：真相只有两处 —— SessionStore（有哪些会话）和 SessionRuntime（一个活会话的实时状态）； 其余全是订阅出来的投影。用户操作一律 ▲ 向上报到这两处，改完再 ▼ 向下刷回来。 每屏的 VM/VC 是短命的（随挂载生死），而 runtime 是长命的（住在 RunningSessionsStore、跨挂载存活）—— 这就是"一个 VM 对应一份 model"和"CLI 不能断"能同时成立的原因。没有 SessionManager、没有大 map、没有跨三层的共享 Session、没有 Util。
