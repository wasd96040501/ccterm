心智模型与核心术语

在写第一行代码之前，先在脑子里装一张地图。AppKit 的世界由“一个应用 + 若干窗口 + 一棵视图树”构成，被一个永不停歇的“事件循环”驱动着转。这一章不写复杂代码，只帮你把这些名词从陌生符号变成你能“看见”的东西——之后所有章节都是在这张地图上添砖加瓦。

1. AppKit 是什么

AppKit 是苹果为 macOS 提供的原生图形界面框架：你屏幕上 Mac 程序里的窗口、菜单栏、按钮、文本框、表格、滚动条……几乎都是它画出来、管起来的。你可以用 Swift 或 Objective-C 来调用它。

它有个“老灵魂”。你很快会注意到 AppKit 里的类名几乎都以 NS 开头：NSWindow、NSView、NSButton……这个 NS 不是随便取的，它是 NeXTSTEP 的缩写——乔布斯 1985 年离开苹果后创立的 NeXT 公司所做的操作系统。1996 年底苹果收购 NeXT，把 NeXTSTEP 的技术变成了后来 macOS 的基础，那套界面框架也一路沿用下来，类名前缀就这样保留成了历史的“指纹”。所以当你看到 NS，可以读作“这是从 NeXTSTEP 一脉相承下来的老牌、稳定的 API”。

AppKitmacOS 的原生 UI 框架，负责窗口、控件、事件、绘制与应用生命周期。用 Swift / Objective-C 调用。
NS 前缀类名前的 NS 源自 NeXTSTEP（NeXT 公司的操作系统，后被苹果收购并成为 macOS 的技术源头），是历史命名习惯。
框架（Framework）苹果打包好的一整套现成代码库；你不必从零造轮子，import 进来就能直接用它定义的类。

它和 SwiftUI 是什么关系

SwiftUI 是苹果 2019 年推出的更新、更“声明式”的界面框架。打个比方：如果说 AppKit 是让你“亲手把每块砖摆到位”（命令式：先创建按钮，再设置标题，再加到窗口里），那么 SwiftUI 更像是“你描述墙长什么样，系统去帮你砌”（声明式：你写 Button("点我")，框架决定怎么落地）。

命令式（imperative）你一步步下达指令：创建谁、改谁的属性、把谁加到哪。怎么变、何时变都由你亲手安排。AppKit 是这种风格。
声明式（declarative）你只描述“在某个数据状态下界面应该长什么样”，由框架负责把界面同步成那个样子。SwiftUI 是这种风格。

关键事实：在 macOS 上，SwiftUI 的很多控件底层其实就是由 AppKit 支撑的——你写一个 SwiftUI 的侧边栏列表，系统内部很可能用的就是 AppKit 的表格/大纲视图。两者也能互相嵌套（用 NSHostingView 把 SwiftUI 塞进 AppKit，用 NSViewRepresentable 把 AppKit 塞进 SwiftUI）。

本教程的选择

本教程聚焦 AppKit。原因有二：一是它能让你看清 UI 框架“底下到底发生了什么”（这套心智模型对理解 SwiftUI 也极有帮助）；二是在需要精细控制窗口、性能、事件时机的场景里，AppKit 往往是不可替代的。学会 AppKit，你就握住了 macOS 界面的“地基”。

2. 对象模型的三大基石

无论一个 AppKit 程序多复杂，它的骨架永远是这三个对象，从大到小、从外到内层层包裹，最终连成一棵 对象树：

NSApplication应用的“总管”，整个进程里只有一个（单例），通过全局变量 NSApp 或 NSApplication.shared 拿到。它负责取系统事件并分发、管理所有窗口、驱动主事件循环。
NSWindow屏幕上的一扇窗口。它有标题栏区域和一块“内容区域”；事件先送到窗口，再由窗口找到合适的视图来处理。
NSView视图——窗口里负责“画一块矩形区域 + 响应这块区域里事件”的基本单元，是 AppKit 里最常见的对象。按钮、文本框都是 NSView 的子类。

它们怎么连成一棵树？打个比方，像俄罗斯套娃：

  • NSApplication（唯一的总管）→ 管理着若干 NSWindow。
  • 每个 NSWindow 有一个 内容视图（content view），它是这扇窗里视图树的顶端。
  • 内容视图里再放更多 NSView。官方的说法是：每个视图都有一个 父视图（superview），并可以拥有任意多个 子视图（subview），它们连接成一个 视图层级（view hierarchy）。

本质上，“界面长什么样”就等于“这棵对象树长什么样”。你之后写的每一行界面代码，做的事都可以归结为：往这棵树上加节点、改节点、或删节点。

3. AppKit 是个 MVC 框架

AppKit 的设计遵循 MVC（Model-View-Controller）模式。逐词拆开看：

M — Model（模型）你的“数据与业务逻辑”。比如一份待办清单的内容、一个文档的文字。它不关心界面长什么样。
V — View（视图）“看得见、点得到”的界面部分，负责显示数据、接收用户操作。对应 NSView 及其子类。
C — Controller（控制器）夹在 M 和 V 中间的“协调员”：把模型的数据摆到视图上，把视图上的操作翻译成对模型的修改。AppKit 里常用 NSViewController 充当它。

对照一下：SwiftUI 是声明式的，你描述“在某个数据状态下界面应该是什么样”，由框架负责保持同步；AppKit 是命令式 + MVC 的，当数据变了，你要明确地“亲手去更新对应的视图”。本教程会用 NSViewController 来组织界面，正是扮演那个 C 的角色。

4. 主线程与 RunLoop 的直觉

这是初学者最容易忽略、却最关键的一条心智模型。两句话先记住：

第一，UI 必须在主线程上操作。所有创建窗口、改控件、刷新界面的代码，都只能在 主线程（main thread） 跑。别的线程去碰 UI，行为是未定义的（轻则不刷新，重则崩溃）。

第二，你的程序不是“从头跑到尾就结束”，而是被一个事件循环驱动着等待 → 响应 → 再等待。这个循环就是 RunLoop（运行循环）；应用启动时由 NSApplication 调用 run() 把它转起来，这条专门处理用户事件的循环称为 主事件循环（main event loop）。

一次完整的“点一下”在内部是这样走的：

  1. 你点了一下鼠标 → 窗口服务器（window server，macOS 负责合成画面与采集输入的系统进程） 产生一个底层事件。
  2. 事件被放进应用的一个 先进先出的事件队列（event queue） 排队。
  3. NSApplication 从队头取出事件，把它表示成一个 NSEvent 对象。
  4. 它把这个事件 分发给事件发生的那扇 NSWindow。
  5. 窗口再找到“最该处理它的那个视图”（鼠标事件→指针下方的视图；键盘事件→第一响应者），回调你写的代码。
  6. 你的代码处理完、控制权交还，应用就去取下一个事件——如此往复，直到应用退出。
主线程（main thread）程序里专门跑 UI 与事件处理的那条执行线索。一切界面操作必须在它上面进行。
RunLoop / 主事件循环主线程上一个“取事件 → 分发 → 等下一个”的循环。由 NSApplication.run() 启动并持续运行。
第一响应者（first responder）当前“握着键盘焦点”的那个视图。键盘事件优先送给它（比如正在输入的文本框）。
NSEvent一次用户输入（点击、按键、滚轮等）被表示成的事件对象。
事件队列（event queue）底层事件排队等待处理的先进先出缓冲区，保证按到达顺序逐个处理。
为什么“卡住主线程”会让界面冻住

因为界面刷新和事件处理都靠主线程的这个循环。如果你在主线程上做了很耗时的事（比如同步下载一个大文件），循环就转不动了——取不到新事件、画不了新帧，窗口就“转菊花/无响应”。这也是后面所有“耗时操作放到后台线程、改 UI 再切回主线程”规矩的根源。

5. 三大协作模式预告

“事件回调你的代码”具体怎么回调？AppKit 反复用到三种约定俗成的模式。这里只点名，后续章节各有专章细讲：

target-action（目标-动作）控件被触发时，向一个指定的“目标对象（target）”发送一条“动作消息（action）”。比如：按钮被点 → 调用 self 的某个方法。最常用于按钮、菜单。
delegate（委托）一个对象把“某些决策/通知”委托给另一个对象处理。比如窗口即将关闭时，先问它的委托“该不该关”。是“问一问、报一报”式的一对一协作。
dataSource（数据源）表格/列表这类视图不自己存数据，而是反过来向“数据源”对象索要：一共几行？第 3 行该显示什么？把“数据”和“显示”解耦。

记住一句话就够了：target-action 是“我被点了，我来通知你”；delegate 是“做决定前，我先问你”；dataSource 是“要显示了，我来问你要数据”。

6. 用代码看一眼这棵树

下面是一段最小、现代、纯程序化（不依赖 Storyboard/xib）的代码，把本章的概念落地：一个 NSViewController 充当控制器（C），在它的内容视图（V）里用 Auto Layout 摆一个标题和一个按钮，按钮用 target-action 绑定点击。读的时候，对照前面那棵“对象树”和那条“事件循环”。

import AppKit

// 控制器（MVC 里的 C）：负责把界面搭起来，并接住用户操作。
final class GreetingViewController: NSViewController {

    // 不连 Storyboard 时，必须自己提供 view，它是这一层视图树的根（content view）。
    override func loadView() {
        view = NSView(frame: NSRect(x: 0, y: 0, width: 320, height: 160))
    }

    override func viewDidLoad() {
        super.viewDidLoad()

        // 一个标签：NSTextField 在“只读 + 无边框”的便捷初始化器下当标签用。
        let titleLabel = NSTextField(labelWithString: "你好，AppKit")
        titleLabel.font = .systemFont(ofSize: 20, weight: .semibold)

        // 一个按钮：用 target-action 绑定点击事件。
        // target 指向 self（本控制器），action 指向下面的 sayHello 方法。
        // 默认就是标准的圆角推送按钮，无需再设 bezelStyle。
        let button = NSButton(title: "打个招呼", target: self, action: #selector(sayHello))
        button.keyEquivalent = "\r"   // 让回车键也能触发，它便成为“默认蓝”按钮

        // 用 NSStackView 做纵向布局，省去逐条手写约束。
        let stack = NSStackView(views: [titleLabel, button])
        stack.orientation = .vertical
        stack.spacing = 16
        // 关键：用 Auto Layout 时必须关掉自动转换的旧式约束，否则约束会打架。
        stack.translatesAutoresizingMaskIntoConstraints = false

        view.addSubview(stack)   // 把 stack 挂到视图树上：view 是 superview，stack 是 subview

        // 用 anchors 把 stack 居中固定在内容视图里。
        NSLayoutConstraint.activate([
            stack.centerXAnchor.constraint(equalTo: view.centerXAnchor),
            stack.centerYAnchor.constraint(equalTo: view.centerYAnchor),
        ])
    }

    // target-action 的“action”：按钮被点击时，事件循环最终回调到这里。
    // 现代 AppKit 里仍需 @objc，因为 action 通过 Objective-C 的消息机制派发。
    @objc private func sayHello() {
        // 这一句运行在主线程上 —— 因为事件循环本就跑在主线程。
        print("按钮被点击了！事件已从 NSEvent 一路分发到这个方法。")
    }
}

把它对回地图：NSViewController 是 C；它的 view 是这一层视图树的根（V）；titleLabel 和 button 是挂在树上的子视图；点击按钮时，NSApplication 取到 NSEvent → 分发给窗口 → 命中按钮 → 通过 target-action 回调 sayHello()。整条链路你现在都能叫出名字了。

本质

一个 AppKit 程序 = 一棵对象树（NSApplication → 若干 NSWindow → 每扇窗的 NSView 视图层级）+ 一个事件循环（主线程上 NSApplication.run() 不停地：取 NSEvent → 分发给窗口 → 命中视图 → 回调你的代码）。你后续学的一切——加控件、做布局、响应点击、加载数据——本质都是“往这棵树上动手脚”和“在循环回调你时填上你的逻辑”这两件事。把这张地图记牢，后面就不会迷路。

思考题：为什么“在后台线程下载完文件后，直接在那个后台线程里更新窗口上的文字”是错误的做法？请用本章的两条心智模型来解释。

查看答案

因为本章第 4 节的第一条规则：所有 UI 操作必须在主线程进行。窗口、视图（NSView/NSTextField）都不是线程安全的，后台线程去改它们属于未定义行为，可能不刷新、显示错乱，甚至崩溃。

更深一层（第二条心智模型）：界面的刷新与事件处理都依赖主线程上的那个事件循环。视图状态的改动需要被这个循环纳入下一帧的绘制，而后台线程根本不在这个循环里。正确做法是：耗时下载放后台线程，拿到结果后把“更新文字”这一步切回主线程执行（例如 DispatchQueue.main.async { ... }），让它回到事件循环的轨道上。

应用骨架：从入口到一个窗口

每一个 macOS 应用，剥到最里面，都是同一副骨架：一个"应用本体"在后台不停地转圈听消息，前台摆着一块或几块"玻璃壳"——也就是窗口。本章我们从程序的第一行代码出发，一步步搭出这副骨架：搞清楚谁是应用、谁在听消息、窗口又是怎么被造出来并显示到屏幕上的。读完你会明白，所谓"启动应用"和"打开窗口"是两件完全不同的事。

2.1 应用只有一个：NSApplication

先打个比方。把一个 macOS 应用想象成一家餐厅：餐厅本身（招牌、营业执照、那个一直在前台接待的经理）只有一个；而餐厅里的桌子可以有很多张，客人来了摆一张，走了收一张。在 AppKit 里，这家"餐厅"就是 NSApplication，而每一张"桌子"就是一个窗口 NSWindow。

整个应用进程里，NSApplication 的实例有且只有一个（术语叫"单例 / singleton"）。你不会自己去 new 它，而是通过一个全局入口拿到这个唯一的实例：

单例（Singleton）一个类在整个程序里只允许存在一个实例，所有人都通过同一个全局入口访问它。NSApplication.shared 第一次被访问时创建那个唯一实例，以后每次返回的都是同一个；在代码里也常写作 NSApp（同一个对象的简写）。
事件循环（Event Loop / Run Loop）应用启动后进入的一个"永不停歇的圈"：不断地向系统要事件（鼠标点击、键盘敲击、窗口缩放……），拿到后分发给对应的窗口和视图处理，处理完再回头继续要下一个。它是应用"活着"的标志。

这个唯一实例最核心的工作，就是跑事件循环。启动它的方法是 run()：调用之后程序就钻进了那个"永不停歇的圈"里，不停地接收事件、用 sendEvent(_:) 把事件派发给该处理它的窗口/视图，直到你调用 terminate(_:) 让它退出。

下面这段代码展示的是"程序化路线"最赤裸的心智模型——假设你新建了一个名叫 main.swift 的文件（这是 Swift 里唯一允许写顶层可执行语句的文件名），手动把这几步串起来：

import AppKit

// NSApplication.shared 拿到的永远是同一个应用单例
let app = NSApplication.shared

// 给应用装一个"代理"（下一节讲），它会在关键时刻被回调
app.delegate = AppDelegate()

// 让应用以普通带 Dock 图标的方式激活
app.setActivationPolicy(.regular)

// 进入事件循环。这一行不会"返回"——程序从此活在循环里，
// 直到用户退出。可以把它理解为"餐厅开始营业，经理开始接待"。
app.run()

注意 run() 这一行"不会返回"——它不是普通函数调用完就往下走，而是一头扎进循环里。写在 run() 后面的代码，要等应用退出后才会执行。这正是"应用一直活着"的代码体现。

2.2 入口怎么写：@main 与现代/传统两条路

程序总得有个"第一行代码从哪开始执行"的起点，这就是入口。在 AppKit 的世界里，入口有几种写法，你选一条就好——但要注意：@main 标记和上面那种顶层语句二选一，不能同时用（编译器只认一个 main）。

@mainSwift 的属性标记，贴在某个类型上，告诉编译器"程序从这里开始跑"。它替你生成了真正的 main 入口函数。一个程序里只能有一个 @main。
@NSApplicationDelegateAdaptorSwiftUI 提供的桥，写在 SwiftUI 的 App 结构体里，把一个传统的 AppKit NSApplicationDelegate 接进来。等于"用 SwiftUI 的外壳，包住一个 AppKit 的内核"。
NSApplicationMain传统 AppKit 入口：贴 @main 在你的 AppDelegate 上，由 AppKit 自动创建应用单例、装好代理、调用 run() 一条龙（早期是手写 main.swift 调用 NSApplicationMain(...) 函数，现在用 @main 标记代替）。

三条路本质做的是同一件事：创建应用单例 → 装上代理 → 调 run() 进入事件循环。区别只在"谁来替你写这几步"：

  • 现代 SwiftUI 包 AppKit：用 @main struct CCTermApp: App（这里的 App 是 SwiftUI 的协议），配合 @NSApplicationDelegateAdaptor(AppDelegate.self) 把 AppKit 代理接进来。本项目（CCTerm）走的就是这条——SwiftUI 当外壳，真正的窗口和界面交给 AppKit。
  • 传统纯 AppKit：在 AppDelegate 上贴 @main，由系统自动完成上面那一套。
  • 完全程序化：自己写一个 main.swift，手动 NSApplication.shared → app.delegate = ... → app.run()，就是 2.1 那段代码。

本教程的所有示例走"程序化路线"：所有窗口、视图都用代码创建，不用 Storyboard、不用 xib（这两者是 Xcode 里用鼠标拖拽界面、再存成文件的可视化方案）。不走那条路的理由很简单——代码可读、可 diff、可控。界面长什么样、布局怎么写，全在 Swift 文件里看得见、改得动、能进版本管理对比，不会藏在一个拖出来的、人眼难读的文件里。（Storyboard / xib 当然存在，也有它的舞台，本教程只是不走那条路。）

2.3 谁在关键时刻被叫到：NSApplicationDelegate

应用单例自己只管"跑循环"，但它跑到一些关键节点时（启动完成了、快退出了……）需要通知"业主"来做点事。这个"业主"就是代理（delegate）——一个遵守 NSApplicationDelegate 协议的对象，被赋值给 app.delegate。

代理 / 委托（Delegate）一种回调模式：A 在特定时机"回头喊一声"B，把决定权或后续动作交给 B。这样不用改 A 的源码、也不用去继承 A，就能定制行为。NSApplication 通过代理把生命周期事件交给你处理，你不需要去继承 NSApplication。
生命周期（Lifecycle）应用从启动、激活、隐藏到退出的一系列阶段。每个阶段都有对应的代理方法被回调。
协议（Protocol）一份"方法清单/契约"。某类型声明"遵守"某协议，就承诺实现清单上的方法。NSApplicationDelegate 列出的就是各生命周期回调，你按需实现其中几个即可。

最重要的一个回调是 applicationDidFinishLaunching(_:)：它在应用启动流程全部完成之后被调用一次。这是"造主窗口、把它显示出来"的标准时机。

import AppKit

final class AppDelegate: NSObject, NSApplicationDelegate {

    // 持有窗口控制器，避免它被提前释放（下一节解释为什么用控制器）
    var mainWindowController: MainWindowController?

    // 启动流程跑完后，系统回调这里。我们在这里造窗口、显示窗口。
    func applicationDidFinishLaunching(_ notification: Notification) {
        let controller = MainWindowController()
        self.mainWindowController = controller   // 先攥住，再显示
        controller.showWindow(nil)               // 加载窗口并推到最前、变成 key 窗口
    }

    // 关掉最后一个窗口时，是否顺带退出整个应用。
    // 返回 true：窗口全关了 = 应用退出（很多文档型/工具型 app 的习惯）
    func applicationShouldTerminateAfterLastWindowClosed(_ app: NSApplication) -> Bool {
        true
    }

    // 应用即将退出前的最后一站，适合做收尾（保存状态等）
    func applicationWillTerminate(_ notification: Notification) {
        // 收尾工作写这里
    }
}
注意

上面把 mainWindowController 存成了 AppDelegate 的属性。这一步不是凑数——如果你只在 applicationDidFinishLaunching 里造一个局部变量，方法一返回它就没人引用、被回收，窗口也跟着消失了。让代理"攥住"控制器，窗口才活得下来。

2.4 程序化造一个窗口：NSWindow

现在来造那张"桌子"。窗口由 NSWindow 表示，它的指定初始化器是 init(contentRect:styleMask:backing:defer:)。四个参数里，styleMask 最值得一讲。

styleMask（样式掩码）一个"选项集合"（OptionSet），用 [ ... ] 把若干开关组合起来，决定窗口长什么样、有哪些能力。常见开关：.titled 有标题栏、.closable 有关闭按钮、.resizable 可缩放、.miniaturizable 可最小化、.borderless 无边框（不能和 .titled 一起用）。
contentViewController窗口的内容视图控制器。给它赋值，窗口就用这个控制器的 view 当自己的内容区——这就是"以控制器为单位组织界面"的接口。

左上角那三颗"红黄绿"交通灯按钮，其实就是 styleMask 里几个开关的直接产物：.closable 点亮红灯（关闭）、.miniaturizable 点亮黄灯（最小化）、.resizable 点亮绿灯（缩放/全屏）。没开对应的开关，那颗灯就是灰的、点不动。

import AppKit

func makeMainWindow(content: NSViewController) -> NSWindow {
    let window = NSWindow(
        contentRect: NSRect(x: 0, y: 0, width: 720, height: 460),
        // 标题栏 + 关闭 + 缩放 + 最小化，四个开关组合成一扇标准窗口
        styleMask: [.titled, .closable, .resizable, .miniaturizable],
        backing: .buffered,   // 离屏缓冲后再合成上屏，标准选择
        defer: false
    )
    window.title = "CCTerm"
    window.center()                       // 居中到屏幕
    // 把"界面是什么"交给一个视图控制器，而不是直接往窗口塞视图
    window.contentViewController = content
    return window
}

窗口造好后，光"存在"还不够，它默认是不在屏幕上的。要让它显示并接收键盘事件，调用 makeKeyAndOrderFront(_:)：

// 把窗口推到同层级的最前面，并使它成为 key 窗口（接收键盘输入的那一个）
window.makeKeyAndOrderFront(nil)
本质

"造窗口"和"显示窗口"是两步：NSWindow(...) 只是在内存里捏出一个对象；makeKeyAndOrderFront 才把它真正请上屏幕、并交给它键盘焦点。这也解释了为什么应用可以早就启动着、却一扇窗口都不显示——应用的"活"靠事件循环，窗口的"显"靠这一声召唤。

2.5 为什么用控制器：NSWindowController 与 NSViewController

你完全可以裸用 NSWindow，但 AppKit 鼓励再包两层"控制器"。它们各管一摊：

NSWindowController管理一扇窗口的生命周期：持有窗口、用 showWindow(_:) 显示、用 close() 关闭，常兼任窗口的 delegate（接收 windowWillClose 等回调）。它是窗口级逻辑的安放处，也可复用。
NSViewController管理一棵视图树（界面内容）：它有一个 view 属性作为根视图，负责把控件搭起来、配置数据与交互。现代 AppKit 提倡"以控制器为单位"组织界面——一块界面 = 一个 view controller。

分工很清晰：WindowController 管"壳"（窗口的生老病死），ViewController 管"瓤"（窗口里装的界面）。本项目 CCTerm 的主窗口正是这套结构：MainWindowController 管窗口本身，里面挂着负责具体内容的视图控制器。

提示

你可能在文档里看到 "NSWindowController 会懒加载窗口"——那说的是它从 Storyboard/nib 加载窗口的情况（第一次 showWindow(_:) 时才读文件造窗口）。走纯代码路线时我们更常在控制器的 init 里就 super.init(window:) 把窗口造好（CCTerm 的 MainWindowController 就是这样），窗口是立即就位的——showWindow(_:) 只负责把它显示出来。两种都对，知道区别即可。

对 NSViewController，初学者最容易混的是两个方法的分工：

loadView()负责创建根视图。第一次访问 view 属性、而它还没被加载时，系统自动调用一次。纯代码搭界面时，在这里 new 出根视图并赋给 self.view。你不要主动调用它。
viewDidLoad()在 loadView() 完成、view 已就绪之后被调用一次。负责配置：加子视图、设约束、绑数据、装监听。记得先调 super.viewDidLoad()。

一句话记法：loadView() 造根视图，viewDidLoad() 配内容。执行顺序是 访问 view → loadView() → view 就绪 → viewDidLoad()。下面是一个纯代码视图控制器的样板，里面也演示了本教程偏好的布局方式——Auto Layout 的 anchors 配合 NSStackView：

import AppKit

final class ContentViewController: NSViewController {

    // —— 造根视图：纯代码路线必须自己 new 出 view 并赋值 ——
    override func loadView() {
        // 给一个初始尺寸；约束会接管后续布局
        let root = NSView(frame: NSRect(x: 0, y: 0, width: 720, height: 460))
        self.view = root        // 关键：必须赋给 self.view
    }

    // —— 配内容：搭控件、设约束 ——
    override func viewDidLoad() {
        super.viewDidLoad()

        // labelWithString: 是现代 view-based 工厂方法，造一个只读文本标签
        let title = NSTextField(labelWithString: "你好，AppKit")

        // 用 target/action 把按钮点击连到自身的方法上
        let closeButton = NSButton(
            title: "关闭窗口", target: self, action: #selector(closeWindow))

        // 用纵向 StackView 把控件竖着排好，间距交给 StackView 管
        let stack = NSStackView(views: [title, closeButton])
        stack.orientation = .vertical
        stack.spacing = 16
        // 用 Auto Layout 时，这一行必须关掉旧的 autoresizing 转换
        stack.translatesAutoresizingMaskIntoConstraints = false

        view.addSubview(stack)

        // anchors：把 stack 钉在父视图正中央
        NSLayoutConstraint.activate([
            stack.centerXAnchor.constraint(equalTo: view.centerXAnchor),
            stack.centerYAnchor.constraint(equalTo: view.centerYAnchor),
        ])
    }

    @objc private func closeWindow() {
        view.window?.close()    // 通过视图找到所在窗口并关闭
    }
}
提示

用 Auto Layout 时，凡是自己 new 出来、要用约束摆放的视图，都得把 translatesAutoresizingMaskIntoConstraints 设成 false——否则系统会偷偷给它加一套基于 frame 的旧约束，和你的 anchors 打架。如果某个场景你想完全手动算坐标（不用 Auto Layout），那就改成 override func layout()，在里面直接写每个子视图的 frame，两条路线别混用。至于已废弃的 cell-based 老 API（比如直接摆弄 NSCell），本教程一律不碰。

2.6 把骨架连起来看一遍

现在从入口到屏幕，整条链路是这样的：

// 入口（程序化路线的心智模型）
let app = NSApplication.shared   // ① 拿到唯一的应用单例

// applicationDidFinishLaunching:
//   ④ MainWindowController()                造控制器（管壳）
//   ⑤ 窗口的 contentViewController 挂 ContentViewController（管瓤）
//   ⑥ showWindow(nil) → makeKeyAndOrderFront 窗口"显"上屏幕

记住这条链：单例活着（事件循环）→ 启动完成回调代理 → 代理造窗口控制器 → 控制器显示窗口 → 窗口里装着视图控制器的界面。下面的 demo 就是把这条链做成了能点的动画。

这个 demo 一一对应上面的 Swift 代码：点 启动应用 run() = app.run() 进入事件循环（进度条开始转圈，代表"应用活着"），但此刻屏幕上一扇窗口都没有——这就是"应用 = 单例 + 事件循环"。点 显示窗口 = makeKeyAndOrderFront(nil)，窗口这才被造出来并请上屏；窗口里那颗"关闭窗口"按钮对应 window.close()，点它窗口消失、但应用照样活着（循环还在转）。点窗口左上角的红色交通灯（=.closable）同样能关窗。最后 退出 terminate() 才让循环停下、应用谢幕。把"应用活着"和"窗口显示"分成两步，正是本章的本质。

思考题：在 applicationDidFinishLaunching 里，如果你写成 let c = MainWindowController(); c.showWindow(nil)，把控制器只放在一个局部变量里，没有让 AppDelegate 持有它。运行后窗口可能一闪就没了、或者干脆没出现。为什么？怎么修？

查看答案

因为局部变量 c 是这个方法内唯一引用控制器的地方。方法一返回，c 离开作用域、引用计数归零，控制器（连同它管理的窗口）被 ARC 回收，窗口自然消失。应用单例本身靠事件循环活着，不会因此退出，但"壳"没了。

修法：让一个生命周期更长的对象持有这个控制器。最常见的就是把它存成 AppDelegate 的属性（self.mainWindowController = c）——只要代理活着，窗口控制器就活着。这也正是 2.3 节样板里"先攥住、再 showWindow"的原因，呼应了"WindowController 负责窗口生命周期"这件事：但前提是它自己得先有人攥着。

视图与坐标系：NSView 的世界

上一章我们让窗口出现在屏幕上。但窗口本身只是一个空房间——真正住进去的家具、墙上的画、能按的开关，全都是「视图」。这一章我们把房间拆开看：屏幕上每一块能看见、能点的东西，本质上都是一个矩形，叫 NSView。理解它的两套尺寸（frame / bounds）和那个让无数人栽跟头的坐标原点，你就拿到了 AppKit 的「空间观」。

3.1 一切都是矩形：NSView 与视图树

先建立一个最朴素的直觉：你在 macOS 应用里看到的任何东西，都是一个矩形区域。一个按钮是矩形，一段文字是矩形，整块侧边栏也是矩形。这些矩形就是 NSView。

这些矩形不是平铺散落的，而是像「俄罗斯套娃」一样层层嵌套：窗口里有一个最大的根视图（即上一章提到的 contentView），根视图里放着侧边栏视图和内容视图，内容视图里又放着按钮、文字……每个视图都知道自己的「父亲」是谁（superview），也知道自己装了哪些「孩子」（subviews）。这就构成了一棵视图树。

NSViewAppKit 中屏幕上一切可见 / 可交互内容的基本单元。本质是一个矩形区域，负责在这块区域内绘制内容、响应鼠标键盘事件。
superview（父视图）包含当前视图的那个视图。一个视图最多只有一个 superview。
subviews（子视图）当前视图所包含的视图数组。子视图被「画」在父视图内部。
视图树（view hierarchy）由 superview / subviews 关系串起来的一棵树。窗口持有一个根视图（contentView），其余视图都挂在它下面。

把一个视图挂到另一个视图下面，用的是 addSubview(_:)：

let parent = NSView()                 // 一个空白矩形当容器
let blue = NSView()
parent.addSubview(blue)               // blue 成为 parent 的子视图
// 此时 blue.superview === parent，且 parent.subviews 里包含 blue

这里有一个常被忽略的细节：subviews 数组的顺序就是绘制（遮挡）顺序。越靠后加入的子视图，画得越「晚」，也就越「靠上」、越容易盖住前面的兄弟视图——就像往桌上一张张叠纸，后放的盖住先放的。如果两个子视图位置重叠，后 addSubview 的那个会显示在上面。

3.2 两套尺寸：frame 与 bounds

每个 NSView 都同时拥有两个矩形：frame 和 bounds。初学者最容易把它们搞混，但只要抓住一句话就再也不会错：

frame 是「别人眼里的我」，bounds 是「我眼里的我」。

打个比方。你住在一栋楼里：

  • frame：相当于「你家在这栋楼里的门牌号 + 房间大小」。它用父视图的坐标系来描述「我在哪、我多大」。比如「我在父视图里位于 (50, 100)，宽 200、高 150」。父视图想挪动你、放大你，改的就是你的 frame。
  • bounds：相当于「站在你自己家门口，从你自己内部往外量」。它用视图自身的坐标系来描述这块区域，origin 默认是 (0, 0)。你在自己内部画东西、摆放子视图时，参照的就是 bounds。
frame视图在其 superview 坐标系中的位置和尺寸（一个 NSRect，含 origin 与 size）。决定「这个视图被摆在父视图的哪个地方、占多大」。
bounds视图在自身坐标系中的可绘制区域，origin 默认为 (0, 0)。视图内部绘制、摆放子视图时以它为参照。
NSRect描述一个矩形的结构体：包含 origin（x, y 起点）和 size（width, height）。frame 和 bounds 都是 NSRect。（它是 CGRect 的别名。）
// 创建一个视图，告诉它在父坐标系里的位置和大小
let view = NSView(frame: NSRect(x: 50, y: 100, width: 200, height: 150))

print(view.frame)   // (50.0, 100.0, 200.0, 150.0) —— 相对于父视图
print(view.bounds)  // (0.0,  0.0,   200.0, 150.0) —— 相对于自己，origin 归零

注意两者的 size 一样（都是 200×150，先不考虑旋转 / 缩放），但 origin 不同：frame 的 origin 是「我在父视图的哪个角落」，bounds 的 origin 默认是 (0, 0)「我自己内部从头开始算」。

3.3 那个让所有人栽跟头的坑：坐标原点在左下角

现在到了这一章最关键、也最反直觉的地方。请记住：

AppKit 默认的坐标原点在视图的左下角，Y 轴朝上。

这意味着：当你写 frame.origin = (0, 0)，视图会贴在父视图的左下角；增大 Y 值，视图会往上跑。这和你熟悉的几乎所有东西都相反：

环境
原点位置
Y 增大方向
AppKit（默认）
左下角
向上 ↑
iOS / UIKit
左上角
向下 ↓
网页 / CSS
左上角
向下 ↓

为什么 AppKit 这么特立独行？因为它的血统来自数学课本里的笛卡尔坐标系（原点在左下、Y 朝上），这套体系在排版和 PostScript 打印时代很自然。但对今天习惯了网页和手机的我们来说，它就是个坑。

好消息是：你可以翻转它。NSView 有一个只读属性 isFlipped，默认返回 false。在子类里把它重写为返回 true，这个视图的坐标系（也就是它内部摆放子视图、以及 draw(_:) 里作画时所用的坐标系）就变成「原点在左上角、Y 轴朝下」——和 iOS、网页一致：

/// 一个翻转坐标系的视图：原点在左上角、Y 向下
final class FlippedView: NSView {
    // isFlipped 是只读属性，不能直接赋值，只能通过重写来改变
    override var isFlipped: Bool { true }
}
注意：isFlipped 是「类的属性」，不是开关

isFlipped 只读，意味着你不能在运行时把某个实例从 false 改成 true。它的取值是「这个视图子类天生是否翻转」。所以本章 demo 里那个「翻转开关」，在真实代码里对应的不是「改一个 Bool」，而是「换用另一个子类的视图」（或在重写里读一个标志位、并在切换后重建 / 重新布局）。心里要清楚：翻转影响的是子视图如何被定位，不只是「重画一下」。

实务提示

在真实项目里，给自定义容器视图重写 isFlipped { true } 非常常见——尤其是要「从上往下」摆放内容（列表、表单）时，左上原点的心智模型更顺手。本教程后面的很多布局也默认这么做。

3.4 视图怎么把自己画出来：needsDisplay 与 draw(_:)

视图不只是占位的矩形，它得在自己那块区域里画出内容（背景色、边框、图形）。AppKit 的绘制模型可以用一句话概括：

你不主动「画」，你只负责「喊脏」；系统在合适的时机回头叫你画。

具体流程是：当视图的内容需要更新时，你把 needsDisplay 设为 true（相当于举手喊「我这块脏了，要重画」）。你不会立刻被重画——系统会把这些请求攒起来，在当前 runloop 的合适时机统一回调你重写的 draw(_ dirtyRect:) 方法。dirtyRect 参数告诉你「这次实际需要重画的区域」，性能敏感时可以只画这一块。

needsDisplayBool 属性。设为 true 表示「这个视图需要重画」，向系统登记一次重绘请求，并不会立即重画。
draw(_ dirtyRect:)你在子类里重写、由系统在重绘时机回调的方法。在里面用 Core Graphics / NSBezierPath 自己绘制内容。绝不要直接调用它，要重画就设 needsDisplay = true。
NSBezierPathAppKit 里描述路径（直线、矩形、椭圆、贝塞尔曲线）的类，配合 NSColor 的 setFill / setStroke 完成填充和描边。
/// 自绘一块画布：填背景 + 画一个白色椭圆边框
final class CanvasView: NSView {
    override var isFlipped: Bool { true }   // 用左上原点，心智更顺

    override func draw(_ dirtyRect: NSRect) {
        super.draw(dirtyRect)

        // 1) 用 bounds（自身坐标系）填整块背景
        NSColor.systemBlue.setFill()
        bounds.fill()                       // NSRect.fill() 填满这个矩形

        // 2) 在内部缩一圈，画一个白色椭圆边框
        let path = NSBezierPath(ovalIn: bounds.insetBy(dx: 20, dy: 20))
        path.lineWidth = 2
        NSColor.white.setStroke()
        path.stroke()
    }
}

要让它重画，外部只需 canvas.needsDisplay = true，绝不要去手动调 canvas.draw(_:)。

3.5 不想自己画？交给图层：wantsLayer 与 CALayer

自绘很灵活，但有些常见效果（圆角、阴影、平滑动画）用 draw(_:) 手画又累又卡。这时候请出「图层支持视图」：把 wantsLayer 设为 true，AppKit 就给这个视图配一块 CALayer——一块由 GPU 缓存、专门负责合成与动画的「画布底板」。

直觉上：draw(_:) 是你拿画笔在纸上一笔笔画；而 CALayer 像是一张可以贴在玻璃上、能整体平移、缩放、加圆角阴影的「贴纸」，移动它不需要重画内容，所以又快又顺。圆角 / 阴影 / 动画这类需求，一律走 layer：

wantsLayerBool 属性。设为 true 表示「请给这个视图配一块 CALayer 做底板」，此后视图内容被缓存成 GPU 纹理。
CALayerCore Animation 的图层对象。承载视图的合成内容，圆角（cornerRadius）、阴影（shadow*）、变换、动画都作用在它身上。访问时用可选链 view.layer?。
let card = NSView()
card.wantsLayer = true                               // 开启图层支持
card.layer?.backgroundColor = NSColor.white.cgColor  // 背景色走 layer（注意是 cgColor）
card.layer?.cornerRadius = 12                        // 圆角走 layer
card.layer?.shadowOpacity = 0.2                      // 阴影走 layer
card.layer?.shadowRadius = 8
注意

开了 wantsLayer 后，内容会被缓存成 GPU 纹理，所以 draw(_:) 被调用的次数变少——只在缓存真正失效时才重画。圆角、阴影这些属性写在 layer 上，而不是去 draw(_:) 里手画。还有一个易错点：layer 上的颜色是 CGColor，要用 .cgColor 转换，不能直接塞 NSColor。

3.6 把它们装进 NSViewController

按本教程的约定，界面不靠 Storyboard，全程序化、用 NSViewController 组织，布局用 Auto Layout 的 anchors / NSStackView。下面把上面的画布、一个翻转开关、两个滑块拼起来——这正是本章 demo 对应的 Swift 实现骨架。其中画布用约束（anchors）填充，而子视图刻意用 frame 定位，因为这一章要演示的正是 frame.origin：

final class CoordinateDemoController: NSViewController {

    private let canvas = CanvasView()      // 父画布（看它的 bounds）
    private let child  = NSView()          // 子视图（看它的 frame）

    private let flipSwitch = NSSwitch()
    private let xSlider = NSSlider(value: 30, minValue: 0, maxValue: 170,
                                   target: nil, action: nil)
    private let ySlider = NSSlider(value: 30, minValue: 0, maxValue: 170,
                                   target: nil, action: nil)

    override func loadView() {
        // 程序化界面：自己给控制器造一个根视图并指定初始尺寸
        view = NSView(frame: NSRect(x: 0, y: 0, width: 440, height: 360))
    }

    override func viewDidLoad() {
        super.viewDidLoad()

        // 子视图：用 layer 画出橙色圆角方块；位置由 frame.origin 决定
        child.wantsLayer = true
        child.layer?.backgroundColor = NSColor.systemOrange.cgColor
        child.layer?.cornerRadius = 6
        child.frame = NSRect(x: 30, y: 30, width: 90, height: 90)
        canvas.addSubview(child)            // child 挂到 canvas 下，成为子视图

        // 控件接 target-action
        flipSwitch.target = self; flipSwitch.action = #selector(toggleFlip)
        xSlider.target = self;   xSlider.action = #selector(moveX)
        ySlider.target = self;   ySlider.action = #selector(moveY)

        // 用 NSStackView 把「控制台」竖直堆起来
        let controls = NSStackView(views: [flipSwitch, xSlider, ySlider])
        controls.orientation = .vertical
        controls.alignment = .leading
        controls.spacing = 8

        // 画布 + 控制台横向并排
        let root = NSStackView(views: [canvas, controls])
        root.orientation = .horizontal
        root.spacing = 16
        root.translatesAutoresizingMaskIntoConstraints = false
        view.addSubview(root)

        NSLayoutConstraint.activate([
            root.leadingAnchor.constraint(equalTo: view.leadingAnchor, constant: 20),
            root.trailingAnchor.constraint(equalTo: view.trailingAnchor, constant: -20),
            root.topAnchor.constraint(equalTo: view.topAnchor, constant: 20),
            root.bottomAnchor.constraint(equalTo: view.bottomAnchor, constant: -20),
            canvas.widthAnchor.constraint(equalToConstant: 260),
        ])
    }

    // 翻转开关：注意 isFlipped 不能直接改，真实代码要换用翻转 / 不翻转的子类。
    // 这里只示意「翻转后，子视图 frame.origin 的含义随之改变，需要重新摆放」。
    @objc private func toggleFlip(_ sender: NSSwitch) {
        canvas.needsDisplay = true          // 触发画布重画
        layoutChild()                       // 按当前坐标系语义重新放 child
    }

    // 滑块改子视图的 frame.x / frame.y —— 直接挪动它在父坐标系里的位置
    @objc private func moveX(_ sender: NSSlider) {
        child.frame.origin.x = CGFloat(sender.doubleValue)
    }
    @objc private func moveY(_ sender: NSSlider) {
        child.frame.origin.y = CGFloat(sender.doubleValue)
    }

    private func layoutChild() {
        child.frame.origin = NSPoint(x: CGFloat(xSlider.doubleValue),
                                     y: CGFloat(ySlider.doubleValue))
    }
}

下面这个 demo 复刻了上面 Swift 代码的效果。蓝色大框是父画布（对应 canvas，你看到的是它的 bounds——内部坐标系）；橙色小方块是子视图（对应 child，它的位置由 frame.origin 决定）。拨动开关翻转坐标系，看原点标记（带十字的小圆）在左下 / 左上之间跳动，子视图也随之换算位置；拖两个滑块，直接改 frame.origin.x / frame.origin.y，盯着小方块移动。两个滑块的范围都是 0–170，正好是「父边长 − 子边长」，所以无论怎么拖，方块都不会跑出蓝框——和 Swift 代码里 maxValue: 170 完全对应。

开关对应 Swift 里的 override var isFlipped：关时原点在左下、Y 朝上（AppKit 默认），子方块从底部往上摆；开时原点跳到左上、Y 朝下（与 iOS / 网页一致），子方块从顶部往下摆。两个滑块就是代码里的 child.frame.origin.x / .y——你拖动它们时，改的正是「子视图在父坐标系里的位置」，而蓝框（父 bounds）始终不动。把 demo 的每个控件和上一节 Swift 代码里的同名属性一一对上，你就把「代码 ↔ 效果」连起来了。

本质

NSView 就是一棵嵌套矩形的树，每个节点同时握着两套坐标：frame（在父坐标系里「我在哪、多大」）和 bounds（在自身坐标系里「我内部从哪算起」）。AppKit 默认把原点钉在左下角、Y 朝上，这是与 iOS / 网页最大的认知差，重写 isFlipped 返回 true 能把它翻回左上（注意这是「类的属性」，不是运行时开关）。绘制是「声明式」的——你只设 needsDisplay 喊脏，系统择机回调 draw(_:) 让你用路径作画；而圆角、阴影、动画这类合成层面的效果，则交给 wantsLayer 背后的 CALayer。一切空间问题，归根结底都是「这个矩形，是在谁的坐标系里被谁度量」。

思考题：在默认（未翻转）的 AppKit 视图里，一个父视图高 300。你把子视图的 frame.origin.y 从 0 增大到 100，子视图在屏幕上是往上移还是往下移？如果改用一个重写了 isFlipped { true } 的父视图，同样的操作结果又是什么？

查看答案

默认（未翻转）：原点在左下、Y 朝上，frame.origin.y 表示「离父视图底边的距离」。从 0 增大到 100，子视图离底边更远，所以向上移动 100 点。

翻转后（isFlipped = true）：原点跳到左上、Y 朝下，frame.origin.y 变成「离父视图顶边的距离」。同样从 0 增到 100，子视图离顶边更远，于是向下移动 100 点。

关键点：同一行代码（改 frame.origin.y）的视觉效果，完全取决于该父视图的坐标系是否翻转。这正是新手最常踩的坑——位置算错、视图「跑反了」，十有八九是忘了确认父视图的 isFlipped 状态。

布局全景：四种摆放视图的方式

你已经会把一个 NSView 放进窗口里了。但"放进去"只是第一步——真正的问题是：当窗口被拉宽、拉窄、最大化时，里面的标题、输入框、按钮该怎么跟着动？这一章把 AppKit 里"摆放视图"的四条路线一次讲透，并诚实回答你最关心的问题：到底谁用得多、谁更好用？

先建立一个心智模型。任何一个视图最终都靠一个叫 frame 的东西决定"我在哪、我多大"。所谓"布局"，无非就是谁来填写每个视图的 frame。四条路线的区别，仅仅是"填写 frame 这件事，由谁、在什么时候、用什么规则来做"。

frame一个视图在它父视图坐标系里的位置和尺寸，本质是一个矩形 NSRect（含 x、y、宽、高）。所有布局方式最后都归结为给每个视图填一个 frame。
父视图 / 子视图（superview / subview）视图是一棵树。子视图的 frame 坐标是相对父视图来算的。父视图变大变小，子视图要不要跟着变、怎么变，正是布局要解决的事。
macOS 坐标系默认 NSView 的坐标系原点 (0,0) 在左下角，y 轴向上为正（这点和 iOS 的 UIKit 相反，UIKit 原点在左上、y 向下）。手动算坐标时务必记住这一点。（补充：很多自绘视图会重写 isFlipped 返回 true 把坐标翻成"左上原点、y 向下"，这样画文字、排行更顺手——但那是后话，本章用默认坐标系讲。）

路线一：手动 frame —— 自己当"排版工"

最朴素的办法：每个视图的位置我亲自算、亲自填。打个比方，这就像在一张方格纸上，拿尺子量好每个零件该钉在第几行第几列，然后一颗一颗钉上去。

关键不在于"我会写 view.frame = ..."，而在于写在哪里。新手常犯的错是把坐标写在 viewDidLoad 里——那只在加载时算一次，窗口一缩放就全乱了。正确的位置是重写 NSView 的 layout() 方法：系统保证每次需要重新布局时（包括尺寸变化）都会调用它，你把所有坐标计算集中写在这里，就能在任何尺寸下都重新算对。

override func layout()NSView 的布局回调。系统在视图需要重新摆放子视图时调用（首次显示、尺寸改变、你主动把 needsLayout 设为 true 之后）。重写它、在里面读自己的 bounds 再给子视图填 frame，是"手动布局"的标准姿势。注意：重写时务必先调 super.layout()。
bounds视图自己内部的坐标系矩形（默认原点为 0,0，尺寸等于自身大小）。在 layout() 里你读 bounds 来知道"我现在多大"，据此分配子视图位置。区别于 frame：frame 是"我在父视图里多大/在哪"，bounds 是"从我自己看出去多大"。

路线二：autoresizingMask —— 老式的"弹簧与支柱"

在 Auto Layout 出现之前，AppKit 就有一套轻量的自动伸缩规则，叫 autoresizingMask。直觉模型是"弹簧（spring）和支柱（strut）"：你给每个视图声明——它和父视图四条边之间的距离，是"钉死的支柱"（固定不变），还是"会拉伸的弹簧"（跟着父视图一起变）。父视图一缩放，子视图就按这套规则自动重排。

比如 view.autoresizingMask = [.width, .height] 表示宽和高都是弹簧——父视图变大，它就等比变大。它比手动 layout() 省事，但表达能力很弱：只能描述"我和父视图边距的关系"，无法描述"我和兄弟视图的关系"（例如"按钮永远在输入框右边 8 点"）。这正是它被 Auto Layout 取代的原因——今天你只会在很老的代码或极简场景里见到它，了解概念即可。

路线三：Auto Layout —— 描述"关系"，让系统去解

这是现代 AppKit 的主力，也是你日常该默认选的方式。它的思路根本性地变了：你不再亲手算坐标，而是描述视图之间的关系（约束），剩下的算术交给系统的"约束求解器"。

打个比方：手动 frame 是你告诉工人"把这块板钉在 (120, 40) 这个点"；Auto Layout 是你说"这块板要贴着上面那块、左右各留 16 的边"——具体钉哪个坐标，由系统现场算。窗口怎么变，关系不变，坐标自动重算。

约束（NSLayoutConstraint）一条关于位置/尺寸的"等式或不等式"，例如"A 的左边 = B 的右边 + 8"。一组约束凑齐后，系统解出每个视图唯一的 frame。
anchors API写约束的现代简洁写法。每个视图暴露 leadingAnchor、trailingAnchor、topAnchor、widthAnchor 等"锚点"，用 a.leadingAnchor.constraint(equalTo: b.trailingAnchor, constant: 8) 连起来，再 .isActive = true 激活（或用 NSLayoutConstraint.activate([...]) 批量激活）。比手动 init 一个 NSLayoutConstraint 可读得多。
translatesAutoresizingMaskIntoConstraints一个开关。默认 true，意思是"把我老式的 autoresizingMask 自动翻译成约束"——这会和你手写的约束打架。凡是用 Auto Layout 摆放的视图，第一件事就是把它设成 false，否则会报"约束冲突"。这是初学者最常踩的坑。
NSStackView一个"会自动排队"的容器视图。你把子视图 addArrangedSubview 进去，告诉它方向（横/竖）、间距、对齐，它就在内部自动生成一堆约束帮你排好。绝大多数表单、工具栏、列表行用它就够了，几乎不用手写约束。
内容拥抱 / 抗压优先级（hugging / compression resistance）当空间有富余或不足时，"谁该被拉伸、谁该被压扁"的优先级。拥抱优先级高 = 不爱变大（紧贴自己内容，像标签）；抗压优先级高 = 不爱被压小（宁可挤别人也保住自己内容，像必须完整显示的按钮）。直觉记法：拥抱=抗拉伸，抗压=抗压缩。把输入框的水平拥抱优先级设低，它就会主动"长大"去吃掉多余空间。

路线四：把 SwiftUI 嵌进来 —— NSHostingView（点到为止）

如果某块界面你想用 SwiftUI 写，可以用 NSHostingView(rootView:)（或视图控制器版 NSHostingController）把一段 SwiftUI 视图包成一个普通 NSView，再用上面任意一种方式摆进 AppKit 里。它对外就是个 NSView，SwiftUI 内部自己负责布局。本教程聚焦 AppKit，这里知道"有这扇门、能混用"即可，细节后续章节再谈。

那到底谁用得多、谁更好用？（诚实结论）

Apple 官方推荐、绝大多数 App 和初学者的标准答案是 Auto Layout + NSStackView：写得快、改得稳、窗口缩放自动适配，可读性好。这应当是你的默认选择。

但"更好用"不等于"永远最优"。在高性能、海量 cell、复杂自绘的专业 UI 里，Auto Layout 的约束求解会成为瓶颈，于是这类项目常常下沉到手动 layout()，用直接写 frame 换取性能和像素级控制。两者不是对错，是不同场景的不同取舍。

来自真实项目的证据（已核实）

偏手动 layout()： Telegram 的 macOS 客户端 overtake/TelegramSwift 自建了 UI 框架 TGUIKit，其基础 View 类重写了 layout()、setFrameSize(_:)、setFrameOrigin(_:)，并在初始化里把 autoresizesSubviews 设为 false——典型的"全程手动算 frame"，为的是聊天列表里海量 cell 的滚动性能。

偏 Auto Layout： 更常见的开源 macOS 应用走标准路线——大量项目（如 Ranchero-Software/NetNewsWire、CotEditor、IINA 等）的界面用 Interface Builder 的 xib/storyboard 搭建，背后即 Auto Layout，配合 NSStackView 手写少量约束。结论照旧：日常选 Auto Layout，性能敏感再手动。

并排对照：同一组控件，两种写法

需求：一个标题、一个输入框、一个按钮，横向排成一行，整体随容器宽度变化。先看 Auto Layout（NSStackView + anchors） 的写法。

import Cocoa

// 路线三：Auto Layout —— 描述关系，系统自动重排
final class AutoLayoutFormController: NSViewController {

    override func loadView() {
        // 这个根视图就是“容器”，宽度由外部（窗口 / 分割视图）决定
        view = NSView()
    }

    override func viewDidLoad() {
        super.viewDidLoad()

        let title = NSTextField(labelWithString: "名称")     // 只读标签
        let field = NSTextField(string: "")                  // 单行输入框
        let button = NSButton(title: "保存", target: self, action: #selector(save))

        // 一个横向栈：方向 .horizontal，子视图间距 8
        let stack = NSStackView(views: [title, field, button])
        stack.orientation = .horizontal
        stack.spacing = 8
        stack.alignment = .centerY                           // 垂直方向居中对齐

        // 关键：凡是用 Auto Layout 摆的视图，先关掉自动转换的旧 mask
        stack.translatesAutoresizingMaskIntoConstraints = false
        view.addSubview(stack)

        // 让输入框“爱长大”：水平拥抱优先级调低 -> 多余宽度被它吃掉；
        // 标题和按钮拥抱优先级拉满，紧贴自己内容、不被拉伸。
        field.setContentHuggingPriority(.defaultLow, for: .horizontal)
        title.setContentHuggingPriority(.required, for: .horizontal)
        button.setContentHuggingPriority(.required, for: .horizontal)

        // 用 anchors 把栈钉在容器内、左右留 16 的边距、垂直居中
        NSLayoutConstraint.activate([
            stack.leadingAnchor.constraint(equalTo: view.leadingAnchor, constant: 16),
            stack.trailingAnchor.constraint(equalTo: view.trailingAnchor, constant: -16),
            stack.centerYAnchor.constraint(equalTo: view.centerYAnchor),
        ])
        // 注意：从头到尾没有出现任何坐标数字。容器变宽变窄，
        // 系统自动重解约束、自动重排，输入框自动伸缩。
    }

    @objc private func save() {
        // 按钮点击：现代写法用 target:action:，不碰任何废弃的 cell API
        print("保存")
    }
}

import Cocoa

// 路线一：手动 frame —— 所有坐标集中写在 layout() 里
final class ManualLayoutFormController: NSViewController {

    private let title  = NSTextField(labelWithString: "名称")
    private let field  = NSTextField(string: "")
    private let button = NSButton(title: "保存", target: nil, action: nil)

    override func loadView() {
        let root = NSView()
        button.target = self
        button.action = #selector(save)
        root.addSubview(title)
        root.addSubview(field)
        root.addSubview(button)
        view = root
    }

    // 系统在“首次显示 / 尺寸改变 / needsLayout 之后”都会调用这里。
    // 把全部算术集中在此，任何宽度都能重新算对。
    override func layout() {
        super.layout()

        let inset: CGFloat = 16
        let gap: CGFloat = 8
        let h: CGFloat = 24
        let b = view.bounds                            // 读“我现在多大”
        // 垂直居中：因为是对称的 (height - h) / 2，无论 y 朝上还是朝下都成立
        let y = (b.height - h) / 2

        // 标题：宽度按内容算（fittingSize 给出“刚好装下内容”的尺寸）
        let titleW = title.fittingSize.width
        title.frame = NSRect(x: inset, y: y, width: titleW, height: h)

        // 按钮：同样按内容宽，钉在最右侧（默认坐标系 x 向右）
        let buttonW = button.fittingSize.width
        button.frame = NSRect(x: b.width - inset - buttonW, y: y,
                              width: buttonW, height: h)

        // 输入框：吃掉中间剩下的所有宽度 —— 这一步“吃满”全靠你减出来，
        // 没有自动伸缩，容器一变就得重算（而这正是 layout() 会重跑的原因）
        let fieldX = inset + titleW + gap
        let fieldW = (b.width - inset - buttonW - gap) - fieldX
        field.frame = NSRect(x: fieldX, y: y, width: max(0, fieldW), height: h)
    }

    @objc private func save() { print("保存") }
}

对照着读，差别一目了然：Auto Layout 版本里没有一个坐标数字，"输入框吃满剩余宽度"是靠拥抱优先级声明出来的；手动版本里每个 x、宽都得你减出来，"吃满"是 (b.width - inset - buttonW - gap) - fieldX 这一行算出来的。容器一变宽变窄——前者系统自动重解，后者靠 layout() 被重新调用、你重新算。

本质

布局只有一件事：给每个视图填一个 frame。四条路线的唯一区别是"谁来填、何时填、按什么规则填"。手动 layout() 是你亲手算坐标（最大控制力、最高性能、最累）；autoresizingMask 是父子边距的弹簧规则（很弱，基本退役）；Auto Layout 是你只描述视图间的关系、由求解器算出坐标（现代默认、最省心）；NSHostingView 则是把 SwiftUI 当成一块 NSView 嵌进来。记住那条诚实结论：日常默认 Auto Layout + NSStackView；只有当性能/海量 cell/复杂自绘逼你下沉时，才回到手动 layout()。

思考题：在手动 layout() 那段代码里，"输入框吃满剩余宽度"是用 fieldW = (b.width - inset - buttonW - gap) - fieldX 算出来的。如果改用 Auto Layout，你不写任何坐标，又是靠什么让输入框自动吃满？为什么标题和按钮不会跟着一起被拉伸？

查看答案

靠 内容拥抱优先级（content hugging priority）。在栈里，三个视图共享一行宽度，多余的空间会分给"最不爱拥抱自己内容"的那个。我们把输入框的水平拥抱优先级设成 .defaultLow（低，"我不介意被拉大"），把标题和按钮设成 .required（高，"我要紧贴内容、别拉我"）。于是求解器把所有富余宽度都塞给输入框，标题和按钮保持各自的内容宽度不变。

对照之下你会发现两种范式的根本差异：手动版把"吃满"算成了一个具体数字，容器一变就得重算；Auto Layout 版把"吃满"声明成了一条优先级规则，容器怎么变，规则都成立，系统自动重解——这正是它在日常开发里更省心的原因。

常用控件与 Target-Action

界面上能点、能拖、能勾、能选的那些小东西，叫"控件"（control）。但控件只负责"长成那个样子，并感知用户的手指"，它自己并不知道点完之后该干嘛——真正干活的代码住在另一个对象里。这一章我们先弄清楚控件怎么把"我被操作了"这件事告诉你的代码——这就是 AppKit 几十年没变的核心套路 Target-Action——然后挨个认识一遍最常用的控件，最后亲手搭一个"控件画廊"，让每一次操作都实时反映到屏幕上。

5.1 控件为什么需要"告诉别人"

想象一个真实的开关面板：墙上的开关只管"被按下 / 弹起"，它并不知道接的是客厅灯还是排风扇——那是电线另一头的事。控件也一样：一个 NSButton 只懂"我被点了"，至于点完要保存文件、要发消息、还是要弹窗，它一概不管。

那它怎么把"我被点了"传出去？AppKit 的答案不是回调函数、也不是事件监听器，而是一个非常"Cocoa 味"的机制：每个控件身上挂着两个属性——一个 target（接收消息的对象，相当于电线另一头接的电器）和一个 action（要调用对象身上的哪个方法，相当于"按一下要做的那个动作"）。控件被操作时，就向 target 发送 action 这条消息。

控件 (Control)界面上可交互的元素，AppKit 里几乎都是抽象基类 NSControl 的子类，如按钮、滑块、文本框。它负责显示自己并感知用户操作，但不负责"操作之后做什么"。
Target-Action一种把"谁触发"和"做什么"解耦的设计模式：控件存着一个 target 对象引用和一个 action 选择器，被操作时就向 target 发送 action 这条消息。这样控件无需被子类化，就能驱动任意逻辑。
target控件的属性，类型 AnyObject?，是动作消息的接收者。它是 weak（弱引用）的，避免控件和它的处理者互相强引用造成内存泄漏。若为 nil，消息会沿"响应者链"向上寻找谁能处理。
action控件的属性，类型 Selector?。Selector（选择器）是"一个方法名的运行时身份证"，用 #selector(方法名) 写出来。

5.2 #selector 与 @objc：为什么要这两个"咒语"

Target-Action 这套机制比 Swift 还老——它来自 Objective-C 的运行时（runtime）。在 Objective-C 里，"调用一个方法"叫"发送一条消息"，而消息靠名字（selector）在运行时动态查找。Swift 想用这套机制，就得满足两个条件：

  • @objc：给你的处理方法加上这个标记，等于告诉编译器"把这个方法暴露给 Objective-C 运行时，让它能被名字找到"。不加，运行时就看不见它，触发时会因为"找不到这个 selector"而崩溃。
  • #selector(...)：把方法名安全地转换成一个 Selector 值。比起手写字符串 "buttonClicked:"，#selector 会让编译器帮你检查这个方法真的存在，写错了当场报错。

action 方法有个固定长相：接收一个参数 sender，就是"是谁触发了我"。在方法里通常立刻把 sender 转回它的真实类型，去读它现在的值：

@objc func buttonClicked(_ sender: NSButton) {
    // sender 就是被点击的那个按钮本身
    print("按钮被点了，当前状态：\(sender.state == .on)")
}
提示

你可能在 iOS（UIKit）里见过 UIAction 这种"直接传一个闭包"的写法，没有 selector 也没有 @objc。AppKit 的 NSButton 等控件没有内置这种闭包式初始化器，连接控件和逻辑一律走 target-action。所以本章的 #selector / @objc 不是可选项，而是必修课。

5.3 两种"挂钩"写法

把控件和处理方法连起来，现代 AppKit 有两条等价的路，区别只是"分两步写还是一步写"：

// 写法一：先建控件，再分别赋值 target 和 action（最原始，能看清机制）
let button = NSButton(title: "保存", target: nil, action: nil)
button.target = self
button.action = #selector(save(_:))

// 写法二：用便利初始化器一步到位（推荐，最常见）
let button2 = NSButton(title: "保存", target: self, action: #selector(save(_:)))

本章代码主要用写法二，因为它最能体现 target-action 的本质："这个控件的动作 = 调用 self 的某个方法"。当一个控件先建出来、target/action 暂时填 nil（比如要先存进属性、稍后再统一接线），就退回写法一补上——两者效果完全一样。

提示

一组 NSButton(radioButtonWithTitle:...) 单选按钮，只要共用同一个 action 选择器，AppKit 就会自动把它们识别成互斥的一组——选中一个，其余自动取消。不需要你手动管理"分组"。

5.5 动手：用代码搭一个控件画廊

下面这个 NSViewController 程序化地放进若干控件，全部接到同一个 controlChanged(_:) 方法上。这正是 target-action 的威力：一个回调，靠 sender 是谁、靠 tag（给控件编号的小整数）来分辨是谁触发的，然后统一刷新底部那块"输出区"。布局沿用上一章学的 NSStackView + anchors。

import Cocoa

final class GalleryViewController: NSViewController {

    // 给每个控件一个 tag 编号，便于在回调里区分
    private enum Tag: Int { case button, check, toggle, slider, popup, segment, field }

    // 底部的输出标签（用不可编辑的 NSTextField 当"标签"）
    private let output = NSTextField(labelWithString: "等待操作…")

    // 持有各控件的引用，方便随时读它们的当前值；target/action 先留空，viewDidLoad 里统一接线
    private let slider  = NSSlider(value: 50, minValue: 0, maxValue: 100,
                                   target: nil, action: nil)
    private let popup   = NSPopUpButton(frame: .zero, pullsDown: false)
    private let segment = NSSegmentedControl(labels: ["列表", "网格", "分栏"],
                                             trackingMode: .selectOne,
                                             target: nil, action: nil)
    private let field   = NSTextField(string: "")
    private let check   = NSButton(checkboxWithTitle: "接收通知",
                                   target: nil, action: nil)
    private let toggle  = NSSwitch()

    override func loadView() {
        // 控制器没连 xib，必须自己给一个根视图
        view = NSView(frame: NSRect(x: 0, y: 0, width: 360, height: 320))
    }

    override func viewDidLoad() {
        super.viewDidLoad()

        // 1) 普通按钮：便利初始化器一步连好 target-action（写法二）
        let button = NSButton(title: "执行",
                              target: self,
                              action: #selector(controlChanged(_:)))
        button.tag = Tag.button.rawValue

        // 2) 复选框（先建好、这里补 target/action，即写法一）
        check.tag    = Tag.check.rawValue
        check.target = self
        check.action = #selector(controlChanged(_:))

        // 3) 开关 NSSwitch
        toggle.tag    = Tag.toggle.rawValue
        toggle.target = self
        toggle.action = #selector(controlChanged(_:))

        // 4) 滑块（前面初始化时 target/action 留空，这里补上）
        slider.tag    = Tag.slider.rawValue
        slider.target = self
        slider.action = #selector(controlChanged(_:))

        // 5) 下拉菜单：加几个选项
        popup.addItems(withTitles: ["红", "绿", "蓝"])
        popup.selectItem(withTitle: "绿")
        popup.tag    = Tag.popup.rawValue
        popup.target = self
        popup.action = #selector(controlChanged(_:))

        // 6) 分段控件
        segment.selectedSegment = 0
        segment.tag    = Tag.segment.rawValue
        segment.target = self
        segment.action = #selector(controlChanged(_:))

        // 7) 文本框：回车提交时触发 action
        field.placeholderString = "输入后按回车…"
        field.tag    = Tag.field.rawValue
        field.target = self
        field.action = #selector(controlChanged(_:))

        // 输出标签换行显示
        output.lineBreakMode = .byWordWrapping
        output.maximumNumberOfLines = 0

        // 用竖直 stack 把所有控件码起来
        let stack = NSStackView(views: [button, check, toggle, slider,
                                        popup, segment, field, output])
        stack.orientation = .vertical
        stack.alignment = .leading
        stack.spacing = 12
        // 程序化布局必须关掉自动转换 mask，改用 anchors
        stack.translatesAutoresizingMaskIntoConstraints = false
        view.addSubview(stack)

        NSLayoutConstraint.activate([
            stack.topAnchor.constraint(equalTo: view.topAnchor, constant: 20),
            stack.leadingAnchor.constraint(equalTo: view.leadingAnchor, constant: 20),
            stack.trailingAnchor.constraint(equalTo: view.trailingAnchor, constant: -20),
        ])
    }

    // 唯一的回调：所有控件都连到这里，靠 sender / tag 区分
    @objc private func controlChanged(_ sender: NSControl) {
        let what: String
        switch Tag(rawValue: sender.tag) {
        case .button:  what = "点击了「执行」按钮"
        case .check:   what = "复选框：\((sender as! NSButton).state == .on ? "勾选" : "取消")"
        case .toggle:  what = "开关：\((sender as! NSSwitch).state == .on ? "开" : "关")"
        case .slider:  what = "滑块拖到 \(Int(sender.doubleValue))"
        case .popup:   what = "下拉选了 \((sender as! NSPopUpButton).titleOfSelectedItem ?? "")"
        case .segment: what = "分段选了第 \((sender as! NSSegmentedControl).selectedSegment) 段"
        case .field:   what = "文本框提交：\(sender.stringValue)"
        case .none:    what = "未知控件"
        }

        // 汇总当前所有控件的值，一并显示
        let summary = """
        最近操作：\(what)
        ──────────
        复选框：\(check.state == .on ? "✓" : "✗")  开关：\(toggle.state == .on ? "开" : "关")
        滑块：\(Int(slider.doubleValue))  下拉：\(popup.titleOfSelectedItem ?? "")
        分段：第 \(segment.selectedSegment) 段  文本：\(field.stringValue)
        """
        output.stringValue = summary
    }
}

读这段代码时，把目光锁定一条主线：操作发生 → 控件向 target（self）发送 action（controlChanged）→ 在方法里读 sender 的当前值 → 更新 output。七个控件、一个回调，这就是 Cocoa 几十年来组织交互的标准骨架。

每个控件的事件回调就对应 Swift 里的 controlChanged(_:)：操作发生 → 触发回调 → 读出当前值 → 重绘底部输出。注意"最近操作"那一行随着你动的不同控件而变，正如代码里靠 sender 的 tag 分辨触发者；而下方的汇总区始终显示所有控件的当前值，对应代码里那段 summary。文本框故意只在按下回车时才触发，对应 NSTextField 的 action 默认在回车时才发。

本质

Target-Action 把一次交互拆成三段、解耦成两个对象：控件只负责"感知操作并发出一条具名消息"，处理逻辑住在另一个对象（target）里，两者用一个方法名（action / selector）作为约定连接。这样做的回报是：控件无需被你子类化就能驱动任意逻辑；多个控件可以共用一个回调；连接关系还能在运行时改写。你后面会反复见到的菜单项（NSMenuItem）、工具栏按钮，背后都是这同一套"发消息给 target"的思路——认准它，AppKit 的交互层就只剩下"换皮"的区别。

思考题：代码里七个控件全部把 action 设成同一个 #selector(controlChanged(_:))，靠每个控件不同的 tag 在回调里分辨是谁。换个思路：如果给每个控件单独写一个 action 方法（buttonClicked、sliderChanged……），各有什么利弊？什么场景下你会倾向"一个回调 + tag"，什么场景下倾向"一控件一方法"？

查看答案

一个回调 + tag：优点是省代码、便于"统一处理"（比如这里要把所有控件的值汇总成一段文本，集中在一处最自然），新增控件只要加个 tag 分支。缺点是回调内部要做 switch 分发、还要把 sender 强转回具体类型（sender as! NSButton），类型不安全且容易写错；逻辑一多，这个方法会变成又长又杂的"大杂烩"。

一控件一方法：优点是每个方法职责单一、参数类型可以直接写成具体控件类型（_ sender: NSSlider），可读性和类型安全都更好。缺点是方法多、有重复的样板代码。

经验法则：当多个控件本质上在做同一类事、且需要共享 / 汇总状态时（如本章的画廊、或一组筛选条件联动刷新列表），用"一个回调"更顺手；当每个控件触发的是语义完全不同的独立动作时（保存、删除、分享……），用"一控件一方法"更清晰、也更容易让编译器替你检查参数类型。

列表与数据驱动：NSTableView / NSOutlineView

从邮件列表到访达侧边栏，从设置面板到聊天记录——你在 macOS 上看到的几乎每一份"上下滚动的列表"，背后都是同一个主角：NSTableView。它和树形版本 NSOutlineView 一起，撑起了系统里所有"行与列"的界面。本章我们要弄懂一件最反直觉、却最关键的事：表格自己并不存数据。它更像一个尽职的服务员，需要展示哪一行，就转头问你要——这就是"数据驱动"。理解了这一点，你才会明白为什么一个表格能轻松装下上万行还丝般顺滑，也才会明白本项目为什么用 NSTableView 自绘整段聊天记录。

1. 先建立直觉：表格是"服务员"，不是"仓库"

初学者最容易犯的错，是把表格想象成一个"装满了行的盒子"——好像你得把 100 行数据一行行塞进去，它才能显示。事实恰恰相反。

打个比方：餐厅里有一张能坐很多人的长桌（屏幕上可见的那几行），但厨房里的菜单（你的真实数据）可能有上千道菜。服务员（表格）不会把上千道菜一次端上来，而是你坐到第几桌、想看第几页菜单，他才去厨房取对应的那几样。表格也是如此：它只关心"现在屏幕上能看见的那十几行"，需要显示第 37 行时，它才回过头问你："第 37 行的内容是什么？"

这种"用到才问、按需索取"的工作方式，有个专门的名字。

Pull 模型（拉取模型）表格不主动持有全部数据，而是在需要绘制某一行时，临时向"数据源"提问、索取那一行的内容。与之相对的是 Push 模型（你把全部数据一次性推给控件，控件自己存着）。AppKit 的表格、大纲都采用 Pull 模型。
数据驱动（data-driven）界面上显示什么，完全由背后的数据决定。你改数据、然后告诉表格"刷新一下"，界面就跟着变；你从不直接去操作某一行的像素。

2. NSTableView：行与列的根基

NSTableView 是一个把数据按"行（记录）+ 列（字段）"排布的视图。一个表格可以只有一列（就是最常见的竖直列表），也可以有多列（像一张电子表格）。每一列由一个 NSTableColumn 描述。表格几乎总是被放进一个 NSScrollView 里，这样行多了就能滚动。

关键在于：表格把"要数据"和"怎么显示"这两件事，交给了两个不同的助手。这是本章最重要的分工。

dataSource（数据源）遵循 NSTableViewDataSource 协议的对象，负责回答"一共有多少行"（numberOfRows(in:)）以及提供每一行背后的原始数据。它管"有什么"。
delegate（委托）遵循 NSTableViewDelegate 协议的对象，负责"怎么把数据变成画面"——提供每行的视图（tableView(_:viewFor:row:)）、行高，以及处理用户的选择（tableViewSelectionDidChange(_:)）。它管"长什么样、怎么交互"。
记忆法

dataSource 回答 "有什么"（多少行、每行的数据）；delegate 回答 "怎么样"（每行长成什么视图、被选中了怎么办）。在小程序里，常常让同一个 NSViewController 同时兼任这两个角色——本章的示例就是这么做的。

3. view-based vs cell-based：现代写法只用 view-based

历史上 NSTableView 有两种模式。老的叫 cell-based（基于单元格），每个格子是一个轻量的 NSCell 对象——这套 API 已过时，新代码不要用。现代写法是 view-based（基于视图）：每个格子就是一个真正的 NSView，通常是系统提供的 NSTableCellView，里面预置了 textField（文本）和 imageView（图标）两个常用控件出口。view-based 能让你用 Auto Layout 自由排版每一行，能放按钮、进度条、任何视图。

NSTableCellViewview-based 表格里一个格子的标准容器视图，继承自 NSView。自带可选的 textField: NSTextField? 和 imageView: NSImageView? 两个出口，让你快速放文字和图标。
复用（reuse）/ 复用标识符（identifier）表格只为"屏幕上看得见的那几行"创建视图；当某行滚出屏幕，它的视图不会被销毁，而是被回收、擦干净、拿去装即将滚入的新行。makeView(withIdentifier:owner:) 就是"领一个回收好的视图来用"的方法，identifier 是给这一类视图起的名字（领回来的就是同名的那批）。
本质

复用机制是表格能"上万行也流畅"的根本原因。屏幕上同时可见的可能就 20 行，所以内存里永远只活着大约 20 多个行视图，它们被反复擦写、循环利用。无论你的数据是 100 行还是 100 万行，创建的视图数量几乎不变——成本只取决于"屏幕多大"，不取决于"数据多大"。本项目的聊天记录（transcript）正是基于这个理由直接用 NSTableView 自绘：SwiftUI 的 List / LazyVStack 在这种行数、自定义布局和滚动语义下扛不住，而 NSTableView 的复用 + 自绘恰好稳定高效。

4. 用代码搭一个 view-based 表格

下面是一个完全程序化（不依赖 Storyboard / xib）的最小可用表格。它用 NSViewController 组织界面，用一个 Swift 数组当"真实数据"。请重点看三处：数组就是数据源背后的仓库、三个协议方法如何被表格"提问"、以及添加数据后调用 reloadData() 触发重绘。

import AppKit

// 这个控制器同时兼任 dataSource 与 delegate 两个角色
final class FruitListViewController: NSViewController {

    // ① 真正的“数据”就住在这里——一个普通数组。表格本身不存它。
    private var items: [String] = ["苹果", "香蕉", "樱桃"]

    private let tableView = NSTableView()
    private let inputField = NSTextField()                       // 顶部输入框
    private let selectionLabel = NSTextField(labelWithString: "未选中")

    // 给这一类行视图起的“复用名字”
    private let cellID = NSUserInterfaceItemIdentifier("FruitCell")

    override func loadView() {
        // 程序化创建根视图（不走 xib）
        view = NSView(frame: NSRect(x: 0, y: 0, width: 360, height: 360))
    }

    override func viewDidLoad() {
        super.viewDidLoad()
        setupTable()
        setupLayout()
    }

    private func setupTable() {
        // ② 单列表格：建一列、把表格的两个角色都交给 self
        let column = NSTableColumn(identifier: NSUserInterfaceItemIdentifier("name"))
        column.title = "水果"
        tableView.addTableColumn(column)
        tableView.dataSource = self     // 数据源：管“有多少行 / 每行什么数据”
        tableView.delegate = self       // 委托：管“每行长什么视图 / 选中了怎么办”
        tableView.headerView = NSTableHeaderView()
    }

    private func setupLayout() {
        // 把表格放进可滚动容器（行多了能滚动）
        let scrollView = NSScrollView()
        scrollView.documentView = tableView
        scrollView.hasVerticalScroller = true

        // 顶部：输入框 + 添加按钮，横向排布（按钮用 target/action 绑回调）
        let addButton = NSButton(title: "添加", target: self, action: #selector(addItem))
        addButton.bezelStyle = .rounded
        inputField.placeholderString = "输入水果名，回车或点添加"
        inputField.target = self
        inputField.action = #selector(addItem)                  // 回车也触发添加

        let inputRow = NSStackView(views: [inputField, addButton])
        inputRow.orientation = .horizontal
        inputRow.spacing = 8

        // 用垂直 NSStackView 把“输入行 / 表格 / 选中标签”叠起来
        let stack = NSStackView(views: [inputRow, scrollView, selectionLabel])
        stack.orientation = .vertical
        stack.spacing = 12
        stack.translatesAutoresizingMaskIntoConstraints = false  // 用 anchors，必须关掉它

        view.addSubview(stack)
        NSLayoutConstraint.activate([
            stack.topAnchor.constraint(equalTo: view.topAnchor, constant: 16),
            stack.leadingAnchor.constraint(equalTo: view.leadingAnchor, constant: 16),
            stack.trailingAnchor.constraint(equalTo: view.trailingAnchor, constant: -16),
            stack.bottomAnchor.constraint(equalTo: view.bottomAnchor, constant: -16),
        ])
    }

    // ③ 添加：改数组 + 让表格重绘。这就是“数据驱动”的全部精髓。
    @objc private func addItem() {
        let text = inputField.stringValue.trimmingCharacters(in: .whitespaces)
        guard !text.isEmpty else { return }
        items.append(text)             // 先改“真实数据”
        tableView.reloadData()         // 再让表格回过头来重新提问 → 多出一行
        inputField.stringValue = ""
    }
}

// MARK: - 数据源：回答“有什么”
extension FruitListViewController: NSTableViewDataSource {
    // 表格问：“一共多少行？” → 答：数组有多长就多少行
    func numberOfRows(in tableView: NSTableView) -> Int {
        items.count
    }
}

// MARK: - 委托：回答“怎么样”
extension FruitListViewController: NSTableViewDelegate {

    // 表格问：“第 row 行该用什么视图？” → 领一个复用视图，填上数据返回
    func tableView(_ tableView: NSTableView,
                   viewFor tableColumn: NSTableColumn?,
                   row: Int) -> NSView? {
        // 先尝试领一个回收好的视图；领不到（首次）就新建一个
        let cell = tableView.makeView(withIdentifier: cellID, owner: self) as? NSTableCellView
            ?? makeFreshCell()
        cell.textField?.stringValue = items[row]   // 用第 row 行的数据擦写它
        return cell
    }

    // 表格通知：“选中行变了！” → 读 selectedRow，更新下方标签
    func tableViewSelectionDidChange(_ notification: Notification) {
        let row = tableView.selectedRow            // 没有选中时为 -1
        if row >= 0 {
            selectionLabel.stringValue = "已选中：\(items[row])（第 \(row + 1) 行）"
        } else {
            selectionLabel.stringValue = "未选中"
        }
    }

    // 第一次没有可复用视图时，新建一个并打上复用标识符
    private func makeFreshCell() -> NSTableCellView {
        let cell = NSTableCellView()
        cell.identifier = cellID                   // 打上名字，将来才能被回收复用

        let text = NSTextField(labelWithString: "")
        text.translatesAutoresizingMaskIntoConstraints = false
        cell.addSubview(text)
        cell.textField = text                      // 接到标准出口上
        NSLayoutConstraint.activate([
            text.leadingAnchor.constraint(equalTo: cell.leadingAnchor, constant: 4),
            text.centerYAnchor.constraint(equalTo: cell.centerYAnchor),
        ])
        return cell
    }
}

读完这段代码，请把它在脑子里"跑"一遍：表格要显示时，先问 numberOfRows(in:)——你回答"数组有几个就几行"；然后对每个可见行问 tableView(_:viewFor:row:)——你领一个复用视图、用 items[row] 擦写它的文本。用户点了某行，tableViewSelectionDidChange(_:) 被调用，你读 selectedRow 更新提示。新增数据时，你做的只有两步：改数组、调 reloadData()，剩下的"提问—回答"循环表格会自动重跑一遍。

5. NSOutlineView：把列表"长出树枝"

NSOutlineView 是 NSTableView 的子类，专门显示树形（层级）数据——访达的边栏、左侧的文件夹树、本应用的会话侧边栏都是它。它在普通表格的基础上多了"展开/折叠"的三角箭头（disclosure triangle）。

它依然是 Pull 模型，只是提问的方式从"第几行"变成了"某个节点的第几个孩子"。每个树节点对外表现为一个不透明的 item（类型是 Any，AppKit 不关心它具体是什么，只把它原样还给你）；整棵树的根用 nil 表示。大纲会一层层向数据源发问：

outlineView(_:numberOfChildrenOfItem:)某个节点（参数为 nil 表示根）有几个直接孩子？决定每一层要画几行。参数类型是 Any?，因为根用 nil 传入。
outlineView(_:child:ofItem:)某个节点的第 index 个孩子是哪个 item 对象？把抽象的"第几个孩子"映射到你的真实模型，返回类型是 Any。
outlineView(_:isItemExpandable:)这个节点能不能展开（要不要画三角箭头）？通常"有孩子就能展开"。注意这里参数是 Any（非可选）——只会针对真实节点发问，不会传 nil。
易踩的坑：参数到底是 Any 还是 Any?

这三个方法的 item 参数不全是可选类型：numberOfChildrenOfItem 和 child:ofItem: 收 Any?（要用 nil 代表根）；而 isItemExpandable 收的是非可选的 Any。如果你把它也写成 Any?，Swift 会认为你没有实现协议要求的方法（签名对不上），编译能过但运行时该回调根本不被调用——所有节点都画不出三角、点了没反应。照抄 Xcode 的自动补全即可避开。

关键直觉：未展开的分支永远不会被提问。哪怕一棵树有几万个节点，只要它们都收着，大纲就一个都不去碰——这正是 Pull 模型对又深又大的树极其高效的原因。下面是数据源核心方法的样子（视图提供方法 outlineView(_:viewFor:item:) 与表格的 viewFor:row: 同理，此处从略）：

// 一个最简单的树节点模型（用 class，因为 NSOutlineView 靠对象“身份”识别节点）
final class Node {
    let title: String
    let children: [Node]
    init(_ title: String, _ children: [Node] = []) {
        self.title = title
        self.children = children
    }
}

final class TreeViewController: NSViewController, NSOutlineViewDataSource {

    // 顶层节点（即“根 nil 的孩子”）
    private let roots = [
        Node("项目", [Node("源码", [Node("App.swift"), Node("View.swift")]),
                     Node("资源")]),
        Node("收藏夹", [Node("文档"), Node("下载")]),
    ]

    // 某节点（nil 即根）有几个孩子？
    func outlineView(_ outlineView: NSOutlineView,
                     numberOfChildrenOfItem item: Any?) -> Int {
        if let node = item as? Node { return node.children.count }
        return roots.count               // item 为 nil → 返回顶层数量
    }

    // 某节点的第 index 个孩子是谁？
    func outlineView(_ outlineView: NSOutlineView,
                     child index: Int, ofItem item: Any?) -> Any {
        if let node = item as? Node { return node.children[index] }
        return roots[index]
    }

    // 这个节点能展开吗？（有孩子就能）—— 注意 item 是 Any，不是 Any?
    func outlineView(_ outlineView: NSOutlineView,
                     isItemExpandable item: Any) -> Bool {
        guard let node = item as? Node else { return false }
        return !node.children.isEmpty
    }
}

顺带一提：树节点这里用 class 而不是 struct 是有讲究的——NSOutlineView 靠对象的"身份"（同一个引用 ===）来记住"哪个节点是展开的"。本项目的侧边栏 SidebarItemNode 正是 final class，原因就在这里。

思考题：假设一个表格要显示 100 万行数据，而屏幕一次只能看到约 25 行。运行过程中，内存里大概同时存在多少个"行视图"对象？为什么不是 100 万个？如果你在 tableView(_:viewFor:row:) 里领到复用视图后忘了重新赋值文本，滚动时会看到什么诡异现象？

查看答案

内存里大约只有 25~30 个行视图（可见行数再加上下边缘的少量缓冲），而不是 100 万个。原因是复用机制：滚出屏幕的行视图被回收进复用池，makeView(withIdentifier:owner:) 把它们交还给你后，你擦干净再拿去装即将滚入的新行。视图数量只取决于"屏幕多大"，与数据总量无关——这正是表格能扛住海量数据的根本。

如果你领到复用视图却忘了用 items[row] 重新赋值它的文本，那么这个视图会残留上一行的旧内容。滚动时你就会看到"文字串行 / 重影"——比如本该显示第 500 行的格子里，赫然写着之前第 12 行的水果名。这是 view-based 表格最经典的新手 bug：复用视图必须在每次 viewFor 里被完整地重新配置，绝不能假设它是干净的。

事件与响应链 + 委托模式

界面画好了，但用户点一下、敲个键，"消息"到底流向了谁？这一章我们顺着鼠标和键盘走一遍 AppKit 的"消息传递路线图"——你会发现它像办公室里"逐级上报"的流程：能办的人就地办了，办不了的就往上递。理解了这条链，你才能真正掌控：谁来处理点击、谁来接收键盘，以及当一个对象想说"这种情况你帮我拿主意"时该用什么方式开口。

事件 (event)用户对硬件的一次操作被系统打包成的一个对象，类型是 NSEvent：一次鼠标按下、一次键盘敲击、一次滚轮滚动，都是一个 NSEvent。它带着"发生了什么、在哪个位置、按了哪个键"等信息。
响应者 (NSResponder)能接收并处理事件的对象。AppKit 里几乎所有能见到的东西都是响应者：NSView、NSWindow、NSViewController、NSApplication 全都继承自 NSResponder。换句话说，"视图、窗口、视图控制器、应用本身"都能接事件。

7.1 一个直觉：事件像"逐级上报"

想象公司里一张报销单：先递给离你最近的直属同事。他能批就批了；批不了，就交给组长；组长搞不定，交给部门经理；再不行就到总经理。报销单一路向上传递，直到有人处理，或者没人能处理被丢弃。

AppKit 处理事件就是这个流程，这条"逐级上报的路线"有个正式名字。

响应链 (responder chain)一串首尾相连的响应者对象。事件从链条的起点开始，如果当前对象不处理，就沿着链条传给"下一个响应者"，直到有人处理或到达终点。这个"传给上级"的动作，就是我们俗称的事件"冒泡"。
下一个响应者 (nextResponder)每个响应者身上的一个属性 var nextResponder: NSResponder?，指向链条上它的"上级"。当自己处理不了时，就把消息转交给 nextResponder。这正是响应链能一节一节连起来的原因。

对于一个普通视图里的事件，典型的上报路线大致是这样的（从近到远）：

// 一次事件大致沿这条链向上"冒泡":
//
//   被点中的那个 NSView          ← 离用户最近,先处理
//        ↓ nextResponder
//   管理它的 NSViewController     ← 视图的 nextResponder 通常指向它的控制器
//        ↓
//   各级父视图 (superview)        ← 顺着视图层级继续往上
//        ↓
//   NSWindow                     ← 视图所在的窗口
//        ↓
//   NSWindow 的 delegate
//        ↓
//   NSApplication (NSApp)        ← 链条顶端:整个应用
//        ↓
//   NSApplication 的 delegate    ← 还没人理,事件就被丢弃
别死背顺序

响应链的确切形状会随视图层级、是否有控制器接管而变化（例如某个视图若由 NSViewController 管理，它的 nextResponder 会先指向那个控制器）。你只需要记住一个不变的方向感：从"离用户最近、最具体的对象"出发，一级级往"更大、更笼统的容器"上报。具体某一环的连接，用调试时打印 view.nextResponder 一看便知，不必背。

记住这条链的方向，后面所有"为什么我点了却没反应""键盘事件跑去哪了"的疑惑，都能在这张图上找到答案。

7.2 鼠标事件：override mouseDown(with:)

想让一个视图响应鼠标按下，你只要重写它从 NSResponder 继承来的方法。鼠标事件会被精准地送到鼠标位置正下方的那个视图——它就是这条链的起点。

import Cocoa

// 一个会响应点击的自定义视图
final class ClickableView: NSView {

    // 系统在用户按下鼠标左键时调用这里,event 携带了位置等信息
    override func mouseDown(with event: NSEvent) {
        // 把窗口坐标转换成"本视图内部坐标",方便判断点在哪
        let p = convert(event.locationInWindow, from: nil)
        print("我被点了,本地坐标 = \(p)")

        // 关键选择:
        // ① 自己处理掉 —— 那就到此为止,事件不再上报
        // ② 自己不处理 —— 调用 super,事件会沿 nextResponder 继续"冒泡"给上级
        super.mouseDown(with: event)   // 注释掉这一行,事件就被本视图"截胡"了
    }
}
关键点

"调用 super 还是不调用"，就是你决定让事件"继续上报"还是"就地截停"的开关。这与 7.5 节 demo 里的冒泡演示是同一件事。

7.3 键盘事件：谁是"第一响应者"？

鼠标有明确的"点在哪个视图上"，可键盘没有位置——你敲一个字母，系统怎么知道送给谁？答案是：送给当前的第一响应者。

第一响应者 (firstResponder)当前窗口里"拥有键盘焦点"的那个响应者。键盘事件总是先交给它处理；它处理不了才沿响应链向上冒泡。一个窗口在同一时刻只有一个第一响应者（比如你正在输入的那个文本框）。

一个自定义视图默认不接收键盘焦点。要让它能当第一响应者、能收到 keyDown，需要两步：

final class KeyCatcherView: NSView {

    // ① 声明"我愿意当第一响应者",默认是 false,必须重写为 true
    override var acceptsFirstResponder: Bool { true }

    // (可选) 真正成为第一响应者时被调用,返回 true 表示接受
    override func becomeFirstResponder() -> Bool {
        needsDisplay = true   // 比如:获得焦点时重绘出高亮边框
        return true
    }

    // ② 重写键盘按下;若它当前不是第一响应者,这里根本不会被调用
    override func keyDown(with event: NSEvent) {
        if event.charactersIgnoringModifiers == " " {
            print("空格被按下,我自己处理掉")
        } else {
            // 不是我关心的键 —— 交给 super,沿响应链继续冒泡
            super.keyDown(with: event)
        }
    }
}

// ③ 想让某个视图"现在就拿到键盘焦点",由它所在的窗口来指派:
//    view.window?.makeFirstResponder(theView)
//    makeFirstResponder 是 NSWindow 的方法,它负责切换焦点归属。
常见坑

只重写了 keyDown 却没把 acceptsFirstResponder 改成 true，视图永远拿不到焦点，keyDown 一次都不会触发——这是新手最常见的"为什么键盘没反应"。两件事必须配齐：愿意当焦点（acceptsFirstResponder 返回 true）+ 被指派为焦点（窗口调用 makeFirstResponder）。

7.4 三种"协作模式"+ 一种"广播"

响应链解决的是"系统的事件怎么找到对象"。但对象与对象之间还要互相协作——按钮要通知你"我被点了"，文本框要问你"用户这么输对不对"，表格要向你索取"该显示什么数据"。AppKit 把这些协作收口成几种固定套路，认全它们，你就读懂了大半 AppKit 代码。

target-action (目标-动作)控件 → 你。控件（如按钮）记着一个"目标对象 target"和一个"动作方法 action"，被触发时就向目标发送这个动作。一对一、即触发即调用。最简单直接，适合"点一下做一件事"。
delegate (委托)把"遇到这种情况该怎么办"的决定权，交给另一个对象去拍板。控件遇到某个时机就回头问它的 delegate："该怎么处理？"——比如 NSTextFieldDelegate 的 controlTextDidChange（文本变了）、NSTableViewDelegate（这一行长什么样）。一对一，可以带返回值（委托能回答"可以/不可以"）。
dataSource (数据源)委托的"近亲"，但专管数据：控件问它"一共多少行？第 3 行显示什么？"。delegate 管"外观与行为"，dataSource 管"内容与数量"，职责分开。
NotificationCenter (通知中心)一对多的"广播站"。一个对象 post 一条通知，所有事先订阅（addObserver）了这条通知的对象都会收到。发送方完全不知道、也不关心谁在听——这叫松耦合。

target-action 我们在按钮那章见过，这里聚焦最常用的 delegate。把它当成"回调问卷"：控件在关键时刻递给你一张问卷，你（delegate）填好答案交回去。

// 让控制器充当文本框的"委托",实时校验输入
final class FormViewController: NSViewController, NSTextFieldDelegate {

    private let field = NSTextField()
    private let tip = NSTextField(labelWithString: "")

    // 程序化界面:不依赖 Storyboard/xib,在 loadView 里自己搭
    override func loadView() {
        view = NSView()

        field.placeholderString = "只能输入数字"
        field.delegate = self          // ★ 把"该怎么办"的决定权交给 self

        // 用 NSStackView 做程序化布局(回顾第 4 章)
        let stack = NSStackView(views: [field, tip])
        stack.orientation = .vertical
        stack.alignment = .leading
        stack.spacing = 8
        stack.translatesAutoresizingMaskIntoConstraints = false
        view.addSubview(stack)

        NSLayoutConstraint.activate([
            stack.centerXAnchor.constraint(equalTo: view.centerXAnchor),
            stack.centerYAnchor.constraint(equalTo: view.centerYAnchor),
            field.widthAnchor.constraint(equalToConstant: 220),
        ])
    }

    // controlTextDidChange 来自 NSControlTextEditingDelegate
    // (NSTextFieldDelegate 继承了它),用户每敲一个字都会回调一次
    func controlTextDidChange(_ obj: Notification) {
        // 通知的 object 就是发生变化的那个控件
        guard let tf = obj.object as? NSTextField else { return }
        let text = tf.stringValue

        if text.isEmpty {
            tip.stringValue = "⚠️ 不能为空"
        } else if !text.allSatisfy(\.isNumber) {
            tip.stringValue = "⚠️ 只能是数字"
        } else {
            tip.stringValue = "✓ 输入合法"
        }
    }
}

注意 controlTextDidChange 的参数是一个 Notification——这其实是个伏笔：delegate 与 NotificationCenter 在底层常常是同一套机制的两种用法。它们的取舍直觉是：

delegate vs notification 怎么选

一对一、需要回答 / 返回值（比如"这个值合法吗？能不能结束编辑？"）→ 用 delegate，控件只认你一个，还能听你的裁决。
一对多、只是广播一个事实（比如"语言切换了""登录状态变了"，可能十个界面都要跟着刷新）→ 用 NotificationCenter，发送方一声吼，谁订阅谁响应，彼此互不相识。

本质

AppKit 的事件系统，本质是一条按"远近 / 上下级"排好序的对象链（响应链）：事件先给离用户最近、最具体的响应者（鼠标给点中的视图，键盘给第一响应者），处理不了就沿 nextResponder 向上冒泡，直到有人接手或被丢弃——"调不调用 super"就是你掌控冒泡的闸门。而 target-action / delegate / dataSource / NotificationCenter，本质都是对象之间"把决定权或事实交出去"的不同耦合度：从一对一即时调用（target-action），到一对一可问可答（delegate / dataSource），再到一对多互不相识的广播（notification）。先想清楚"谁该处理、是一对一还是一对多、要不要回答"，API 的选择自然就浮出来了。

思考题：你做了一个自定义视图，重写了 keyDown(with:) 想接收回车键，但运行时无论怎么按它都毫无反应，keyDown 断点一次也没停。可能漏了哪一步？如果"漏的那一步"补上了仍然没反应，还可能是什么原因？

查看答案

最常见的一步：没有重写 acceptsFirstResponder 返回 true。它默认是 false，视图根本不被允许成为第一响应者，自然拿不到键盘焦点，keyDown 永远不触发。

补上后仍没反应，继续排查：① 视图虽然"愿意"当焦点，但还没人"指派"它当焦点——需要在合适时机调用 view.window?.makeFirstResponder(theView) 把焦点交给它；② 键盘焦点此刻可能在别的控件（比如某个文本框）身上，那个第一响应者先把事件吃掉了；③ 视图还没被加到窗口里（window 为 nil），或窗口本身不是 key window，事件根本到不了它。把"愿意当焦点 + 被指派为焦点 + 处在 key 窗口的响应链上"三件事配齐，keyDown 才会如期触发。

综合实战：迷你 To-Do + 综合思考题

前七章我们把零件一个个拆开看了：App 怎么启动、窗口怎么来、视图控制器怎么管界面、约束怎么排布局、按钮怎么响应点击、表格怎么显示数据。这一章不再讲新零件，而是把它们拼成一台能跑的机器——一个真正可用的迷你 To-Do。拼装的过程，就是检验你是否真正理解每个零件"为什么长这样"的过程。读完代码、玩过 demo，再用四道综合思考题逼自己想到本质。

迷你 To-Do一个最小的待办清单应用：输入框里打字、回车或点按钮把它加进列表、点选某一行高亮、点"删除选中"去掉它、顶部实时显示"总数 / 已选"。它麻雀虽小，却串起了输入、布局、事件、数据源、选择五件事。
程序化（programmatic）不用 Storyboard / xib 这种"拖控件"的图形文件，而是纯用 Swift 代码创建并摆放所有界面元素。好处是界面的来龙去脉全在代码里，可被搜索、可被 review、可被 diff。本教程全程程序化。

1. 先在脑子里把数据流画出来

动手前，先想清楚"数据从哪来、到哪去"。一个 To-Do 的核心其实只有一个数组 items（待办文字的列表）。所有界面都只是这个数组的"镜子"：

  • 输入框负责往数组里追加一条。
  • 表格负责把数组显示出来（它问数据源："有几行？""第 N 行显示什么？"）。
  • 删除按钮负责从数组里移除选中那条。
  • 顶部计数 label负责读数组长度和"当前选中行"。

记住这条铁律：界面是数组的函数。每次数组变了，就让相关界面"重新照一次镜子"（表格 reloadData()、label 重新赋值）。把"改数据"和"刷界面"分开想，后面所有代码都会顺理成章。

提示

"先改数组、再刷界面"听起来啰嗦，但它是 AppKit 里最可靠的写法。千万别想着"我直接往表格里塞一行就好"——那样数组和界面会悄悄对不上，调试到崩溃。

2. 完整代码（分块讲解）

下面把整个迷你 To-Do 拆成五块。每块开头我都标注它对应前面哪一章的知识，你可以边读边回忆。

第 1 块 · 对应第 1、2 章（AppDelegate + 应用启动 + 窗口）。程序化项目里我们通常去掉 main storyboard，让 AppDelegate 在"启动完成"这个时机里亲手把窗口造出来。@main 标记程序入口，NSApp 是那个全局唯一的应用对象。

import Cocoa

@main
final class AppDelegate: NSObject, NSApplicationDelegate {
    // 窗口要被 AppDelegate 强引用住，否则启动完就被释放、窗口一闪而过

    func applicationDidFinishLaunching(_ notification: Notification) {
        // 第 2 章：手动创建一个有标题栏、可关闭、可最小化、可缩放的窗口

第 2 块 · 对应第 3 章（NSViewController 的生命周期）。我们重写 loadView() 给控制器一个根视图，再在 viewDidLoad() 里搭界面。因为没有用 Storyboard，所以必须自己提供 view。数组 items 就是上一节说的"唯一真相"。

final class TodoViewController: NSViewController,
                                NSTableViewDataSource,
                                NSTableViewDelegate,
                                NSTextFieldDelegate {

    // === 唯一真相：所有界面都是它的镜子 ===
    private var items: [String] = []

    // 持有几个待会要操作的控件

第 3 块 · 对应第 4 章（Auto Layout + NSStackView）+ 第 5 章（target-action）+ 第 6 章（NSTableView）。这是界面装配的核心。用一个垂直 NSStackView 把"计数行 → 输入行 → 表格 → 删除按钮"从上到下码好；输入行内部再用一个水平 stack 装"输入框 + 添加按钮"。所有手动创建的视图都要把 translatesAutoresizingMaskIntoConstraints 关掉，否则系统会偷偷塞一堆和你打架的旧式约束。

private func setupUI() {
    // --- 第 6 章：把表格放进可滚动容器，并接上数据源 / 代理 ---
    let column = NSTableColumn(identifier: NSUserInterfaceItemIdentifier("todo"))
    column.title = "待办"
    tableView.addTableColumn(column)
    tableView.headerView = nil            // 单列，藏掉表头更清爽
    tableView.dataSource = self           // “有几行 / 第 N 行什么内容”问我
    tableView.delegate = self             // “选中变化 / 给我一个单元视图”也问我
    tableView.style = .inset

    let scroll = NSScrollView()
    scroll.documentView = tableView       // 表格必须被 scrollView 包住才能滚动
    scroll.hasVerticalScroller = true
    scroll.translatesAutoresizingMaskIntoConstraints = false
    // 关键：让滚动容器在垂直方向“乐意被拉长”。垂直 stack 默认按内容紧凑排，
    // 不降这个优先级，表格就只会保持很矮的高度、撑不满中间区域。
    scroll.setContentHuggingPriority(.defaultLow, for: .vertical)

    // --- 第 5 章：输入框回车提交 + 添加按钮 target-action ---
    inputField.placeholderString = "输入待办，回车或点“添加”"
    inputField.delegate = self            // 监听回车（见第 5 块）
    inputField.translatesAutoresizingMaskIntoConstraints = false
    inputField.setContentHuggingPriority(.defaultLow, for: .horizontal) // 让它吃掉多余宽度

    let addButton = NSButton(title: "添加", target: self, action: #selector(addTapped))
    addButton.bezelStyle = .rounded

    let inputRow = NSStackView(views: [inputField, addButton])
    inputRow.orientation = .horizontal
    inputRow.spacing = 8

    // --- 第 5 章：删除按钮 ---
    deleteButton.title = "删除选中"
    deleteButton.bezelStyle = .rounded
    deleteButton.target = self
    deleteButton.action = #selector(deleteTapped)
    deleteButton.isEnabled = false        // 没选中时不可点

    // --- 计数 label ---
    countLabel.font = .systemFont(ofSize: 13, weight: .medium)

    // --- 第 4 章：用垂直 NSStackView 一把搞定主轴布局 ---
    let root = NSStackView(views: [countLabel, inputRow, scroll, deleteButton])
    root.orientation = .vertical
    root.spacing = 10
    root.alignment = .leading              // 子项左对齐
    root.translatesAutoresizingMaskIntoConstraints = false
    view.addSubview(root)

    // 用 anchors 把 root 钉在四边（留 16pt 边距）。
    NSLayoutConstraint.activate([
        root.topAnchor.constraint(equalTo: view.topAnchor, constant: 16),
        root.leadingAnchor.constraint(equalTo: view.leadingAnchor, constant: 16),
        root.trailingAnchor.constraint(equalTo: view.trailingAnchor, constant: -16),
        root.bottomAnchor.constraint(equalTo: view.bottomAnchor, constant: -16),
        // stack 默认按内容紧凑排，下面两条让“输入行”和“表格”横向铺满
        inputRow.widthAnchor.constraint(equalTo: root.widthAnchor),
        scroll.widthAnchor.constraint(equalTo: root.widthAnchor),
    ])
}
易踩的坑

很多人照着写完发现"表格只剩薄薄一条、撑不满中间"。原因是垂直 NSStackView 默认让每个子项保持自己的内容大小（content hugging），没人主动"长高"。解法就是上面那行 scroll.setContentHuggingPriority(.defaultLow, for: .vertical)——把滚动容器的"垂直抗拉伸意愿"调低，它才会乖乖吃掉计数行、输入行、删除按钮之外的全部剩余高度。

第 4 块 · 对应第 6 章（dataSource 供数 + delegate 供视图与选择）。表格自己不存数据，它靠"问问题"获取一切：numberOfRows(in:) 回答"有几行"，tableView(_:viewFor:row:) 回答"第 N 行该显示哪个视图"。这是视图型（view-based）表格的现代写法——每行是一个真正的 NSView，而不是老式 cell。选择变化由 tableViewSelectionDidChange(_:) 通知我们去刷新顶部计数和删除按钮的可用状态。

// MARK: - NSTableViewDataSource：供应“有几行”
func numberOfRows(in tableView: NSTableView) -> Int {
    items.count
}

// MARK: - NSTableViewDelegate：供应“第 row 行长什么样”
func tableView(_ tableView: NSTableView,
               viewFor tableColumn: NSTableColumn?,
               row: Int) -> NSView? {
    let id = NSUserInterfaceItemIdentifier("TodoCell")
    // 先问表格“有没有可复用的旧视图”，有就拿来改文字，没有才新建
    let cell = (tableView.makeView(withIdentifier: id, owner: self) as? NSTableCellView)
        ?? makeCell(id: id)
    cell.textField?.stringValue = items[row]   // 永远从 items 取值
    return cell
}

// 新建一个内含 label 的单元视图（只在没有可复用视图时调用）
private func makeCell(id: NSUserInterfaceItemIdentifier) -> NSTableCellView {
    let cell = NSTableCellView()
    cell.identifier = id
    let label = NSTextField(labelWithString: "")
    label.translatesAutoresizingMaskIntoConstraints = false
    cell.addSubview(label)
    cell.textField = label
    NSLayoutConstraint.activate([
        label.leadingAnchor.constraint(equalTo: cell.leadingAnchor, constant: 4),
        label.trailingAnchor.constraint(equalTo: cell.trailingAnchor, constant: -4),
        label.centerYAnchor.constraint(equalTo: cell.centerYAnchor),
    ])
    return cell
}

// 选中变化 → 刷新计数 + 删除按钮可用性
func tableViewSelectionDidChange(_ notification: Notification) {
    deleteButton.isEnabled = (tableView.selectedRow != -1) // -1 表示没选中
    refreshCount()
}

第 5 块 · 对应第 5 章（target-action）+ 回车的 delegate 写法。三个动作方法：添加、删除、刷新计数。@objc 是因为 target-action 走的是 Objective-C 的消息机制，方法必须暴露给运行时。回车提交则用 NSTextField 的代理方法来捕捉"按下了 Return 键"这个命令（思考题①会专门追问为什么是这种写法）。

// --- target-action：点“添加” ---
@objc private func addTapped() {
    let text = inputField.stringValue.trimmingCharacters(in: .whitespaces)
    guard !text.isEmpty else { return }   // 空白不入列
    items.append(text)                    // 1) 改数据
    inputField.stringValue = ""           // 清空输入框
    tableView.reloadData()                // 2) 刷界面：让表格重新照镜子
    refreshCount()
}

// --- target-action：点“删除选中” ---
@objc private func deleteTapped() {
    let row = tableView.selectedRow
    guard row != -1 else { return }
    items.remove(at: row)                 // 1) 改数据
    tableView.deselectAll(nil)            // 删除后清掉选中，避免 selectedRow 指向已不存在的行
    tableView.reloadData()                // 2) 刷界面
    refreshCount()                        // deselectAll 会触发 selectionDidChange，这里再保险刷一次
}

// --- 顶部计数：永远从 items 与表格状态读取，不自己记账 ---
private func refreshCount() {
    let total = items.count
    let selected = tableView.selectedRow == -1 ? 0 : 1
    countLabel.stringValue = "总数 \(total) · 已选 \(selected)"
}

// MARK: - NSTextFieldDelegate：捕捉回车
// 该方法来自 NSControlTextEditingDelegate（NSTextFieldDelegate 继承了它）
func control(_ control: NSControl,
             textView: NSTextView,
             doCommandBy commandSelector: Selector) -> Bool {
    // 回车在文本系统里对应 insertNewline(_:) 这个命令
    if commandSelector == #selector(NSResponder.insertNewline(_:)) {
        addTapped()      // 复用同一个“添加”逻辑
        return true      // 告诉系统：这个命令我处理了，别再走默认行为
    }
    return false         // 其它命令（Tab、方向键…）交还系统默认处理
}

注意"添加"逻辑只写了一份（addTapped()），点按钮和按回车都调用它。这正是"动作与触发解耦"的好处：触发方式可以有很多种，业务逻辑只有一处真相。

4. 综合思考题

下面四道题都跨越多个知识点。先自己想，再展开看答案——答案不止给"怎么做"，更要点透"为什么本质上只能这么做"。

思考题①：回车提交，为什么宜用 NSTextField 的 delegate（control(_:textView:doCommandBy:)），而不是"按下回车后去读它的 stringValue"？

查看答案

先厘清一个误解：stringValue 只是输入框里"当前有什么文字"，它不携带任何"用户做了什么动作"的信息。你随时都能读到它，但读到它并不代表"用户刚刚按了回车要提交"。换句话说，stringValue 是状态，而回车是事件——拿状态去推断事件，本身就错位了。更现实的问题是：你根本没有一个"按下回车后"的时机去读它，除非有人在那个时间点回调你——而那个回调正是 delegate。

那"按下回车"这个事件由谁报告？在 macOS 文本系统里，编辑文本的其实是一个隐藏的"字段编辑器"（field editor，本质是一个共享的 NSTextView）。你按下的每个键先经过它，被翻译成命令：回车 → insertNewline(_:)，Tab → insertTab(_:)，Esc → cancelOperation(_:)。NSTextField 会把这些命令转发给它的 delegate 的 control(_:textView:doCommandBy:)。所以代理方法是系统为你准备的、能精确捕捉"回车这一下"的入口。

那 action 不也能触发吗？能——NSTextField 在结束编辑（回车 / 失焦，取决于 sendAction(on:)）时也会发 action。但它的语义更模糊（你分不清是回车还是失焦触发的），且无法像代理那样"返回 true 吃掉这次回车、阻止换行 / 跳焦"。当你需要精确区分是哪个键、并决定是否拦截默认行为时，代理是更可控的工具。

本质：UI 编程里要分清"读状态"和"听事件"。提交是一个时间点上的事件，必须由系统在那个时间点主动告诉你（delegate 回调 / action），而不是你事后去翻一个静态字段猜它发生过。

思考题②：当待办上千条、表格滚动开始卡顿时，"全用 Auto Layout" 和 "在 layout() 里手动算 frame" 该怎么取舍？为什么？

查看答案

先想卡在哪。NSTableView 已经帮你做了视图复用（makeView(withIdentifier:) 回收离屏的行视图反复用），所以同时存在的视图数量只跟"屏幕能看到几行"有关，不随总条数膨胀——这一步本身不卡。真正可能卡的是每一行内部的布局成本：每次行被复用、重新填数据，约束系统都要为这行重新求解一遍。

Auto Layout 的代价在哪？约束不是免费的：每个 NSLayoutConstraint 都是布局引擎里的一条方程，引擎要把同一视图层级里所有方程联立求解。一行里几条约束没问题；但当行视图复杂（嵌套多层 stack、十几条约束）、又在快速滚动中每秒重排几十次时，求解开销会累积成可感的掉帧。Auto Layout 的优点是声明式、自适应、好维护——绝大多数界面都该首选它，本章的整体布局就用它。

什么时候换手动 layout()？当某个高频重排的视图（典型就是表格行 / 自绘 cell）的布局规则其实非常简单且固定（比如"label 距左 4pt、垂直居中"），用一行 frame = ... 直接算，比让引擎联立求解一组方程快得多。做法：在那个行视图的子类里重写 override func layout()，super.layout() 后按 bounds 手算每个子视图的 frame。这就是把"通用但偏重的求解器"换成"专用但极轻的算式"。

取舍原则：① 默认用 Auto Layout，别过早优化；② 只在用 Instruments 测出布局确实是瓶颈后，才把那个高频、规则简单的热点（几乎总是表格行）改成手动 frame；③ 不要全盘抛弃约束去手算整个界面——那样维护成本爆炸，且大部分视图根本不在热路径上。

本质：性能优化的方向永远是"把成本花在刀刃上"。Auto Layout 用"每次求解"换"开发期省心"；手动布局用"开发期多写几行算式"换"运行期省 CPU"。只有在高频 × 简单同时成立的地方，这笔交换才划算。

思考题③：macOS 视图的坐标系默认是"原点在左下、Y 轴向上"。这件事对自绘 draw(_:) 和命中测试 / 事件坐标分别有什么影响？

查看答案

先建立直觉。大多数人脑子里的屏幕坐标是"原点在左上、Y 向下"（和 iOS / 网页一样）。但 macOS 的 NSView 默认相反：原点在左下、Y 向上，像数学课的第一象限。这不是 bug，是历史约定（源自 PostScript 绘图模型）。

对 draw(_:) 的影响：你在 draw(_:) 里画的所有坐标都按这套系来。想"在顶部画一条线"，那条线的 Y 应该接近 bounds.height（顶边），而不是接近 0（那是底边）。文字、图片如果按"左上原点"的思维去摆，就会上下颠倒位置。两种解法：① 重写 override var isFlipped: Bool { true }，把这个视图翻转成"左上原点、Y 向下"，从此画图、布局都和直觉一致——这也是为什么很多自定义视图都加这一行；② 不翻转，但所有 Y 都用 bounds.height - y 自己换算。推荐第一种，省心且不易错。

对命中测试 / 事件坐标的影响：鼠标事件给你的是窗口坐标，你通常用 convert(event.locationInWindow, from: nil) 把它转成本视图坐标，再判断点中了哪里（命中测试）。关键是：这个转换的结果一定遵循该视图当前的坐标系。如果视图没翻转，转出来的点也是"左下原点"的；你拿它和某个用"左上思维"算出来的矩形比，就会判错。所以——画图的坐标系和命中测试的坐标系必须是同一套。这也是 isFlipped 如此好用的原因：翻转一次，画图、布局、命中全部统一到"左上原点"，三者天然对齐，不会出现"看着点在按钮上、判定却没中"的诡异现象。

本质：坐标系不是"画图的细节"，而是视图里一切几何运算共享的参照系。绘制、布局、命中测试只要用了不一致的参照系，就必然错位。isFlipped 的价值是把整套参照系一次性对齐到你大脑习惯的方向。

思考题④：同一个交互，有时 target-action 能做、delegate 也能做。该怎么选？顺便把 dataSource、notification 一起放进来辨析。

查看答案

这四种其实是四种不同性质的"对话"，区分它们最好的办法是问："谁在对谁说什么样的话？"

  • target-action（"喂，发生了一件事，你处理一下"）：控件 → 一个目标，一对一、单向通知一个动作发生了，没有返回值、不询问意见。适合简单、独立、就一种结果的交互——按钮被点、滑块被拖。本章"添加 / 删除"按钮就是它。优点是极轻、解耦（控件不认识谁在听）。
  • delegate（"我要做某件事，先问问你该怎么办 / 要不要拦"）：对象 → 它的一个代理，一对一，可以有返回值、可以决定流程。当交互需要"决策、定制、或拦截默认行为"时用它。本章回车的 doCommandBy 返回 true 来"吃掉"这次回车，就是 delegate 才有的拦截能力；target-action 给不了这种"我处理了，你别走默认了"的回话。
  • dataSource（"我要显示东西，但数据在你那儿，给我"）：本质是一种专门负责"供应数据"的 delegate。区别在语义：delegate 多用于"行为 / 事件"（选中了、要不要展开），dataSource 专管"内容 / 数量"（有几行、第 N 行是什么）。AppKit 故意把它们拆成两个协议（如 NSTableViewDataSource 与 NSTableViewDelegate），就是为了让"数据"和"行为"职责分明——本章表格里 numberOfRows 属前者、tableView(_:viewFor:row:) 属后者。
  • notification（"我广播一声，谁关心谁来听"）：发出者 → 任意多个、互不相识的监听者，一对多、单向、无返回。适合"一件事发生了，可能有好几处都想响应，且发出者不该认识它们"。tableViewSelectionDidChange(_:) 收到的参数就是一个 Notification——表格选择变了，它广播出来，你恰好实现了这个代理方法去刷新计数。

选择决策树：① 需要返回值 / 拦截 / 定制流程吗？→ delegate。② 是在供应数据（数量、内容）吗？→ dataSource。③ 只是"一个动作发生了、一对一处理、不要回话"？→ target-action（最简就用最简）。④ 是"一件事、多个互不相识的对象想各自响应"？→ notification。

本质：四者的真正分野是两条轴——基数（一对一 vs 一对多）和方向性（单向通知 vs 双向问答）。target-action 是"一对一单向"，delegate / dataSource 是"一对一双向（可回话 / 可拦截）"，notification 是"一对多单向"。先想清楚这次交互在这两条轴上的位置，选哪个就不再纠结。

本质

恭喜你走完了入门。回头看，AppKit 并不神秘：它把"启动、窗口、控制器、布局、事件、数据"这几件事，用明确的协议和回调固定下来，让你只需填空——告诉系统"有几行""第 N 行什么""被点了怎么办"。真正难的从来不是记住 API，而是在每个岔路口分清"这是状态还是事件""谁该拥有这份数据""这次该用哪种对话方式"。把这几个判断练成直觉，你就从"会照着写"升级成了"知道为什么这么写"。
