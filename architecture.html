CCTerm 是怎么搭起来的

这是一份写给小白的架构导览。我们从整体到局部、从直觉到机制，带你认识这个 macOS 原生应用由哪几大块组成、它们怎么拼在一起，以及为什么这么设计。

这是个什么 App，以及这篇文章怎么读

本节用一句话讲清 CCTerm 是什么，告诉你读完能带走哪三样东西，再给出一份可点击的章节目录，方便随时跳转。

一句话：CCTerm 是 Claude Code 的原生 macOS 客户端

很多人第一次用 Claude Code，都是在终端里敲命令、看回复刷屏。这种方式能用，但体验就像“在一个终端标签页里跟 AI 聊天”：聊天记录混在命令行里，多个项目挤在同一个窗口，想回头翻某次对话也不方便。

CCTerm 做的事，就是把“终端里的一个标签页”升级成一个真正的 Mac App。打个比方：终端里的 Claude Code 像是在记事本里跟人发消息，而 CCTerm 把它搬进了一个专门的聊天软件——窗口、列表、滚动条、弹窗一样不少，而且都是 macOS 原生的样子。

用过它，你会得到这些：

  • 左侧的会话侧边栏——每一次对话叫一个会话（对应代码里的 Session）。会话整齐列在左边，可以拖进文件夹分组，也可以拖动重新排序。
  • 能瞬间滚动的长聊天记录——中间那块滚动显示历史消息的区域叫聊天记录（对应代码里的 NativeTranscript2）。哪怕对话已经很长，上下拖动也不卡顿。
  • 实时流式回复——Claude 的回答像打字机一样一个字一个字刷出来，不用等全部生成完才一次性显示。
  • 原生的权限确认框——Claude 想执行某条命令时，App 会在输入栏上方浮出一张 macOS 原生风格的确认卡片，让你点“允许”或“拒绝”，而不是把决定权藏进命令行参数。这张卡片对应代码里的 PermissionCardView。

技术上，CCTerm 用 SwiftUI 和 AppKit 两套界面框架搭成（这两个名词后面会详细解释，这里先记住：它们是苹果做 Mac 界面的两套工具），最低支持 macOS 14（Sonoma）。它的入口非常干净：一个标了 @main 的 CCTermApp，通过 @NSApplicationDelegateAdaptor 把启动权交给 AppDelegate；AppDelegate 再在 applicationDidFinishLaunching 里亲手创建主窗口（MainWindowController）。具体怎么一步步搭起来，后面的“启动链路”一章会拆开讲。

鸟瞰：四大区域

本节给你画出整个 App 窗口的第一张全局地图：它由四个一眼能记住的可见区域拼成，每个区域用哪套 UI 框架（SwiftUI 还是 AppKit），图上也一并标清楚。

钻进细节之前，先把窗口当成一栋房子来看。打开 CCTerm，你的目光会落在四个固定位置：最上面横着一条窗口工具栏，左边竖着一列侧边栏，右侧的大块区域主体显示聊天记录，底部浮着一个输入栏。先记住这四块的相对位置，后面所有组件和数据流都挂在它们身上。

下面把图里的四块逐一点名，说清楚每块干什么、用哪套框架、为什么这么选：

  • ① 侧边栏（SidebarViewController）——窗口最左边的会话列表。会话指一次完整的对话；侧边栏把你所有的会话竖着列出来，还能把它们装进文件夹分组、拖来拖去重新排序。它用 AppKit 的 NSOutlineView（系统的「来源列表」样式）来画。之所以直接用 AppKit，是因为只有这样才能拿到标准的文件夹拖拽和现成的展开/折叠动画。
  • ② 聊天记录（NativeTranscript2）——右侧大块区域的主体，一行接一行滚动显示历史消息，Claude 的回复在这里逐字流式刷出。每一行（一个渲染单元）在代码里叫一个块（Block）。它用 AppKit 的 NSTableView 加 Core Text 自己画。原因是：会话一长，SwiftUI 的 List 和 LazyVStack 扛不住那么多行，也扛不住这种自定义布局和选择语义。这是全项目下沉到 AppKit 最核心的理由——性能。
  • ③ 输入栏（InputBarView2）——浮在聊天记录底部之上的输入区，你在这里敲字、发送给 Claude。它是地地道道的 SwiftUI 视图，靠寄宿（把 SwiftUI 视图放进 AppKit 容器，用的是 NSHostingView）叠在聊天记录上面。它体现了本项目的总策略：默认用 SwiftUI，只有性能或时序逼到墙角时才下沉到 AppKit。
  • ④ 窗口工具栏——窗口最上面的横条，放着当前项目信息和聊天记录的搜索框。横条本身是 AppKit 的 NSToolbar（搜索框是 NSSearchToolbarItem）。因为搜索框需要首响应者切换和 ⌘F 语义，而 SwiftUI 的 .searchable 给不了这两样。横条里那些小控件（项目信息块、归档筛选器）又是 SwiftUI，同样靠寄宿放进工具栏的格子里。
为什么橙色这么多？本项目的口号是「默认 SwiftUI，AppKit 是例外」，可这张鸟瞰图里偏偏三块是橙色。道理在于：越靠近窗口骨架、越需要精确控制性能和时间顺序的地方，越要用 AppKit；真正承载「内容」的小块（输入栏、工具栏里的控件）才放心交给 SwiftUI。换句话说，橙色是外壳与骨架，蓝色是填进骨架的内容——这条分界线会贯穿后面所有章节。

这张鸟瞰图只回答了「窗口长什么样、各块归谁管」。接下来我们顺着时间往回倒一步，看看双击图标之后，这四块是谁、按什么顺序一个接一个搭起来的——这就是下一节的「启动链路」。

启动链路：从双击到见面

本节只讲清楚一件事：你双击 Dock 里的图标后，App 内部按什么顺序、一层接一层把对象创建并拼装起来，最后才显示出那张熟悉的"左侧边栏 + 右聊天区"界面。

上一节的鸟瞰图告诉你 App 由哪几大块组成，那是一张静态地图。可地图不会告诉你这些块谁先出生、谁把谁创建出来。这一节就把镜头切到"出生顺序"：从最顶层的 App 入口出发，沿一条单向的创建链往下走，一直走到屏幕上挂出第一块可见面板。先理解这条链，等你后面看组件树时，就能一眼看出每个节点"在哪一步、被谁、装了进来"。

先交代一个贯穿全程的术语。视图控制器（代码里是 NSViewController，AppKit 的概念）是 AppKit 里"管理一块界面"的基本单位，负责一块视图的生命周期与内容。你可以把它想成一个房间的"房间管理员"；对应到代码就是：本节出现的 MainWindowController、MainSplitViewController、DetailRouterViewController 都属于这一族。它们用橙色标记，因为主窗口骨架是用 AppKit 搭的。最顶层的 App 入口、以及具体面板里寄宿的内容则属于 SwiftUI，用蓝色标记。

有一个细节值得点明，它解释了"为什么点一下侧边栏，右边能马上整块换掉，而且不卡顿、不撕裂"。用户点击时，侧边栏（AppKit）调用 MainSelectionModel.select(_:) 写入新选择；这个方法在更新状态的同一刻，就直接同步通知详情路由器去换面板。详情路由器是这个选择的唯一结构性观察者——换句话说，整套"换面板"的动作都收口在它一个人身上。正是这条同步通路，保证了"换会话"这件事在点击的同一个 source 阶段里一次做完，不会拖到下一帧。

为什么主窗口骨架要用 AppKit，而不是全用 SwiftUI？

本项目的总策略是"默认 SwiftUI，AppKit 是例外"，而主窗口这条骨架链恰好踩中了例外。原因在于时序：聊天记录的挂载、以及随之而来的尺寸级联，必须跑在 AppKit 的 source 阶段（也就是处理点击事件的那一刻），不能拖到 SwiftUI 的提交阶段。所以 AppDelegate 选择在 applicationDidFinishLaunching 里亲手创建窗口，而不是声明一个 SwiftUI Window 场景让系统自动开窗——只有这样，换会话、滚动到底这类动作才能和用户的点击落在同一帧里。至于"source 阶段是什么、为什么差一帧就会出问题"，本文后面"关键设计抉择"一节会专门讲透，这里先记住结论即可。

到这里，对象的"出生顺序"就理清了：SwiftUI 入口交棒给 AppKit 代理，代理造出窗口，窗口里分出左右两栏，右栏的路由器再按当前选择挂上恰好一个面板。下一节我们把镜头拉远，用一棵组件树把这些块连同它们寄宿的 SwiftUI 内容一并铺开，并逐个标出谁是 SwiftUI、谁是 AppKit。

组件树总览

本节把第 2 章鸟瞰图里的"四大区域"落实成一棵真实的对象树：从 App 入口往下，每一层是谁、归 SwiftUI 还是 AppKit，一眼看清。

第 2 章告诉我们 App 分成几大块，第 3 章讲了它们启动时按什么顺序被创建。现在我们把这些块装进一棵组件树——也就是"谁包含谁"的层级图。先把这棵主干树打好底，第 5 章再钻进每一块讲细节，读者就不会迷路。

怎么读这棵树。树表达的是"父子包含"关系：上一层是容器，下一层是被它装进去的内容。本节只画主干结构节点——也就是搭起骨架的那几个关键对象。每个文件都塞进来，反而看不清主干。叶子级的细节（比如聊天记录内部还挂了哪些协调器）留到第 5 章逐块讲。

逐块讲解（一）：窗口外壳与导航

本节把"主窗口骨架"这条链一块一块拆开：窗口、左右分栏、详情路由器、选择状态、侧边栏、窗口工具栏，各自是什么、像什么、为什么大多用 AppKit 写。

上一节的组件树给了你一张全景图。现在我们走进树的左半边，也就是用户打开 App 就直接面对的那副"骨架"：一个窗口，一道把屏幕劈成左右两半的分栏，左边的侧边栏，右边随选择切换的面板，再加上顶上那条工具栏。这几块只有一处例外，其余都用 AppKit 写，原因下面会逐块点明。先用一张小表摆出"谁负责什么、用什么技术"，再逐块展开。

MainWindowController AppKit

是什么。它持有那个真正的 macOS 窗口（NSWindow），设好标题、尺寸、透明标题栏，把工具栏装上去，再把左右分栏塞进窗口当内容。它是主窗口骨架这条链的最顶端。

打个比方。它像一栋楼的"地基加外墙"：先把这一圈框架立起来，里面才能隔房间、摆家具。对应到代码里，它在 AppDelegate 的 applicationDidFinishLaunching 里被创建，是 App 启动后第一个立起来的可见对象。

为什么用 AppKit？ SwiftUI 也能用一个 Window 场景开窗口，但本项目刻意不走那条路。原因写在代码注释里：聊天记录（NativeTranscript2）的挂载和尺寸变化级联，必须跑在 AppKit 的 source 阶段，不能和 SwiftUI 的 commit 阶段交错。（source 阶段是 macOS 一次界面刷新周期里"你的代码真正在跑"的那一段，下文还会提到。）把窗口的根握在 AppKit 手里，时序才可控。

MainSplitViewController AppKit

是什么。它是一个两栏分栏控制器：左边那栏放侧边栏，右边那栏放详情路由器。它还干一件关键的"分发"活——把进程级状态容器（AppState，全 App 共享的状态袋子）拆开，按需要打包成两个更小的袋子分别下发。

打个比方。它像办公室里的一道隔断墙，把空间分成"会客区"和"工作区"；同时又像前台分拣员，把进来的一大箱物资拆成"左屋要的"和"右屋要的"两小箱，各送各家。对应到代码里，左栏拿到的是 SidebarContext，右栏拿到的是 DetailContext。

为什么这么拆？ 项目的规矩是不把整个状态袋子一股脑塞给所有人，而是谁用什么就只给什么。所以这里把侧边栏要用的四样（选择状态、会话管理、文件夹排序、"在某 App 中打开"服务）装进 SidebarContext，把右侧面板要用的五样（选择状态、会话管理、最近项目、输入草稿、语法高亮引擎）装进 DetailContext。这样以后增减一个依赖，只改这一个袋子，不用动一长串初始化器。

DetailRouterViewController AppKit

是什么。它是右侧那块空容器，叫详情路由器。它的工作很专一：看当前选中了什么，就在右侧挂上恰好一个对应的子面板，绝不同时挂两个。选会话就挂聊天面板（ChatSessionViewController），选"新建会话"就挂配置面板（ComposeSessionViewController），选"归档"就挂归档面板（ArchiveViewController），选中的会话还只是草稿就挂草稿落地页（DraftSessionLandingViewController）。

打个比方。它像剧院舞台的换景机制：同一块舞台，根据剧目换上不同布景，但台上任何时刻只有一套布景。对应到代码里，只有"换剧目"（切到不同类型的面板）才真的拆旧搭新；如果只是从一个会话切到另一个会话（同属"聊天"这一类），它复用同一块面板，只把新会话通过 present(sessionId:) 递进去，省下重建开销。

为什么它是选择状态的"唯一结构性观察者"？ 这是本项目最讲究的一处时序设计。用户在侧边栏点一下，切换面板这件事必须发生在点击的同一刷新周期里，否则会话切换会"跨帧撕裂"——右侧面板晚一拍才换，画面闪一下。SwiftUI 的 @Observable 自动刷新会把这件事推迟到下一段（beforeWaiting），来不及。所以路由器专门注册成选择状态的结构性观察者（selectionObserver），由选择写入时同步叫醒它，当场把面板换好；之后的淡入只是一层不影响结构的动画。项目明确规定：这条"向上的结构性通路"只此一条，不要再加第二个。

MainSelectionModel @Observable 状态

是什么。它记住"现在选中的是哪一项"，是选择的唯一真相。它是一块共享状态（用 @Observable 标注），既不是 AppKit 控件，也不是 SwiftUI 视图，而是两边都来读写的同一份数据。选项不是一串随便的字符串，而是一个类型明确的枚举 MainSelection：要么"什么都没选"，要么"新建会话"，要么"某个具体会话"，要么"归档"。它还顺带记着草稿会话的 id、归档页的文件夹过滤等小状态。

打个比方。它像电视遥控器上的"当前频道"指示：所有人想知道"现在在看哪个台"，都来问它。对应到代码里，侧边栏一被点中，就调它的 select(_:) 写入新选择；这个方法做两件事——先更新 @Observable 的值（这样 SwiftUI 的输入栏、侧边栏单元格等会自动重画），再同步通知上面那个详情路由器去换面板。

为什么把选项做成枚举？ 代码注释解释得很直白：早先这里是一个"拿字符串当类型用"的字段，值要么是真实会话的 id、要么是 __archive__ 这种占位符。结果每个用到它的地方都得先判断"这到底是会话还是某个标签页"，一旦漏判，就会出现"在归档页上画出聊天界面"这种 bug。改成枚举后，编译器逼着每处都把所有情况列全，错误在编译期就被挡住。

SidebarViewController AppKit

是什么。它就是左侧那列侧边栏：顶部两个固定入口（新建会话、归档），下面按文件夹分组列出历史会话。它基于 NSOutlineView（macOS 标准的"源列表"树形列表）写成，会话可以折叠进文件夹，文件夹之间可以拖拽换顺序。

打个比方。它像访达（Finder）左边那条边栏，又像文件管理器里可展开的目录树：点文件夹能收起展开，拖一下能换位置。对应到代码里，文件夹的展开/折叠用的就是 NSOutlineView 自带的 expandItem(_:) / collapseItem(_:) 动画，文件夹的拖拽用的是 AppKit 标准那套 pasteboardWriterForItem / validateDrop / acceptDrop。

为什么用 AppKit？ SwiftUI 的 .listStyle(.sidebar) 底层其实也是 NSOutlineView，可隔着一层就拿不全想要的能力。直接用 AppKit，才能一次性拿到标准的文件夹拖拽排序和内建的展开/折叠动画——这两样正是侧边栏的核心交互。换句话说，这里下沉到 AppKit 是为了"补齐 SwiftUI 给不了的能力"，符合"默认 SwiftUI、AppKit 是例外"的总原则。

窗口工具栏 · NSToolbar AppKit

是什么。它是窗口顶部那条横栏，由 MainWindowController 装配。上面有切换侧边栏的按钮、显示当前项目名/分支的小标签、归档页才出现的文件夹过滤按钮，以及右侧的聊天记录搜索框（用专门的 NSSearchToolbarItem）。这些条目会跟着当前选择动态增减——比如不在某个会话里时，项目标签就不显示。

打个比方。它像浏览器顶上的地址栏加工具条：你在看什么页面，上面的按钮就跟着变。对应到代码里，工具栏里那些用 SwiftUI 写的小部件（项目标签、过滤按钮）是寄宿进来的——也就是用 NSHostingView 把 SwiftUI 视图放进 AppKit 容器里显示。

为什么搜索框用 AppKit？ SwiftUI 有现成的 .searchable 修饰符，但它给不了聊天记录搜索需要的两样东西：让搜索框成为"首响应者"（键盘焦点落在它身上），以及响应 ⌘F 这个快捷键。所以这里改用 NSSearchToolbarItem，再用一个桥接对象（TranscriptSearchToolbarBridge）把"在框里打字、回车跳下一个、Shift+回车跳上一个、⌘F 抢焦点"接到聊天记录的搜索逻辑上。

到这里，窗口外壳与导航这一层就讲完了：一个 AppKit 窗口立起骨架，分栏把屏幕劈成两半，侧边栏负责"选什么"、路由器负责"右边显示什么"，中间用一块 @Observable 的选择状态串起来，顶上还有一条随选择变化的工具栏。下一节我们走进右侧最重的那块——聊天区，看看一条对话记录是怎么被画出来、又怎么瞬间滚动的。

逐块讲解（二）：聊天区

本节拆开屏幕中间最热闹的那块区域——聊天区，讲清楚它由哪几块拼成：谁负责"这个面板显示什么"，谁负责"把历史消息高速画出来"，谁负责"把后台数据翻译成一行行能渲染的内容"，以及底部那条输入栏和它的自动补全。

上一节我们看完了主窗口骨架：窗口、分栏、详情路由器、侧边栏、工具栏。用户在侧边栏选中一个会话（一次完整的对话，对应代码里的 Session）时，详情路由器（按选择挂载对应面板的那一层，对应 DetailRouterViewController）就挂载聊天面板。下面这些组件全都住在这块面板里。

读这一节前先记住一个原则：聊天区大约九成是"单向数据流"——状态向下流，事件少量向上。这里的"九成"是个约数、不是实测统计，意思只在强调"绝大多数数据向下走、极少量事件向上回报"。说白了就是 AppKit 搭外壳，SwiftUI 当叶子。哪块用 AppKit、哪块用 SwiftUI，下面每张卡片都用颜色加文字标签标了出来。
图例： SwiftUI（蓝） AppKit（橙） 服务 / 数据层（绿） 中性 / 纯逻辑（灰） 颜色只是辅助；每块都另带文字标签，不靠颜色单独区分。

一、聊天面板：ChatSessionViewController

打个比方，ChatSessionViewController 像一间房子的"软装管家"：它不亲手砌墙（不直接管聊天记录怎么渲染），只负责房间里挂什么、摆在哪——顶部和底部的渐隐遮罩、底部寄宿（把 SwiftUI 视图放进 AppKit 容器，对应 NSHostingView／NSHostingController）进去的输入栏、悬浮在最上层的权限确认卡，还有焦点处理。对应到代码里，它就是详情路由器为 .session(_)／.none 这两种选择挂载的那个面板。

它有意把一件重活交了出去：聊天记录的构建、绑定、滚到底、换台和淡入，全都委托给下面要讲的 TranscriptSwapCoordinator。这样分工是因为聊天记录的挂载和尺寸级联必须跑在 AppKit 的 source 阶段（runloop 一个迭代里"你的代码在跑"的那一段），所以面板本身用 AppKit 的 NSViewController，而不是 SwiftUI 视图。

ChatSessionViewController AppKit · NSViewController

聊天面板。只管"这个面板显示什么"：上下渐隐遮罩、底部输入栏寄宿、权限确认卡、焦点。聊天记录的活全部委托给 TranscriptSwapCoordinator。

它管的几样东西：上下渐隐遮罩（TranscriptTopScrimView／TranscriptBottomScrimView，点击会"穿透"到底下的表格，不挡住交互）、底部贴边的输入栏寄宿（restingBarHost）、悬浮在最上层且点击可穿透的权限卡寄宿（permissionCardHost）。它不亲自观察选择状态，而是由详情路由器调用 present(sessionId:) 来驱动挂载。

二、聊天记录的"换台员"：TranscriptSwapCoordinator

用户在侧边栏来回切换会话时，中间这块聊天记录要跟着"换一整套内容"。TranscriptSwapCoordinator 就是专门干这件事的"换台员"。它有一个很关键的身份：它是 currentSession（当前正在显示的会话）以及"每次挂载所用资源"（滚动视图 Transcript2ScrollView、弹窗呈现器 Transcript2SheetPresenter 等）的单一所有者。

"单一所有者"听起来抽象，落到代码里就是：判断要不要换、记下要换走的旧会话、绑定新会话、清理现场，这一整套都只在它一个对象里发生。面板和它绝不能各持一份 currentSession。否则切换动画进行到一半时，两边记的"当前会话"会对不上，就乱了。

它还负责同会话淡入：详情路由器判定这是一次"新鲜内容"的首次进入、而且窗口可见时，新旧聊天记录之间只做一个透明度的淡入淡出过渡。换台时它遵守一条铁律——先在上层把新的搭好、绑好、滚到底，最后才拆掉旧的。整个结构性切换裹在一个关闭隐式动画的事务里，这样面板永远不会闪出一块空白。

TranscriptSwapCoordinator AppKit · @MainActor 类

currentSession 与每次挂载资源的单一所有者。负责聊天记录的构建、绑定、滚到底、换台和同会话淡入。

为什么单独抽出来："换台"这套逻辑有先后顺序、又容易出错，把它收进一个对象，边界就清晰了——面板只管"显示什么"，换台员只管"怎么换"。代价是新人得先理解这套"谁拥有什么"的划分，才好动它。

三、聊天记录本体：NativeTranscript2

聊天记录（滚动显示历史消息的区域，对应 NativeTranscript2）是整个 App 性能要求最高的地方。一次长对话可能有上万行（每一行是一个"块"，对应 Block），用户拖动滚动条时还得稳稳保持 60 帧。它的做法是：用 AppKit 的 NSTableView 当表格骨架，每一行的内容用 Core Text 亲手画出来（自绘），而不是交给现成的视图。

为什么聊天记录非用 AppKit 自绘不可？

SwiftUI 的 List 和 LazyVStack 在这里扛不住，原因有三：

  • 行数太多。上万行的滚动需要一套"只画看得见的行、其余惰性计算"的机制，而且行高必须能同步算出来。NSTableView 会同步问"这一行多高"，自绘方案能立刻从缓存里答上来；List／LazyVStack 要么阻塞，要么先用估算值、滚动时再跳变。
  • 自定义布局太复杂。代码块、差异对比、工具调用折叠、用户气泡截断……每种块的排版规则都不一样，自绘能把"行高是宽度的纯函数"算清楚，并按块缓存。
  • 选择语义太特殊。跨行选中文字、搜索高亮叠在选中之上，这些都得精确控制每个字形的位置，只有自绘才做得到。

所以这里是"默认 SwiftUI、AppKit 是例外"原则里的一个明确例外。这是项目里写明的性能契约，不能随手换回 SwiftUI。

聊天记录的"行数据真相"只有一处：Transcript2Coordinator.blocks 这一个块数组。没有第二份镜像，也没有并行的差异结构。所有的增删改都从一个入口流进去。一句话总结这块的内部分工：

五、输入栏：InputBarView2

底部那条输入框区域就是输入栏（对应 InputBarView2）。它是一个纯 SwiftUI 视图，只负责"长什么样"——文本框、发送／停止按钮、附件缩略图条；至于发送、停止、是否正在运行这些行为，都由外部注入进来，它自己不持有会话句柄。它通过寄宿放进面板底部，贴着下边缘，只占自己内容那么高。

InputBarView2 SwiftUI · View

为什么用 SwiftUI：它是一个小小的声明式组件，没有聊天记录那种性能压力，正好落在"默认 SwiftUI"的常规情形里。它通过寄宿（NSHostingView）放进 AppKit 面板的底部。

六、自动补全：Completion

在输入栏里敲 @ 提文件、敲 / 调命令时弹出的那个候选列表，就是自动补全。它分成两侧：UI 侧负责弹窗和键盘交互，服务侧负责"从哪儿取候选项"。

UI 侧 SwiftUI

CompletionState（补全的交互状态，是个 @Observable 对象）加 CompletionListView（候选列表视图），配 CompletionItem、CompletionTriggerRule（哪个字符触发补全）。

服务侧 服务层

FileCompletionStore（文件名候选）与 SlashCommandStore（斜杠命令候选）。两者是按当前目录缓存的 .shared 单例，不走依赖注入。

一个小细节，体现了"克制"原则：FileCompletionStore 和 SlashCommandStore 之所以保持 .shared、而不像别的服务那样注入，是因为它们本质上是"按当前目录的缓存"，并不是要随会话切换重建的服务。这是项目里一个明确标注"不要改成注入"的设计选择。

七、Markdown 解析：Components/Markdown

助手的回复里满是 Markdown——标题、列表、代码块、表格、链接。这些文本要变成聊天记录里的块，先得有人把它解析成结构化的中间表示（IR）。Components/Markdown/ 就是这个解析器：它读 GFM（GitHub 风格 Markdown），产出一套内部 IR（由 MarkdownDocument／MarkdownTypes／MarkdownConvert／MarkdownAutolink／MarkdownMath 组成），再交给上面提到的 MarkdownToBlocks 转成块，最终被聊天记录消费。

这里要澄清一个容易踩的归属问题：Components/Markdown/ 虽然放在"可复用组件"目录下，但它不是 SwiftUI 视图——这几个文件只引入 Foundation 和底层的 Markdown 解析库，既不碰 SwiftUI，也不碰 AppKit。它是一段与界面框架无关的纯解析逻辑，只把文本拆成数据结构，所以下面这张卡片用中性灰，不套蓝色（SwiftUI）或橙色（AppKit）的标签。

Components/Markdown 中性 · 纯解析逻辑（非 UI 框架）

GFM 解析器 → 内部 IR。把 Markdown 源码拆成有序的片段，供聊天记录的翻译层消费。只引入 Foundation 与 Markdown 库，不依赖 SwiftUI／AppKit。

它在链条里的位置：Markdown 源码 → 这里解析成 IR → MarkdownToBlocks 转成块 → 聊天记录渲染。它本身只解析、不渲染，职责单一。

到这里，聊天区这一大块就拆完了：ChatSessionViewController 管"显示什么"，TranscriptSwapCoordinator 管"怎么换台"，NativeTranscript2 管"高速把块画出来"，桥接层管"把消息翻成块"，再加上底部的输入栏、自动补全和 Markdown 解析。下一节我们走进更靠后的一层——会话与数据层，看看这些块背后的消息、状态，以及它们和底层命令行的通信是怎么运转的。

逐块讲解（三）：会话与数据层

本节讲清楚一次会话（Session，一次完整的对话）背后那一摞"看不见的零件"：谁存数据、谁干活、谁跟底层命令行程序说话、流式回复怎么变成逐字打出来的效果，以及这一层为什么坚持"服务靠注入、视图绝不自己造服务"。

前两节讲的是用户能看见的外壳和聊天区。但屏幕上的每一条消息、每一个运行状态，都来自下面这一层默默工作的对象。这一层属于服务层（负责业务逻辑与数据的对象，本文统一用绿色标注）。它不画任何界面，只管"数据是什么、行为怎么发生"。我们从最核心的两块讲起：装数据的 Session 和干活的 SessionRuntime。

核心两块：装数据的 Session 与干活的 SessionRuntime

这两块是一对搭档，职责严格分开：一个管"是什么"，一个管"做什么"。

服务层 Session

@Observable · 一次会话的数据真相

它是一次会话的"档案袋"，把这次对话的全部事实装在一起：消息时间线、当前状态（是否在运行、是否报错）、Claude 列出的待办计划（todos）、后台执行的任务（tasks），以及上下文窗口用量。界面上凡是想知道"这次对话现在怎么样"的地方，都来读它。

打个比方：Session 像一份病历本，只记录事实，本身不做检查。

服务层 SessionRuntime

会话运行时 · 驱动一次会话行为的对象

会话运行时是真正干活的引擎。它隔着一层 CLIClient，间接驱动底层命令行子进程：把用户输入发出去，把 Claude 的回复收进来，一路维护运行状态。界面上的"发送""中断""换模型"等动作，最终都落到它身上。

接着上面的比方：SessionRuntime 像主治医生，做检查、开处方，再把结果写回病历本。

一次会话有两个阶段，Session 用一个内部的 Phase 状态在两者之间切换。在底部新建卡片里填配置时，会话处于草稿阶段（.draft(SessionDraft)）：没有子进程、没有持久记录、也没有消息。用户发出第一条消息的那一刻，Session 造出一个 SessionRuntime，把阶段翻成活动阶段（.active(SessionRuntime)）。这个翻转只发生一次，之后所有动作都转交给运行时。界面代码只读 session.title、session.messages 这类转发属性，从不直接去看它处在哪个阶段。

为什么把 SessionRuntime 拆成一摞扩展文件

会话运行时要管的事很多。如果全塞进一个文件，几千行代码挤在一起，读起来非常吃力。代码采用 Swift 的"扩展（extension）"机制，按职责把它切成多个文件——类还是同一个类，只是把不同职责的方法分门别类，放进不同文件。

一个 SessionRuntime，按职责切成多个扩展文件（每块只管一件事）：
+Start +Messaging +Receive +Streaming +History +ContextUsage +Todos +Tasks +SideQuestion +Configuration

为什么这么做：这是单一职责原则的落地。每个扩展文件只回答一个问题——+Start 管启停、+Receive 管收消息、+Streaming 管逐字效果。你想改哪件事，就打开对应那一个文件，不必在巨型文件里大海捞针。代价很小：文件多了几个，但每个都短、好读，改动影响面也清晰。这并不是把一个类拆成多个类，运行起来它仍是同一个 SessionRuntime。

上一层与外围：谁管全部会话、谁存盘、谁隔离 git

一个 App 里同时有很多会话，重启后还要记得它们。这件事由几块外围服务分工完成。

服务层 SessionManager

会话注册表 · 管所有会话的集合与生命周期

它是会话的"花名册"，按会话 id 惰性创建并缓存每一个 Session 档案袋：第一次取就造出来并存下，之后再取，拿到的是同一个实例。侧边栏列表读的就是它手里的会话记录。

服务层 SessionRepository

持久化 · 把会话存进磁盘

它是"存盘口"，本身是一个协议（接口）。生产环境用 CoreDataSessionRepository，把数据落到本地数据库；单元测试换成 InMemorySessionRepository，只在内存里跑，不碰真磁盘。

工具命名空间 WorktreeProvisioner（配 Worktree 数据）

git worktree 隔离

git worktree 是 git 的一个能力：在同一个仓库下开出一个独立的工作目录，挂在自己的分支上，彼此互不干扰。CCTerm 用它给某些会话开一块隔离的工作区——Claude 在这块区域里改文件，不会污染你正在用的主目录。WorktreeProvisioner 负责在后台线程调用 git worktree add 把它建出来，归档会话时再把它移除；至于"这个工作区在哪、叫什么名、从哪个分支来"，则由一个朴素的数据结构 Worktree 记着（它属于 Model 层的纯数据）。要补一句：WorktreeProvisioner 本身是一个 enum 命名空间，里面只有一组静态函数，并不是一个会被注入、带自身状态的服务对象——这一点和上面 SessionManager、SessionRepository 那类真正的服务不同，所以这里用中性色标注。

打个比方：worktree 像给每位访客单独开一间样板房，让他随便摆弄，而你自己住的主卧始终保持原样。

流式装配：把网络的"一团一团"变成屏幕上的"逐字打出"

这一层的规矩：分层与注入

会话与数据层的行为对象全部属于服务层。整个 App 遵循一条清晰的分层：

层是什么本节对应
Model（模型）朴素数据，结构体优先，跨边界时可编码SessionConfig、SessionRecord、Worktree
View（视图）SwiftUI 声明式界面（不在本节，见聊天区一节）
Service（服务）@Observable，靠注入获得，干业务的活Session、SessionRuntime、SessionManager 等
AppState（进程级状态容器）全 App 共享的状态袋子，靠初始化器逐级下传持有 SessionManager 等服务

这里有两条铁律。理解它们，就理解了这一层的"装配方式"：

规矩一：视图绝不自己 new 服务，服务靠注入。界面想用某个服务，必须由外部"注入"给它，而不是在视图里就地造一个。好处有两点：测试时可以注入一个假的实现（比如 FakeCLIClient、InMemorySessionRepository），把真实的子进程和真磁盘换掉；同一个服务实例也能被多处共享，而不是每个视图各造一份、各说各话。

规矩二：AppState 靠初始化器逐级下传，而不是整包 .environment 注入。进程级状态容器（AppState）是全 App 共享的状态袋子，由 AppDelegate 持有。它不会被原封不动地塞进整个界面环境，而是逐级拆包、按需下传：每一层只拿到它真正需要的那几样东西。这样依赖关系是显式的——看一个对象的初始化器，就知道它到底用了哪些服务，而不是"环境里啥都有，谁都能伸手拿"。

一条消息的旅程（数据流）

本节把前面拆开讲的组件重新串成一个动态故事：你在输入栏敲下一句话、按下发送，到 Claude 的回复一个字一个字刷进聊天记录，数据先后流经了哪些对象，谁把什么交给了谁。

这趟旅程背后有一条统领全局的设计原则，得先记住：聊天区大约九成是单向数据流。所谓单向数据流（对应到代码里就是 data-flow rules），意思是状态向下流、事件少量向上——会话的真相数据从底层一路往上推，交给界面去显示；界面真正需要往回报告的，只有“用户点了发送”“用户切换了会话”这类零星动作。把这条原则记牢，下面每一步的流向就都有了依据。

逐步解说这趟旅程

把上图按时间顺序走一遍。前半程（第 1 到 4 步）是用户的输入“向下”流；后半程（第 5 到 9 步）是 Claude 的回复“向上”流回界面，这正是单向数据流的主体。

  1. 用户在输入栏敲字、按发送。输入栏 InputBarView2 是一个纯粹的 SwiftUI 视图。它不知道会话是谁，只通过回调把“这次提交”（文本加图片附件）交出去。提交落到共享函数 submitSessionInput，由它把附件和文本拼成一条完整消息。
  2. 消息交给会话门面。submitSessionInput 调用 Session.send。界面只跟 Session 这个门面打交道，从不直接碰底层的会话运行时——这正体现了单向数据流“事件向上只走一条窄路”的原则。
  3. 会话运行时先把气泡显示出来。会话运行时 SessionRuntime 的发送主路 enqueueAndSend（位于 SessionRuntime+Start）马上往消息时间线里追加一条标记为“排队中”的本地用户消息，并同步发出一次变更通知。这样你的气泡瞬间出现，不必等底层 CLI 回应，否则会有约 100 到 300 毫秒的空白。
    这一步解释了为什么发送后界面“零延迟”就有反应：界面先行，确认随后补上。
  4. 通过 CLIClient 把消息发给底层 CLI。运行时把消息写进 CLIClient。CLIClient 是一层薄抽象，真实实现是 AgentSDKCLIClient，测试用 FakeCLIClient。它把消息经 AgentSDK 这个独立 Swift Package 写进底层 Claude Code CLI 子进程的标准输入。到这里，用户输入的“向下”流就走完了。
  5. Claude 的回复以流式增量回来。底层 CLI 不会一次性吐出整段回复，而是一小段一小段地流式返回。AgentSDK 把这些回复经两路回调推回运行时：细碎的文字增量走 onStreamEvent，一条定稿的完整消息走 onMessage。
  6. 运行时装配这些回复。流式增量由 SessionRuntime+Streaming 消费，它用打字机效果把文字一个字一个字揭示出来，先放一条“临时预览”消息；定稿消息由 SessionRuntime+Receive 消费，它更新 Session 的消息时间线，必要时把临时预览原地换成定稿内容。
    打字机效果对应到代码里就是 StreamingTurnAssembler / TypewriterReveal / FrameTicker，它们把网络上忽快忽慢的数据块解耦成均匀的逐字显示。
  7. 消息时间线发出变更通知。运行时每次改动消息，都会同步触发 onMessagesChange。会话在创建时就把这个通知接到了桥接层（wireRuntimeMessagesSink），所以变更会立刻流向渲染侧——哪怕这个会话此刻没显示在屏幕上，也照样被处理。
  8. 桥接层把消息翻译成块。NativeTranscript2Bridge 收到变更后，由 MessageEntryBlockBuilder 把一条消息翻译成一个或多个块（Block，即聊天记录里的一行、一个渲染单元）。它给每个块算一个稳定的 id，于是“排队中”气泡变成“已确认”时，同一行能就地更新，不必整行重画。翻译好的增量经 Transcript2Controller 交给 Transcript2Coordinator，更新它持有的 blocks——这份块列表就是聊天记录的行数据真相。
  9. 聊天记录渲染并滚到底。块列表一变，基于 NSTableView 自绘的 NativeTranscript2 就按差量插入、更新对应的行，并滚动到底部。于是 Claude 的回复一行行刷到你眼前，整趟旅程闭环。
为什么这条回程是“单向”的？从第 5 步到第 9 步，数据始终朝一个方向流：底层回复 → 运行时装配 → 写入消息时间线 → 翻译成块 → 渲染。界面（聊天记录）只负责“显示真相”，从不反过来去改运行时的状态。真正“向上”的事件，只有用户的发送（第 1 步）和切换会话这类零星动作。把绝大多数数据流锁成单向，正是这套聊天区“AppKit 外壳 + SwiftUI 叶子”能保持清晰的根本原因：中间没有一个居中的协调式 ViewModel 双向搅动状态，每一步该把什么交给谁都一目了然。

这趟旅程把前几节的静态组件都用了起来：输入栏和发送函数属于输入侧，会话、会话运行时和 CLIClient 属于会话与数据层，桥接层和聊天记录属于渲染侧。它们各司其职，靠一条主要为单向的数据流串成一个整体。下一节会把这背后最关键的三个“为什么”集中讲清楚：SwiftUI 与 AppKit 的取舍、单向数据流，以及 runloop 时序。

关键设计抉择（为什么）

本节集中回答三个"为什么"：为什么大部分界面用 SwiftUI，少数地方却下沉到 AppKit；为什么状态只朝一个方向流动；为什么有些切换必须在"同一帧的同一段时间"里同步完成。读懂这三点，前面看到的结构就不再是死规矩，而是有理由的选择。

前面几节讲的是"有哪些块、怎么拼"。这一节换个角度，讲"为什么这样拼，而不是别的样子"。三个抉择层层相扣：第一个决定界面的技术分工，第二个决定状态怎么传递，第三个决定这些传递必须卡在哪一刻发生。我们从最直观的技术选型说起。

抉择一：SwiftUI 是默认，AppKit 是例外

先定义两个名词。SwiftUI 是苹果较新的声明式界面框架，写起来简洁，你只描述"界面长什么样"，框架自己负责刷新；AppKit 是苹果传统的命令式界面框架，更老、更底层，但你能精确控制每一步发生的时机。这个项目的总原则只有一句：默认用 SwiftUI，只有当 SwiftUI 满足不了需求时，才针对那一处下沉到 AppKit。

打个比方：盖房子默认用预制板（SwiftUI），又快又省力；承重墙、地基这类对受力和时机有硬要求的地方，才现场浇筑混凝土（AppKit）。绝不会因为某处用了混凝土，就把整栋楼都现浇。对应到代码里，全 App 只有五处用了 AppKit，每一处都有明确的、SwiftUI 给不了的理由。下表逐条列出。

为什么这么定？ 把规则定成"默认 SwiftUI、例外才 AppKit"，而不是"哪个顺手用哪个"，是为了让边界可控。每新增一处 AppKit，都得拿出一个具体理由（性能、生命周期时机，或某个缺失的能力），否则就该用 SwiftUI。这样一来，混合两种框架的复杂度被锁在五个已知的点上，不会到处蔓延。

技术分工定了，接下来的问题是：这些 SwiftUI 叶子和 AppKit 外壳之间，状态该怎么传？答案是单向数据流。

抉择二：单向数据流——状态住在最低的共享作用域

先定义术语。单向数据流指状态主要朝一个方向流动：从上往下传给界面，界面产生的事件只有极少量往上回传。聊天区大约 90% 是这种模式：AppKit 当外壳，SwiftUI 当叶子，数据从外壳流向叶子。

关键规则是：每份状态都住在"所有读它的人都够得到的最低作用域"，不往上拔高。打个比方，公司里的文件不会一股脑全堆进 CEO 办公室；全公司要看的放公共区，一个部门用的放部门柜子，一个人用的就放自己抽屉。放得越低，能改它的人越少，出错的面就越小。对应到代码里，就是下面这座作用域阶梯。

数据向下流好理解，那"事件往上回传"呢？整个聊天区只保留一条结构性的向上边：selectionObserver。当用户在侧边栏点了另一个会话，这个事件需要"往上"通知详情路由器（按选择挂载对应面板的那一层，DetailRouterViewController）去换面板。它是唯一的结构性向上通道，刻意没有做成一个通用的"通知总线"，也没有加第二个观察者槽位。

为什么只留一条向上边？ 多一条向上通道，就多一处状态可能被两个地方同时改、改得不一致。把"对选择做结构性反应"这件事收归到唯一一个观察者（详情路由器）；其它视图各自观察自己作用域里的 @Observable 状态，自动刷新就够了，谁也不需要"往上喊话"。这样数据流向始终清晰：一条主干向下，仅一条细线向上。

这条向上边不只"唯一"，还有个苛刻的时机要求——它必须在点击的那一瞬间同步触发，不能晚一拍。要讲清这个"一拍"是什么，就得请出最后一个抉择：runloop 时序。

这张图带出两个承重结论，正是前两个抉择背后的时机依据。

结论一：@Observable 写入不会在同一个 tick 到达 SwiftUI 的 body。 你在 source 阶段（①）改了一个被观察的状态，SwiftUI 并不会立刻刷新——它要等到本圈的 beforeWaiting（②）才重算 body。打个比方：你在 ① 段往收件箱投了封信，对方要到 ② 段才开箱读信，绝不会"投完瞬间就读到"。所以凡是讲究先后次序、必须"此刻就生效"的操作，都不能依赖 @Observable 自动刷新，得在 ① 段当场用代码做掉。

结论二：结构性切换必须在点击的 source 阶段同步完成，否则会跨帧撕裂。 回到抉择二那条唯一的向上边。用户点击会话发生在 ① 段，MainSelectionModel.select(_:) 在同一个 ① 段里先更新 @Observable 的选择值，紧接着同步通知详情路由器去换面板、挂载新会话的聊天记录。三件事卡在同一段时间里一次做完。

为什么不能交给 @Observable 自动驱动换面板？ 如果让详情路由器靠观察选择值来反应，那个反应会被推迟到下一段（beforeWaiting，甚至下一圈）才发生——这正是结论一说的"晚一拍"。结果就是：SwiftUI 的输入栏在这一帧按新会话刷新了，而聊天记录的挂载落在下一帧，画面在两帧之间裂成两半，看得见的撕裂。所以 select(_:) 选择"自己当场同步喊一声"，让换面板和聊天记录挂载与点击落在同一帧的同一段。这也是为什么主窗口骨架必须用 AppKit：只有它能精确地把这些动作钉在 source 阶段，SwiftUI 的声明式刷新做不到。

这就把三个抉择串成了一条逻辑链：因为聊天记录要快、切换要不撕裂，所以外壳下沉到 AppKit（抉择一）；因为要让状态流向清晰、只留一条向上边，所以采用单向数据流（抉择二）；而这条向上边之所以必须同步触发，根子就在 runloop 一个 tick 分段、跨段就晚一拍这个时序事实上（抉择三）。理解了"为什么"，下一节我们再聊聊"现状未必最优、哪里还能更好"。

可以怎么更好（克制的优化建议）

本节想提醒你一件事：你刚读到的架构是深思熟虑的产物，但"深思熟虑"不等于"已是最优"，保留一点批判的眼光，是健康的工程习惯。

前面几节解释了项目"为什么这么设计"，每一处取舍都有它的理由。但任何真实的工程都会留下权衡的痕迹：为了拿到某个好处，往往要在别处付出成本。下面挑出两处可以再想一想的地方，供你日后熟悉代码后参考。它们都是观点，不是定论——项目维护者很可能早已权衡过，并有意选择了现在的做法。

先把边界说清楚。这里只点 2 处，而且都停留在"主观感受"这一层，不涉及任何确凿的缺陷。凡是没有把握的地方，本节一律不写——无中生有地批评，比不批评更糟。

观点一：跨框架的边界依赖 runloop 时序，这是一种隐式契约

这里先补一句定义。所谓 runloop 时序，指的是 AppKit、SwiftUI、CoreAnimation 共享同一轮事件循环，而这一轮内部又分成几段：source 阶段（你的代码在此运行）、beforeWaiting 阶段（界面在此刷新）等等（第 7 节有详细的时序条带图）。本项目里大量关键操作的正确性，都建立在一个前提之上：某件事必须发生在某一段，而不能发生在另一段。

举两个例子。其一，MainSelectionModel 的 select(_:) 必须在点击的同一个 source 阶段，同步通知它唯一的结构性观察者 DetailRouterViewController，由后者立即切换面板；否则会话切换会跨帧撕裂。其二，TranscriptSwapCoordinator 在挂载新聊天记录时，会"先建新的、后拆旧的"，并把那些一旦跨帧就会撕裂画面的结构性步骤（构建、绑定、锚定、同步拆旧）一并装进同一个事务，这个事务关闭了隐式动画；而真正可见的淡入淡出，则被刻意放在事务之外、交给动画系统去跑——不这样做，透明度动画会被"关闭隐式动画"这条设置一并压掉。这些规则真实存在，也确有必要。

【这是一个观点，不是事实】

这类"X 必须发生在某个 runloop 阶段"的约束，本质上是一种隐式契约——类型系统和编译器都不会强制它，只能靠开发者记在脑子里。这就像一栋老宅里"哪扇门不能猛关、哪级台阶要踩边上"的住户默契：本地人门儿清，新搬来的人却容易撞上。对应到代码就是：约束没写进语法，只能口口相传。项目已经很努力地把规则写在代码旁边——主 CLAUDE.md 里专门画了 runloop 时序图，每个协调器旁也都有注释，这很值得肯定。但即便如此，这类契约的心智负担依然偏重：新人改动边界附近的代码时，或许会在不知情的情况下踩到时序假设，而出错的表现往往是"偶发一帧的视觉抖动"这种难复现的现象。可以考虑的方向是：有没有办法把更多这类约束从"靠文档约定"变成"靠代码结构强制"，从而降低误用的概率。这只是一个值得探讨的方向，并非说现状有错。

观点二：协调器集中了很多"单一所有者"职责，上手有门槛

这里也先补一句定义。项目里把像 TranscriptSwapCoordinator 这样的对象叫做协调器：它负责把若干零散的部件按正确的顺序拼到一起，又不让这些部件彼此直接知道对方。TranscriptSwapCoordinator 是聊天记录挂载机制的协调器，同时也是多项状态的单一所有者——它独占 currentSession、每次挂载的滚动视图 Transcript2ScrollView、弹窗呈现器 Transcript2SheetPresenter 等。

"单一所有者"是一个清晰的设计：把某项状态的归属收拢到一处，避免两个对象各持一份、互相不同步。源码注释讲得很明白：正在进行的淡入淡出会读取被暂存的旧会话，一旦出现第二个持有者，两份状态就会失步。这样收拢之后，边界确实干净。

【这是一个观点，不是事实】

代价在于：这样的协调器，从结构上看更接近一个聚焦的小型状态机——它内部有"幂等短路、暂存旧会话、赋值、清理"等多个环环相扣的步骤，还顺带管着滚动视图、弹窗呈现器这两类每次挂载都要重建的附属资源。它的好处是边界清晰，但或许能预见到一项成本：新人想安全地改动它，得先在脑子里建立起这套所有权划分和挂载顺序，否则很容易破坏不变量。这不是说应该把职责拆散——拆散反而会引入失步风险，源码已经说明过。这里只想点出"上手门槛"这项客观存在的取舍，让你在阅读这部分代码时心里有数。

读到这里，请带走一个心态，而不是一份待办清单。上面两点的共同主题是同一个：本项目为了换取性能和正确的渲染时序，主动选择了一些"靠人来维护"的精巧约定。这些约定目前运转良好，文档也写得很用心。提出它们，不是要否定现状，而是希望你将来动这部分代码时，既理解它"为什么这么做"，也意识到"它的成本在哪里"——这正是从"小白"成长为这个项目贡献者的关键一步。
